Contêineres e Orquestração (Docker e K8s)

🥊 Contêineres (Docker) vs Máquina Virtual (VM)

COMPARAÇÃO CLÁSSICA

Máquina Virtual (VM)

  • • Necessita de um Hypervisor (VMWare, VirtualBox, Xen).
  • • Isolamento de Hardware: Cada VM roda um Sistema Operacional convidado (Guest OS) completo e pesado.
  • • Demora minutos para inicializar (boot do S.O.).
  • • Ocupa Gigabytes (GB) de armazenamento.

Contêiner (Ex: Docker)

  • • Necessita de uma Container Engine (Docker Daemon).
  • • Isolamento no Nível do S.O.: Compartilha o Kernel (núcleo do SO) da máquina hospedeira. NÃO possui SO completo interno.
  • • Inicializa em milissegundos.
  • • Ocupa apenas Megabytes (MB).

🐳 Vocabulário Básico: Docker

Imagem (Image)

O "molde" imutável. Contém a aplicação e todas as suas dependências pré-instaladas. Uma vez construída, não muda (Read-only).

Contêiner (Container)

A instância em execução de uma Imagem. Pode ser iniciado, pausado e destruído (efêmero).

Dockerfile

Um script de texto simples com instruções (ex: FROM ubuntu, RUN apt-get install python) usado para construir a Imagem.

☸️ Orquestração com Kubernetes (K8s)

Se o Docker cria o contêiner, o Kubernetes é o "maestro" que gerencia milhares de contêineres espalhados em dezenas de servidores (Clusters).

Cluster Architecture
  • Control Plane (Master Node): O cérebro. Toma decisões globais sobre o cluster (escalonamento, monitoramento).
  • Worker Nodes (Minions): As máquinas "peão". Onde os contêineres realmente executam o trabalho pesado.
A Unidade Básica: POD

O Kubernetes NÃO executa contêineres diretamente. Ele os encapsula num POD. O Pod é a menor unidade implantável no K8s. Um Pod geralmente contém 1 contêiner, mas pode conter múltiplos contêineres que precisem compartilhar o mesmo IP, rede e armazenamento (tightly coupled).

🚨 Pegadinhas Mortais da Banca (CEBRASPE)

  • Contêineres contêm S.O. próprio?
    "Um dos benefícios do Docker é que cada contêiner possui seu próprio sistema operacional isolado, garantindo máxima segurança e independência."
    Falso! Contêineres NÃO possuem Guest OS. Todos os contêineres executados na mesma máquina compartilham o mesmo Kernel (núcleo) do S.O. hospedeiro. Isso é o que os torna levíssimos comparados às VMs.
  • O Limite do Pod:
    "No Kubernetes, um Pod é rigidamente restrito a hospedar um, e somente um, contêiner por vez."
    Falso! Embora o padrão "um contêiner por Pod" seja o mais comum, um Pod PODE encapsular múltiplos contêineres, desde que eles sejam fortemente acoplados (ex: um contêiner web rodando Nginx e um contêiner auxiliar rodando um gerador de logs, o padrão Sidecar).
  • Docker = Orquestrador?
    "O Docker Engine nativo realiza autonomamente a orquestração, o balanceamento de carga e o self-healing de clusters de instâncias espalhadas em vários servidores."
    Falso! O Docker (Motor/Engine base) serve para rodar contêineres em uma única máquina. Para orquestrar em múltiplos servidores/clusters, utiliza-se o Kubernetes (K8s) ou o Docker Swarm (a ferramenta nativa de orquestração do Docker, que é diferente do Docker básico).

1. A Revolução dos Contêineres (Docker)

Durante muito tempo, o padrão de isolamento em servidores foi a Máquina Virtual (VM). A virtualização particiona o hardware físico. O problema é que cada VM exigia a instalação de um Sistema Operacional completo (Guest OS). Subir 10 VMs significava carregar 10 Windows/Linux na memória, tornando o processo lento e obeso.

Os Contêineres mudaram o paradigma. Em vez de virtualizar o hardware, eles virtualizam o Sistema Operacional.

A Tecnologia por Trás (Linux)

O Docker, ferramenta mais famosa de conteinerização, baseia-se em dois recursos nativos do kernel Linux para garantir o isolamento sem precisar de outro SO:

  • Namespaces: Garantem o isolamento de visão. Um contêiner acha que tem sua própria rede, seus próprios PIDs de processos e sua própria árvore de arquivos. Ele não "vê" o que está no contêiner vizinho.
  • Cgroups (Control Groups): Garantem o isolamento de hardware. Limitam o quanto de CPU, Memória RAM e banda de I/O um contêiner pode usar.

Por não precisarem fazer "boot" num sistema operacional, contêineres iniciam em frações de segundo e tornaram os Microsserviços e a Integração Contínua (CI/CD) uma realidade viável. O dogma "Na minha máquina funciona, no servidor não" morreu: a Imagem Docker embala o código e todas as suas dependências.

2. O Problema da Escala: Por que o Kubernetes?

O Docker resolve o problema de criar e rodar um contêiner. Mas e se você for a Netflix ou o iFood? Você terá 5.000 contêineres de carrinho de compras espalhados por 50 servidores físicos.

O que acontece se o Servidor 12 queimar? Quem recria os contêineres dele no Servidor 13? Quem avisa o Load Balancer que o IP mudou? Quem garante que quando você lançar a versão 2.0 do software, as pessoas não ficarão sem acesso?

Isso é a Orquestração de Contêineres. E o Kubernetes (K8s), plataforma de código aberto originada no Google, é o padrão de facto da indústria (superando o Docker Swarm e Apache Mesos).

3. Arquitetura do Kubernetes (Vocabulário de Prova)

Componentes do Control Plane (O Mestre)

O Control Plane toma as decisões globais do cluster, detecta e responde aos eventos. Ele não executa a aplicação do cliente.

  • kube-apiserver: O porteiro. Todas as requisições (via linha de comando `kubectl` ou interfaces) passam por ele. É a API front-end.
  • etcd: O banco de dados do cluster. Um armazenamento chave-valor altamente disponível que salva todo o estado do cluster.
  • kube-scheduler: O cérebro logístico. Ele vigia a criação de novos Pods sem nó atribuído e decide em qual "Worker Node" aquele Pod vai rodar, baseando-se em RAM/CPU livre.

Componentes do Worker Node (O Trabalhador)

O nó trabalhador é onde seus contêineres rodam.

  • Kubelet: O agente (capataz) que roda em cada nó. Ele garante que os contêineres descritos pelo Master estão rodando de forma saudável e relata o status de volta ao Master.
  • Kube-proxy: Um proxy de rede em cada nó. Mantém as regras de rede (iptables) que permitem a comunicação com os Pods.
  • Container Runtime: O software responsável por de fato rodar o contêiner (O Docker era usado, mas o Kubernetes adotou hoje padrões genéricos como `containerd` ou `CRI-O`).

Objetos do Kubernetes

  • Pod: Menor unidade do K8s. É uma "cápsula" temporária que abriga um ou mais contêineres. Se o contêiner cai, o Pod morre e um novo Pod nasce (Pods são efêmeros, não são "consertados").
  • Deployment: É o objeto que garante a Resiliência e a Escala. Você declara para o Deployment: "Eu quero 3 Pods do site rodando sempre". O Deployment gerencia os ReplicaSets e cuida de matar ou recriar os pods se algo falhar (Self-healing).
  • Service: Como os Pods morrem o tempo todo, seus endereços IP mudam a toda hora. O Service provê um IP e um nome DNS Estável, atuando como um mini balanceador de carga que roteia o tráfego para os pods (mesmo que eles mudem de IP).

🚀 4. A Essência da "Declaração de Estado"

A principal característica abordada em questões teóricas é que o Kubernetes funciona num modelo Declarativo e não Imperativo.

Você não dá ordens passo a passo como: "Crie a máquina A, ligue a B, depois mate a C". Você envia um arquivo (geralmente YAML) declarando seu Estado Desejado (Ex: "Eu quero 5 Pods rodando a versão 2 do Nginx"). O *Control Plane* do Kubernetes olha para o Estado Atual (Ex: "Tenho apenas 3 Pods na versão 1"). A partir daí, o próprio K8s, de forma autônoma e inteligente, toma as ações corretivas até que o Estado Atual fique exatamente igual ao Estado Desejado.
