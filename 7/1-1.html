Modelos de Serviço (SPI)

📊 A Pirâmide de Controle

Quanto mais alto na pirâmide, MENOS coisas o cliente tem que gerenciar.

SaaS
Software
Foco no Usuário Final

O cliente NÃO gerencia NADA da infraestrutura, nem sistema operacional, nem a aplicação em si.
Exemplos: Gmail, Salesforce, Microsoft 365.

PaaS
Plataforma
Foco no Desenvolvedor

O cliente gerencia a Aplicação e os Dados. O provedor cuida do S.O., Servidores, Bancos de Dados e Runtime.
Exemplos: Heroku, Google App Engine, AWS Lambda (Serverless atua aqui).

IaaS
Infraestrutura
Foco no Administrador (SysAdmin)

O cliente gerencia S.O., Middleware, Aplicação e Dados. O provedor entrega a máquina virtual (Hardware, Rede, Storage).
Exemplos: AWS EC2, Azure VMs, Google Compute Engine.

Matriz de Responsabilidade Compartilhada

Camada IaaS PaaS SaaS
Aplicações / Código Cliente Cliente Provedor
Dados / Informações Cliente Cliente Provedor*
Runtime / Middleware Cliente Provedor Provedor
Sistema Operacional (OS) Cliente Provedor Provedor
Virtualização (Hypervisor) Provedor Provedor Provedor
Servidores / Storage / Rede Física Provedor Provedor Provedor
* No SaaS, o cliente é dono dos dados (ele decide quem acessa, faz upload/download), mas a guarda, formatação e backup técnico desses dados é do provedor.

🚨 Pegadinhas Mortais da Banca (CEBRASPE)

  • Atualização de Sistema Operacional (OS Patching):
    A banca diz: "No modelo PaaS, o consumidor deve gerenciar a aplicação implantada, bem como manter os pacotes de segurança do Sistema Operacional atualizados."
    Falso! A fronteira do PaaS é o "Runtime". O Sistema Operacional subjacente (Windows/Linux) é responsabilidade EXCLUSIVA DO PROVEDOR. O cliente só mexe no SO se ele contratar um IaaS!
  • Banco de Dados Relacional:
    A banca tenta confundir onde fica o DBaaS (Database as a Service - como Amazon RDS). Se o provedor cuida da instalação do SQL Server, do backup e da alta disponibilidade, e o cliente apenas cria as tabelas e insere dados, isso é considerado um PaaS (Plataforma), não IaaS.
  • Customização no SaaS: "No modelo SaaS, o cliente não possui absolutamente nenhum controle, sendo incapaz de alterar o comportamento da aplicação."
    Falso! O cliente não controla a infraestrutura, MAS a definição oficial do NIST afirma que o cliente pode ter controle sobre configurações limitadas do aplicativo específicas do usuário (Ex: Mudar as cores do painel, criar regras de filtro de e-mail no Gmail, cadastrar novos usuários).

1. O Modelo SPI: A Hierarquia da Nuvem

O NIST define os Modelos de Serviço em Cloud Computing frequentemente pela sigla SPI (SaaS, PaaS, IaaS). Eles definem o nível de abstração que a infraestrutura subjacente é escondida do consumidor, e concomitantemente, a divisão de trabalho (Modelo de Responsabilidade Compartilhada) entre o Provedor de Cloud (AWS, Azure, Google) e o Cliente.

IaaS Infrastructure as a Service

No IaaS, a capacidade fornecida ao consumidor é provisionar processamento, armazenamento, redes e outros recursos computacionais fundamentais. A palavra-chave é: o cliente pode implantar e executar software arbitrário.

A Linha de Fronteira (Virtualização)

O provedor é dono do Data Center físico, da energia, da refrigeração, do servidor físico e do Hypervisor (Camada de Virtualização). Daí para cima, o problema é do cliente.

Se o cliente cria uma Máquina Virtual (VM) Linux, ele é quem deve configurar as regras de Firewall interno (iptables), ele é quem deve instalar o Apache/Nginx, e ele é quem será hackeado se esquecer a porta 22 (SSH) aberta para a internet sem senha. O provedor de IaaS garante apenas que o hardware virtualizado não saia do ar.

Palavras-chave de Prova: Máquina Virtual (VM), EC2, Instâncias, Controle sobre o Sistema Operacional Guest, Flexibilidade total de ambiente.

PaaS Platform as a Service

O PaaS eleva o nível de abstração. O consumidor não se preocupa mais em como instalar o Windows ou o Linux, nem qual versão do Java ou do PHP está rodando no servidor raiz. O foco passa a ser puramente o Desenvolvimento e Deploy de Aplicações.

Produtividade vs Controle

Neste modelo, o provedor entrega o ambiente de execução (Runtime). Por exemplo, um ambiente pronto para rodar código Node.js. O desenvolvedor escreve a sua aplicação, clica em "Deploy" e o provedor se encarrega de alocar os recursos de máquina, instalar as bibliotecas base, balancear a carga e aplicar as correções de segurança no S.O.

Atenção (CEBRASPE): O cliente não gerencia a infraestrutura, mas pode controlar as configurações do ambiente de hospedagem da aplicação (ex: definir variáveis de ambiente, número máximo/mínimo de instâncias que a aplicação pode escalar).

SaaS Software as a Service

O mais alto nível de abstração. Aqui, o software já está pronto, compilado, testado e hospedado pela empresa provedora. O consumidor é o Usuário Final.

O acesso ocorre frequentemente por meio de clientes leves (thin clients), como navegadores web ou aplicativos móveis. A arquitetura padrão do SaaS moderno é baseada em Multi-tenancy (Multilocatário), o que significa que vários clientes diferentes (empresas diferentes) usam a exata mesma instância do software e do banco de dados físico, mas os dados são isolados logicamente por IDs de conta.

Atenção (CEBRASPE): O pagamento no SaaS geralmente é baseado no número de licenças/assinaturas (ex: $10 por usuário por mês) em contraste com o IaaS, que foca no consumo técnico por hora (CPU, GB trafegado).

👾 Modelos Derivados (Para Provas Difíceis)

Embora o NIST formalize apenas IaaS, PaaS e SaaS, a indústria criou terminologias secundárias que frequentemente caem nas provas.

  • FaaS (Function as a Service) / Serverless: O desenvolvedor escreve uma pequena função lógica (um trecho de código isolado) e manda para a nuvem. Ele só paga pelos milissegundos exatos em que aquela função foi ativada. A infraestrutura para rodar isso é 100% abstraída. (Ex: AWS Lambda).
  • CaaS (Container as a Service): Fica entre IaaS e PaaS. O provedor não entrega apenas a VM crua, mas uma infraestrutura de orquestração de containers já pronta, e o cliente sobe seus containers (Docker/Kubernetes).
  • DBaaS (Database as a Service): O provedor aloca um banco de dados já configurado, cuidando do backup e alta disponibilidade. Classifica-se, filosoficamente, como um subconjunto de PaaS, focado exclusivamente no fornecimento de motor de dados.
