Autenticação (AuthN)

Verifica a IDENTIDADE.

  • • Responde: "Quem é você?"
  • • Valida as credenciais apresentadas.
  • • Exemplos: Inserir usuário e senha, leitura de biometria, Token de MFA.
🗝️

Autorização (AuthZ)

Verifica as PERMISSÕES.

  • • Responde: "O que você pode fazer?"
  • • Ocorre APÓS a autenticação.
  • • Exemplos: Permitir a leitura de um arquivo, mas bloquear a deleção.

🎭 Como dar permissões? (RBAC)

RBAC (Role-Based Access Control)

Acesso baseado em Papéis (Cargos). Em vez de dar permissão ao "João", você cria o papel "Analista_BD" com as devidas permissões, e atribui o João a esse papel. Muito mais fácil de administrar em larga escala.

ABAC (Attribute-Based Access Control)

Acesso baseado em Atributos (Condições). Ex: "Permitir apagar arquivos" SOMENTE SE "o atributo for Horário Comercial (08h às 18h)" ou "IP de origem for da rede interna". Mais granular, porém mais complexo.

🔐 A Trindade do MFA

MFA (Multi-Factor Authentication) exige pelo menos DOIS fatores de naturezas DIFERENTES para liberar o acesso.

🧠
Algo que você SABE

(Conhecimento)
Senha, PIN, Pergunta de segurança.

📱
Algo que você TEM

(Posse)
Smartphone (Google Authenticator), Token físico USB, Smartcard.

👁️
Algo que você É

(Inerência)
Impressão digital, Reconhecimento facial ou de íris.

🚨 Pegadinhas Mortais da Banca (CEBRASPE)

  • Falso MFA:
    A banca diz: "Um sistema que exige a digitação de uma senha alfanumérica forte e, na tela seguinte, a digitação de um PIN numérico (ambos memorizados), implementa corretamente o MFA."
    Falso! Isso é Single-Factor Authentication com dois passos (ambos são fatores de Conhecimento). O MFA exige fatores de CATEGORIAS DIFERENTES (ex: Senha [Saber] + App Gerador de Código no Celular [Ter]).
  • RBAC: Usuários com permissão direta:
    "No modelo RBAC, a maior segurança é obtida atribuindo as permissões restritivas diretamente ao objeto do Usuário."
    Falso! O propósito absoluto do RBAC é NÃO atribuir permissões a usuários diretamente. As permissões são atreladas aos Roles (Papéis) e os usuários herdam essas permissões ao serem alocados nesses papéis.
  • Usuário Root vs IAM User:
    "O usuário Root (conta principal) deve ser compartilhado entre a equipe de administração sênior para tarefas diárias de infraestrutura na nuvem."
    Falso! O usuário Root deve ter o MFA ativado, as chaves de acesso deletadas, e ser guardado a sete chaves. Ninguém usa a Root Account no dia a dia. Cria-se usuários IAM com privilégios administrativos para o trabalho diário.

1. IAM (Identity and Access Management)

O IAM (Gerenciamento de Identidade e Acesso) é o framework de políticas e tecnologias que garante que as pessoas certas na organização tenham o acesso adequado aos recursos tecnológicos corretos.

Na nuvem, o IAM é o "Segurança da Porta". Ele controla absolutamente tudo, desde quem pode fazer login no painel da AWS/Azure até qual máquina virtual pode ler arquivos num bucket de armazenamento.

A Trindade do IAM

  • Identificação: Estabelecer quem é o sujeito (ex: Username).
  • Autenticação (AuthN): Provar que o sujeito é quem diz ser (ex: Senha, MFA).
  • Autorização (AuthZ): Conceder permissões para que o sujeito autenticado realize ações específicas (ex: "Pode ler", mas "Não pode deletar").
Princípio do Menor Privilégio (PoLP)

Regra universal em cibersegurança: O sistema Nega tudo por padrão (Default Deny). O usuário (ou máquina) deve receber apenas as permissões estritamente necessárias para realizar sua função, e nada além disso.

2. RBAC (Role-Based Access Control)

Gerenciar permissões individualmente (conhecido como Discretionary Access Control - DAC) é um pesadelo logístico. Se uma empresa tem 5.000 funcionários, dar permissão um por um geraria caos.

O RBAC resolve isso abstraindo o usuário das permissões. As permissões são vinculadas a Roles (Papéis/Cargos), e não a indivíduos.

  • Cria-se a Role "Desenvolvedor_Senior".
  • Atribui-se a essa Role permissões para acessar os servidores de Homologação e Banco de Dados de Teste.
  • O funcionário "João" é associado à Role "Desenvolvedor_Senior", herdando as permissões automaticamente.
  • Se o João for promovido a "Gerente", basta retirá-lo da Role atual e colocá-lo na nova. Não é necessário revogar permissões individuais.

Roles para Máquinas (IAM Roles)

Na nuvem, as "Roles" (Funções) têm um papel ainda mais crítico: credenciais temporárias para servidores.

Nunca se deve "chumbar" (hardcode) senhas ou chaves de API no código de um sistema. Em vez disso, cria-se uma Role ("Permissão para gravar no Storage") e a atrela à Máquina Virtual (EC2). Quando a VM liga, ela assume aquela Role e ganha permissões sem precisar de senhas fixas (ela usa tokens temporários rotacionados automaticamente).

3. MFA (Autenticação Multifator)

MFA é a camada de segurança que mitiga o risco de comprometimento de senhas (ex: ataques de *Phishing*, força bruta ou credenciais vazadas em brechas de terceiros).

Mesmo que o atacante tenha a sua senha, ele não conseguirá entrar, pois não possui o "segundo fator". Para ser MFA válido, é mandatório combinar fatores de categorias diferentes:

  • Fator de Conhecimento: Algo que o usuário sabe de cor. (Senha, PIN, Pergunta Secreta, Padrão de desbloqueio).
  • Fator de Posse: Algo que o usuário possui fisicamente. (Telefone celular com app Authenticator gerando TOTP - Token Baseado no Tempo; Chave de Segurança USB FIDO/YubiKey; Token de Hardware do banco).
  • Fator de Inerência (Biometria): Algo que faz parte de quem o usuário é biologicamente ou comportamentalmente. (Impressão Digital, Geometria da face, Scanner de Íris, Reconhecimento de voz).
Fatores Contextuais (Bônus):
A segurança moderna também avalia "Fatores de Localização/Tempo" (Algo de onde você está). Ex: O login feito no Brasil sempre dá certo. Um login feito na Rússia, 2 minutos depois, exigirá um bloqueio ou passo extra de verificação (Autenticação Baseada em Risco).

📄 4. Como as Políticas (Policies) são escritas?

Tanto AWS, Azure quanto GCP utilizam documentos formatados (usualmente JSON) para ditar as regras do IAM. É importante entender a estrutura mental de uma "Policy". Uma apólice de permissão básica responde:

  • Effect (Efeito): É um Allow (Permitir) ou um Deny (Negar)? O Deny explícito sempre sobrepõe o Allow.
  • Action (Ação): O que ele quer fazer? (ex: s3:ListBucket, ec2:StartInstances).
  • Resource (Recurso): Em qual item ele pode fazer isso? (ex: Somente no banco de dados X, e não em todos os bancos).
  • Condition (Condição - Opcional): ABAC entra aqui. "Permitir isso APENAS SE ele fez login usando MFA".
