Ferramentas de IaC: Terraform

🏗️ O que é o Terraform?

O Terraform (criado pela HashiCorp) é a ferramenta líder global em Infraestrutura como Código (IaC). Ele permite definir, provisionar e gerenciar a infraestrutura de nuvem usando código escrito em uma linguagem declarativa simples chamada HCL (HashiCorp Configuration Language).

Agnóstico de Nuvem Diferente do AWS CloudFormation, o Terraform pode gerenciar recursos na AWS, Azure, GCP e on-premises, tudo com a mesma ferramenta (usando Providers).
Abordagem Declarativa Você diz O QUE quer (estado final) e não COMO fazer. O Terraform descobre sozinho a ordem de criação dos recursos.

🔄 O Fluxo Sagrado (Core Workflow)

CAI MUITO EM PROVA!
terraform init Inicializa o diretório. Faz o download dos Providers (plugins da AWS/Azure) necessários para o projeto.
➔
terraform plan "O Teste a Seco". Mostra um plano de execução detalhando o que será criado, modificado ou destruído, sem alterar nada na nuvem real.
➔
terraform apply O botão de disparo. Executa o plano e cria a infraestrutura na nuvem de verdade, atualizando o arquivo de estado.
terraform destroy Destrói toda a infraestrutura gerenciada de forma limpa.

🧠 O Cérebro: O Arquivo de Estado

// terraform.tfstate

{
  "version": 4,
  "resources": [
    {
      "type": "aws_instance",
      "name": "web_server"
    }
  ]
}

O State File (terraform.tfstate) é como o Terraform rastreia os recursos. Ele atua como um "mapa" que liga a configuração escrita no seu computador à infraestrutura real rodando na nuvem.

  • Fonte da Verdade: Antes de modificar algo, o Terraform consulta este arquivo, não a nuvem.
  • Perigo de Senhas: O state file frequentemente guarda dados sensíveis (senhas de DB) em texto puro (plaintext). NUNCA faça commit dele no Git (adicione no .gitignore).
  • Remote State: Em equipes, o state file deve ser salvo remotamente (ex: Amazon S3) com o recurso de State Locking (bloqueio via DynamoDB) para evitar que duas pessoas alterem o estado ao mesmo tempo.

🚨 Pegadinhas Mortais da Banca (CEBRASPE)

  • Terraform vs Configuration Management (Ansible):
    A banca diz: "O Terraform foi desenhado primariamente para gerenciar a configuração interna de um sistema operacional, substituindo ferramentas como o Ansible e o Chef."
    Falso! Eles têm propósitos diferentes. Terraform é focado no Provisionamento da Infraestrutura (Criar a VPC, o Load Balancer e a VM vazia). O Ansible/Chef foca em Gerência de Configuração (Entrar dentro da VM já criada e instalar o Apache e o PHP). Eles costumam trabalhar juntos!
  • O Terraform consulta a nuvem em tempo real?
    "Para saber quais recursos destruir, o Terraform faz uma varredura completa na nuvem da AWS em tempo real analisando todas as instâncias existentes."
    Falso! O Terraform lê as informações do seu Arquivo de Estado (terraform.tfstate). Se você for no painel da AWS e criar uma máquina com o mouse, o Terraform não saberá que ela existe a não ser que você mande ele importar o estado.
  • Imperativo vs Declarativo:
    "Ao programar no Terraform, o administrador deve criar scripts passo a passo indicando a ordem exata de criação: primeiro a rede, depois a máquina, depois o IP."
    Falso! O Terraform é Declarativo. Você joga todos os recursos num ficheiro. O motor interno do Terraform lê o ficheiro, constrói um Grafo de Dependências (Dependency Graph) e cria as coisas na ordem certa automaticamente, maximizando o paralelismo.

1. A Arquitetura do Terraform

O Terraform funciona através de uma arquitetura centralizada muito simples, baseada em código aberto, composta por duas partes cruciais que frequentemente aparecem em questões de concurso mais profundas:

  • Core (Núcleo): É o binário estático que você baixa (o executável terraform). O Core lê os arquivos de configuração (o código HCL), analisa o estado atual (State File) e calcula o Grafo de Dependências. Ele descobre "O que eu preciso criar primeiro?".
  • Providers (Provedores): São plugins. O Core do Terraform não sabe como comunicar com a AWS ou com o Azure. Quem sabe são os Providers. Quando você escreve provider "aws", o Terraform baixa um plugin construído pela Amazon que ensina o Terraform a traduzir o plano em chamadas de API reais (REST) para os servidores da AWS. Essa arquitetura de plugins é o que torna o Terraform uma ferramenta Agnóstica de Nuvem (Multi-Cloud).

2. A Linguagem HCL e o Conceito Declarativo

O Terraform utiliza a HashiCorp Configuration Language (HCL), uma linguagem declarativa focada em ser amigável tanto para humanos lerem quanto para máquinas processarem.

Por que Declarativo?

Em ferramentas imperativas (como um script Shell de bash), você precisa programar loops (if/else) e a ordem de execução. Se o script travar a meio, você terá infraestrutura órfã.

Na abordagem Declarativa do HCL, você declara: resource "aws_instance" "servidor_web" { ... }. Você foca no Estado Desejado. Se o servidor_web já existir na nuvem e estiver perfeito, ao rodar terraform apply o Terraform responderá: "Nenhuma mudança necessária" (Idempotência). Se o servidor não existir, ele o cria.

# Exemplo de bloco HCL no Terraform
resource "aws_s3_bucket" "banco_imagens" {
  bucket = "tce-ma-imagens-2026"
  acl    = "private"
}

3. Aprofundando no State File e Colaboração

O terraform.tfstate é um arquivo JSON gerado automaticamente após o primeiro apply.

O Desafio do Trabalho em Equipe

Se você e o seu colega desenvolvedor estiverem trabalhando no mesmo projeto Terraform, onde fica o arquivo de estado? Se ele ficar no seu laptop local, o Terraform do seu colega não vai saber o que você já criou na nuvem, e pode tentar duplicar recursos.

A Solução Padrão-Ouro (Cai em Discursivas):
  • Remote State (Estado Remoto): Em vez de manter o arquivo .tfstate localmente, o Terraform é configurado para guardar este arquivo na nuvem de forma centralizada (ex: em um bucket do Amazon S3).
  • State Locking (Bloqueio de Estado): O que acontece se duas pessoas digitarem terraform apply ao exato mesmo milissegundo? O arquivo de estado vai corromper-se. Para evitar isso, usa-se o State Locking. Na AWS, configura-se uma tabela no DynamoDB. Quando a Pessoa A digita apply, o Terraform tranca a tabela. Se a Pessoa B digitar apply, ela recebe um erro informando que o estado está "trancado" por outro processo.

📦 4. Módulos no Terraform (Reusabilidade)

Assim como na programação tradicional nós criamos "Funções" para não repetir código, no Terraform nós criamos Módulos.

Se a criação de uma VPC inteira com 4 sub-redes, tabelas de roteamento e NAT Gateways exige 300 linhas de código HCL, a equipe de operações empacota tudo isso em um Módulo chamado vpc_segura_padrao. Quando um desenvolvedor de outro setor precisar de uma rede nova, ele apenas "chama" esse módulo com 3 linhas de código, reaproveitando o conhecimento e garantindo o compliance e padronização da arquitetura do órgão público.
