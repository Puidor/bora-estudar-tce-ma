DevOps, CI/CD e IaC

♾️ O que é DevOps?

DevOps não é uma ferramenta, nem uma linguagem. É uma Cultura e Filosofia. Ele une o Desenvolvimento (Dev) com as Operações (Ops) para quebrar os "silos" e entregar software com mais rapidez e confiabilidade.

🚀 O Pipeline CI/CD

DECORE AS SIGLAS

CI

Continuous Integration
Integração Contínua. Vários programadores enviam código para um repositório central (ex: Git) várias vezes ao dia. Cada envio (commit) aciona um Build e Testes Automáticos. O objetivo é achar "bugs" rápido.

CD

Continuous Delivery
Entrega Contínua. O código testado no CI é empacotado e fica "pronto" para ser enviado para a Produção a qualquer momento. MAS a implantação na Produção requer Aprovação Humana (Aperto de botão).

CD

Continuous Deployment
Implantação Contínua. É a evolução da Entrega. Se o código passou em todos os testes do CI, ele vai direto para o cliente final (Produção) 100% AUTOMATICAMENTE. Nenhuma intervenção humana.

📜 IaC (Infraestrutura como Código)

O Fim dos Cliques

Em vez de ir no painel da AWS e clicar em "Criar Servidor", você escreve um script (ex: arquivo .tf do Terraform) dizendo: "Quero 2 servidores com 16GB de RAM".

Versionamento

Como a infraestrutura agora é "texto" (código), você guarda ela no Git. Você pode dar "Undo/Rollback" no Data Center inteiro se alguém fizer bobagem!

🚨 Pegadinhas Mortais da Banca (CEBRASPE)

  • Delivery vs Deployment:
    A banca diz: "A Continuous Delivery (Entrega Contínua) caracteriza-se por enviar cada alteração que passa nos testes diretamente para o ambiente de produção sem qualquer intervenção humana."
    Falso! Isso é Continuous DEPLOYMENT (Implantação Contínua). Na Entrega Contínua (Delivery), o código fica pronto no repositório final, aguardando um clique de aprovação de um gestor para ir ao ar.
  • Pets vs Cattle (A imutabilidade da IaC):
    "Na abordagem IaC (Infraestrutura como Código) moderna, recomenda-se corrigir servidores com falhas aplicando scripts de manutenção diretamente neles."
    Falso! IaC prega a Infraestrutura Imutável (Servidores como Gado - Cattle). Se um servidor der problema, você não "cura" o servidor instalando patches. Você destrói o servidor defeituoso e manda o script IaC criar um novo do zero em segundos.
  • DevOps como Ferramenta ou Cargo:
    "O Governo contratou um software chamado DevOps para garantir que o setor de desenvolvimento não tenha mais atritos com o setor de infraestrutura."
    Falso! DevOps não se compra, não é um software. É uma cultura organizacional (apoiada por ferramentas como CI/CD). O objetivo é unir pessoas, processos e produtos, eliminando os tradicionais silos isolados.

1. Entendendo a Cultura DevOps

Historicamente, as empresas tinham um "Muro de Confusão". De um lado, os Desenvolvedores (Dev), pagos para criar novas funcionalidades rápido. Do outro lado, os Operadores de Infraestrutura (Ops), pagos para manter o servidor estável e não deixar nada quebrar. O desenvolvedor jogava o código por cima do muro, o servidor quebrava, e começava a briga.

A cultura DevOps elimina este muro. Ela prega que quem programa o código é também responsável por como ele se comporta em produção. O framework cultural mais famoso que define o DevOps é o modelo CALMS:

  • Culture (Cultura de colaboração).
  • Automation (Automatizar tudo que é repetitivo).
  • Lean (Trabalho enxuto, entregas em pequenos lotes em vez de versões gigantes).
  • Measurement (Medir tudo, telemetria).
  • Sharing (Compartilhar sucessos e fracassos abertamente).

2. O Coração Tecnológico: CI / CD

O principal motor prático do DevOps é o Pipeline CI/CD. É a esteira de montagem do código.

Continuous Integration (Integração Contínua)

A prática de fazer o *merge* (integração) do código de todos os desenvolvedores para um repositório centralizado dezenas de vezes ao dia. O servidor de CI (Ex: Jenkins) detecta o código novo, compila (build) e roda testes automatizados unitários e de integração. O objetivo é detectar erros o mais rápido possível (Fail Fast).

Continuous Delivery (Entrega Contínua)

É o passo seguinte. Além de compilar e testar, o sistema empacota o software (ex: cria a imagem Docker) e joga para um ambiente de Staging (Homologação) perfeitamente igual ao de produção. O código está sempre pronto para ser lançado, mas o lançamento final exige aprovação humana.

Continuous Deployment (Implantação Contínua)

A utopia do DevOps. Não há aprovação humana. Se o João enviar um código ao meio-dia e ele passar em todos os testes, às 12:05 aquele código estará ativo no site real afetando os clientes. Exige um grau de confiança absurdamente alto nos testes automatizados e no monitoramento.

3. Infraestrutura como Código (IaC)

A IaC revoluciona o trabalho de Operações. Em vez de administradores de redes configurarem switches ou entrarem em painéis da web para clicar em menus, eles escrevem scripts em linguagem de marcação ou código para descrever a topologia da rede.

Abordagem Declarativa (Mais Cobrada)

O programador foca no O QUÊ ele quer. Ex: "Quero 3 servidores rodando e com a porta 80 aberta." A ferramenta (ex: Terraform) vai calcular como chegar nesse estado sozinha. É a abordagem mais moderna.

Abordagem Imperativa

O programador foca no COMO. Ele escreve os passos em ordem: 1) Crie máquina, 2) Execute este comando shell para abrir a porta 80, 3) Copie este arquivo. (Mais comum em scripts Bash antigos ou Chef).

Idempotência: Um conceito vital em IaC (e muito querido pelas bancas). Uma ferramenta é idempotente se, não importa quantas vezes você rodar o mesmo script de criação, o resultado final seja sempre o mesmo. Ou seja, se você pedir "3 servidores" e já existirem 3, a ferramenta não fará nada (não criará 6).
