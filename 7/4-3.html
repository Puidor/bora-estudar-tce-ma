Criptografia: Trânsito e Repouso

🔄 Os Dois Estados dos Dados

A BASE DA QUESTÃO
🌐

Dados em Trânsito (In Transit)

Os dados estão "viajando" pela rede.

  • • Ameaça Principal: Man-in-the-Middle (MITM) e Sniffing. Alguém "escutar" o pacote no meio do caminho.
  • • Solução Primária: Protocolo TLS (Transport Layer Security) e IPSec (VPNs).
  • • Exemplo Clássico: Acessar uma página web usando HTTPS (onde o S significa *Secure* via TLS).
💾

Dados em Repouso (At Rest)

Os dados estão parados "no disco".

  • • Ameaça Principal: Alguém invadir o datacenter, roubar o HD e ligá-lo num outro computador para ler os ficheiros brutos.
  • • Solução Primária: Criptografia de Disco/Volume ou criptografia de objetos no Bucket (S3/Blob).
  • • Ferramenta de Nuvem: O KMS (Key Management Service) gera as chaves que encriptam estes discos.

🔒 TLS (Transport Layer Security)

O que é?

É o sucessor mais moderno e seguro do antigo SSL. Ele cria um "túnel" criptográfico para os dados enquanto viajam pela rede (internet pública ou dentro da própria VPC).

A Mágica Híbrida do TLS

Usa o "melhor dos dois mundos". Usa Criptografia Assimétrica (Chave Pública/Privada) no momento do "Handshake" (aperto de mão) para combinar as senhas em segurança, e depois muda para Criptografia Simétrica para encriptar a transferência pesada de dados de forma rápida.

🔑 KMS (Key Management Service)

Em nuvem, "gerir senhas de encriptação" é tão perigoso quanto a própria ameaça de invasão. Se você perder a chave que encriptou seu banco de dados (que está em repouso), você perde todos os seus dados para sempre. O KMS é o serviço centralizado da nuvem para criar, guardar e rodar as chaves de criptografia.

Geração Segura

Usa HSMs (Hardware Security Modules) certificados por entidades de segurança (FIPS 140-2) para criar números perfeitamente aleatórios impossíveis de adivinhar.

Rotação Automática

O KMS pode ser configurado para gerar uma chave nova a cada ano de forma automática e "re-encriptar" tudo por baixo dos panos, cumprindo normativos de segurança bancária.

Envelope Encryption

A técnica avançada de performance: A AWS cria uma chave gigante de segurança. Usa essa chave para encriptar os dados (Chave de Dados), e depois usa a Chave Mestre (KMS) para encriptar a própria chave de dados!

🚨 Pegadinhas Mortais da Banca (CEBRASPE)

  • O Nome da Ferramenta errada:
    A banca diz: "Para proteger os arquivos sensíveis de um sistema do governo contra o roubo físico dos discos no datacenter da nuvem, a equipa deve forçar o uso do protocolo TLS (Transport Layer Security) nos Buckets (Armazenamento de Objetos)."
    Falso! Roubo físico de HD ataca dados em Repouso (At Rest). O TLS protege apenas dados em Trânsito (quando "voam" pela rede). Para o disco, a solução correta é Server-Side Encryption gerido pelo KMS.
  • KMS x Senhas Locais (Exportação):
    "Uma grande vantagem do AWS KMS é permitir que o administrador de sistemas do TCE-MA exporte a Chave Mestre (Customer Master Key - CMK) e a guarde num ficheiro de texto localmente num servidor do tribunal como backup."
    Falso Absoluto! As chaves mestres do KMS (CMKs) administradas na nuvem JAMAIS, sob nenhuma hipótese, saem (são exportadas) do KMS de forma desencriptada. Elas nascem lá e morrem lá dentro do HSM. O KMS processa os pedidos de encriptação internamente e só envia os "dados já encriptados" de volta. Essa inexportabilidade é a fundação da segurança do KMS.
  • SSL vs TLS:
    A banca tentará te enganar sugerindo que "o SSL (Secure Sockets Layer) versão 3.0 ainda é o protocolo de eleição oficial para proteger dados na nuvem devido à sua alta performance."
    Falso! O SSL é antigo, obsoleto e foi deprecado devido a graves falhas de segurança (como o bug POODLE). O TLS (1.2 e 1.3) é o único padrão contemporâneo válido e exigido em normativas de cibersegurança do governo federal para substituir o velho SSL.

1. Criptografia em Trânsito (Data in Transit)

Quando os dados se movimentam (seja entre o telemóvel do cidadão e o balanceador de carga do TCE, ou entre a máquina virtual X e o banco de dados Y dentro do próprio datacenter fechado), eles estão num estado vulnerável a interceções. A isso chamamos "Dados em Trânsito".

A Resposta Definitiva: TLS

A proteção padrão para o ambiente web (HTTPS) é o TLS (Transport Layer Security), o sucessor histórico do SSL. No modelo de nuvem, a terminação de TLS (TLS Termination) geralmente acontece no Load Balancer (Application Load Balancer) ou numa CDN (como CloudFront/Cloudflare).

O Paradoxo da Performance no "Backend"

Uma decisão clássica de arquitetura de nuvem: O Load Balancer descriptografa o TLS (gastando muita CPU com criptografia) e analisa o URL. A partir dali, para ganhar velocidade e aliviar a carga nos servidores backend da VPC (já que é uma rede privada interna), o tráfego poderia seguir em formato "puro" (sem TLS). Mas atenção: A tendência atual de "Zero Trust" (Confiança Zero) obriga que mesmo o tráfego de backend interno da VPC seja criptografado via TLS de ponta a ponta (End-to-End Encryption).

2. Criptografia em Repouso (Data at Rest)

Dados em Repouso referem-se aos arquivos salvos permanentemente num meio de armazenamento (HDDs de servidores EC2, Armazenamento em Bloco EBS ou em Objetos como o Amazon S3).

Neste cenário, não adianta ter TLS na ligação. Se um intruso subornar um funcionário do datacenter da Amazon para ele desconectar um disco físico e sair pela porta principal, o invasor terá tudo. A defesa em profundidade obriga a Criptografia no Disco.

SSE (Server-Side Encryption) vs CSE (Client-Side Encryption)

  • CSE (Client-Side Encryption): O cliente (programador do TCE) encripta o PDF na memória RAM do próprio servidor do tribunal, usando uma biblioteca do Java/C#. Ele envia o "lixo ininteligível" pela rede (já protegido) para salvar no provedor da nuvem. A AWS guarda o arquivo e não tem a menor ideia do que está lá dentro nem de qual foi a chave usada.
  • SSE (Server-Side Encryption): A forma mais comum. Você envia o PDF para o serviço da Amazon S3 ou pro Banco de Dados RDS. Antes de guardar o ficheiro no HDD magnético, o próprio serviço da nuvem encripta a informação no lado deles, recorrendo ao serviço KMS, de forma transparente.

3. KMS (Key Management Service)

O AWS KMS (ou Azure Key Vault) é o serviço onde nascem e operam as chaves criptográficas da nuvem. Ele simplifica enormemente a complexidade de gerir "segredos" para milhares de servidores.

O Conceito de Chave Mestra e Envelope Encryption

Na nuvem, lidamos com Petabytes de dados (milhões de GBs). Criptografar toda essa montanha usando uma única chave enviada para um servidor KMS pela internet demoraria dias e paralisaria o sistema. Para contornar a lentidão física de comunicação, utiliza-se a técnica de Envelope Encryption.

  1. O KMS possui uma Chave Mestre do Cliente (CMK) super forte. Esta chave NUNCA SAI das entranhas físicas de hardware do KMS.
  2. Quando o seu servidor (VM) precisa encriptar um arquivo pesado de 10GB, ele pede uma "chave descartável" (Data Key) para o KMS.
  3. O KMS gera a Data Key. Mas ele devolve para a sua VM duas versões desta chave: uma versão em texto simples (que a sua VM vai usar na memória RAM instantaneamente para encriptar o ficheiro gigante e depois vai apagá-la) e uma versão encriptada (que foi selada pela Chave Mestre lá dentro do KMS).
  4. Sua VM anexa a versão encriptada da Data Key (o "envelope") ao lado do ficheiro gigante criptografado e guarda isso no disco.
  5. Se alguém roubar o disco, ele não terá acesso à chave mestra, por conseguinte não poderá "abrir o envelope" para decifrar a Data Key, e jamais conseguirá ler o arquivo!

🏷️ 4. CMK (Gerenciado pela AWS vs Cliente)

As bancas também questionam os "Sabores" das chaves no KMS. Existem três modelos fundamentais de CMK (Customer Master Keys):

  • AWS Managed Key (Gerenciada pela AWS): É gratuita. A AWS cria as chaves para proteger seus buckets (SSE-S3). Você não escolhe quando a chave rotaciona (muda a senha periodicamente) nem pode alterar suas permissões; a AWS assume 100% da gestão do cofre para ti.
  • Customer Managed Key (Gerenciada pelo Cliente): É paga. O material criptográfico é gerado pela AWS, mas o cliente tem total domínio. Ele configura a política do IAM dizendo quais utilizadores ou instâncias da nuvem podem aceder àquela chave específica, pode desativá-la e pode determinar que a rotação ocorra exatamente de 1 em 1 ano, cumprindo normas estritas.
  • BYOK (Bring Your Own Key): Usado em clientes de extrema paranoia de segurança (setor militar). Em vez da AWS gerar a matemática da chave no HSM dela, o banco local gera a chave nos seus HSMs on-premises e importa (faz o upload seguro) para a nuvem da AWS.
