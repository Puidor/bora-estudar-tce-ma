Tipos de Armazenamento

💾 O Trio do Armazenamento

OS ALVOS DE PROVA
🧱

Bloco (Block)

O "Disco Rígido" tradicional (HD/SSD).

  • • Funciona como um drive não formatado (SAN).
  • • Baixa latência, alta performance.
  • • Único capaz de rodar Sistema Operacional e Bancos de Dados Relacionais intensos.
  • • Geralmente ligado a 1 única VM por vez.
  • • Exemplo: Amazon EBS, Azure Managed Disks.
📁

Arquivo (File)

A "Pasta Compartilhada" na rede.

  • • Sistema de arquivos com Hierarquia Real (pastas e subpastas).
  • • Pode ser compartilhado (montado) por dezenas de VMs ao mesmo tempo.
  • • Usa protocolos de rede como NFS ou SMB.
  • • Exemplo: Amazon EFS, Azure Files.
📦

Objeto (Object)

O "Depósito Infinito" para a Web.

  • • Estrutura PLANA. Não tem hierarquia de pastas (é tudo salvo em "Buckets" usando uma chave).
  • • Rico em metadados customizáveis.
  • • Acessado via API Web (HTTP/REST).
  • • Excelente para Backups, Fotos, Vídeos, Big Data.
  • • Exemplo: Amazon S3, Google Cloud Storage.

🚨 Pegadinhas Mortais da Banca (CEBRASPE)

  • S.O. no Armazenamento de Objetos:
    A banca diz: "O administrador de nuvem deve criar um bucket no Amazon S3 (Objeto) e instalar o sistema operacional Linux dentro dele para garantir alta durabilidade."
    Falso! O Armazenamento de Objetos não suporta execução de Sistemas Operacionais. Eles exigem comunicação de baixíssima latência com blocos raw de disco (I/O intensivo), o que só o Armazenamento em Bloco fornece.
  • Hierarquia de Pastas no S3:
    "O armazenamento de objetos permite criar uma hierarquia profunda de subdiretórios e pastas lógicas idêntica a de um sistema Windows tradicional."
    Falso! Armazenamento de objetos é PLANO (Flat). Não existem pastas reais. O provedor até mostra uma "pasta" na interface gráfica por conveniência visual se o nome do arquivo for `fotos/férias/img.jpg`, mas para o sistema, aquilo é apenas uma única string inteira (chave) ligada a um objeto. Se quiser pastas reais, tem que usar Armazenamento de Arquivos.
  • Alteração de Dados em Objetos:
    "Para alterar apenas uma letra de um arquivo de texto de 5GB salvo no Armazenamento de Objetos, o sistema realiza a modificação daquele pequeno setor do disco, poupando banda."
    Falso! Objetos são Imutáveis. Não se pode editar "um pedacinho" de um objeto. Se quiser alterar uma letra, terá que fazer o upload de um novo objeto de 5GB para substituir o antigo inteiro! Quem permite a alteração de pequenos pedaços (setores) é o Armazenamento em Bloco.

1. Armazenamento em Bloco (Block Storage)

O armazenamento em bloco é o mais semelhante ao disco rígido (HDD/SSD) físico que existe dentro do seu computador. Ele abstrai a complexidade do hardware e entrega "volumes" crus para o Sistema Operacional da sua Máquina Virtual (EC2/VM) na nuvem.

Neste modelo, os dados são divididos em blocos de tamanho fixo (ex: 4KB ou 8KB). O Sistema Operacional pega nesse disco "vazio", formata-o com um File System conhecido (como NTFS para Windows ou ext4 para Linux) e passa a gerir como guarda e onde guarda as coisas.

Principais Características e Casos de Uso:

  • Atualização Parcial: É altamente eficiente se você precisa alterar apenas pequenos pedaços de um ficheiro gigante, pois ele altera somente o "bloco" correspondente e não o arquivo inteiro.
  • Baixa Latência: A comunicação é rapidíssima (via rede Fibre Channel/iSCSI ou internamente na nuvem).
  • Casos de Uso Críticos: É mandatório para Sistemas Operacionais (Boot Volume) e para Bancos de Dados Relacionais e Transacionais que requerem leitura e gravação constantes.
  • Isolamento (Regra Geral): Geralmente, um disco de bloco (como o AWS EBS) só pode estar montado/ligado a uma única Máquina Virtual de cada vez, protegendo contra corrupção de sistema de arquivos.

2. Armazenamento de Arquivos (File Storage)

Este é o modelo clássico de NAS (Network Attached Storage) ou de "Partilha de Rede" corporativa. Ele preserva a estrutura de diretórios (hierarquia de pastas) que os humanos estão acostumados a usar.

Características Principais:

  • Montagem Concorrente: A grande vantagem em relação ao Bloco é que um único armazenamento de arquivos (como o AWS EFS) pode ser montado e acedido por milhares de Máquinas Virtuais simultaneamente.
  • Protocolos de Rede Padrão: A comunicação não é crua; ela usa protocolos já estabelecidos que os S.O. conhecem de origem, como NFS (para Linux) ou SMB/CIFS (para Windows).
  • Casos de Uso: Diretórios iniciais de utilizadores de empresas, repositórios de media para servidores web (vários servidores do Auto Scaling partilham as mesmas imagens base), gestão de conteúdos e partilha de arquivos antigos.

3. Armazenamento de Objetos (Object Storage)

Nascido para a nuvem, o armazenamento de objetos foi criado para resolver a necessidade de escala massiva da internet, onde ter uma "árvore de pastas" com 10 bilhões de ficheiros deixaria a pesquisa insuportavelmente lenta.

Num Storage de Objetos (como o Amazon S3 ou o Azure Blob), os dados e os seus metadados (quem criou, que cor tem, etc.) são agrupados e encapsulados juntamente com um Identificador Único (ID / Key). A estrutura é 100% Plana (um gigantesco balde onde se atiram as coisas, sem subpastas reais).

Comunicação e Imutabilidade

Ao contrário do Bloco e do Arquivo que comunicam pelos drivers do S.O., o Objeto comunica via Internet (APIs HTTP RESTful). Você acede ao objeto com requisições PUT, GET, DELETE.

A Regra da Imutabilidade: Em armazenamento de objeto, os ficheiros são salvos como peças únicas. Não é possível entrar e alterar apenas uma linha de um ficheiro de registo (log) guardado. É necessário criar uma versão inteira nova (Upload total) ou apagá-lo. Isso torna-o péssimo para Bancos de Dados Transacionais, mas perfeito para Data Lakes, Big Data, Imagens, Vídeos e Backups Inalteráveis.
