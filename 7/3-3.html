Balanceamento de Carga e Autoescalonamento

🤝 A Dupla Dinâmica

O Gerente de Trânsito Load Balancer (LB)

Recebe as requisições dos utilizadores (tráfego da web) e distribui uniformemente entre os servidores saudáveis. Ele garante a Alta Disponibilidade, mas NÃO cria novas máquinas.

A Fábrica de Máquinas Auto Scaling (ASG)

Monitoriza métricas (ex: CPU > 80%). Quando o limite é atingido, ele cria (provisiona) ou destrói (encerra) instâncias automaticamente. Depois de criar, ele regista a máquina nova no Load Balancer.

⚖️ Tipos de Balanceadores (Camadas OSI)

MUITO COBRADO!

Network Load Balancer (Camada 4)

Burro, porém extremamente rápido.

  • • Atua na camada de Transporte.
  • • Toma decisões baseadas apenas em IPs e Portas (TCP/UDP).
  • • Não consegue ler o conteúdo (HTML/URL) da requisição.
  • • Ideal para milhões de requisições por segundo com latência ultrabaixa.

Application Load Balancer (Camada 7)

Mais lento, porém inteligente.

  • • Atua na camada de Aplicação.
  • • Toma decisões baseadas em HTTP/HTTPS (Lê URLs, Headers, Cookies).
  • • Consegue fazer Roteamento Baseado em Caminho (ex: /api vai para o Servidor A, /imagens vai para o Servidor B).

🧠 Conceitos de Prova (Vocabulário)

Health Checks (Verificações de Integridade)

O Load Balancer "pinga" continuamente os servidores (ex: chama uma página /status a cada 10 segundos). Se o servidor não responder `HTTP 200 OK` após X tentativas, o LB marca a máquina como Unhealthy (Não Saudável) e para de mandar utilizadores para ela, redirecionando o tráfego para os sobreviventes.

Sticky Sessions (Afinidade de Sessão)

Por defeito, o LB manda cada clique do utilizador para uma máquina diferente (Round Robin). Com Sticky Sessions, o LB usa um Cookie para garantir que o Utilizador "A" seja sempre direcionado para o Servidor "1" durante toda a sua visita (útil em aplicações legadas que guardam a sessão na RAM da máquina).

🚨 Pegadinhas Mortais da Banca (CEBRASPE)

  • Quem cria a máquina?
    A banca diz: "O Load Balancer, ao identificar um pico de acesso no site, provisiona imediatamente novas instâncias virtuais para suportar a carga extra."
    Falso! O Load Balancer não cria máquinas, ele apenas distribui o tráfego entre as máquinas que já existem. Quem monitoriza e manda criar novas instâncias é o Auto Scaling Group.
  • Cuidado com o Scale-IN (Redução):
    "As políticas de Auto Scaling garantem o aumento da infraestrutura (Scale-Out) para prevenir inatividade, mas a redução das máquinas após o pico deve ser feita manualmente pelo administrador de sistemas."
    Falso! O Auto Scaling possui políticas tanto para aumentar (Scale-Out) quanto para diminuir (Scale-In) os recursos automaticamente, otimizando assim os custos financeiros (pilar da Elasticidade).
  • Roteamento por URL no Network Load Balancer:
    "Para direcionar as requisições do caminho '/imagens' para um servidor de media e '/api' para um servidor backend, utiliza-se o balanceamento de camada 4."
    Falso! Camada 4 (Transporte) só entende IP e Porta. Para ler o caminho da URL (`/imagens`), o balanceador precisa "abrir" o pacote HTTP, o que exige um Application Load Balancer (Camada 7).

1. Balanceamento de Carga (Load Balancing)

O Balanceamento de Carga é o processo metodológico de distribuição de tráfego de rede concorrente entre um grupo de servidores backend. O objetivo principal é garantir que nenhum servidor singular fique sobrecarregado (o que causaria lentidão ou queda), aumentando assim a responsividade e a Alta Disponibilidade da aplicação.

Algoritmos de Distribuição (Como ele escolhe quem atende?)

Bancas mais exigentes podem questionar a lógica interna do balanceador. Os algoritmos mais comuns são:

  • Round Robin: As requisições são distribuídas sequencialmente (cíclico). O servidor 1 atende o pedido A, o servidor 2 atende o pedido B, o servidor 3 atende o pedido C, e o servidor 1 atende o pedido D. Simples e previsível.
  • Least Connections (Menos Conexões Ativas): O balanceador verifica quem tem a menor quantidade de sessões abertas no momento e encaminha a requisição nova para o servidor "mais ocioso". É excelente quando as requisições duram tempos variados.
  • IP Hash: O balanceador aplica uma função matemática (Hash) sobre o endereço de IP do cliente. Isso garante que o mesmo IP sempre caia na mesma máquina (muito usado como forma alternativa de Sticky Session).

2. Camadas OSI no Balanceamento

Entender o Modelo OSI é fundamental. Na nuvem moderna, dividimos os balanceadores em duas categorias principais de arquitetura:

Layer 4 (Camada de Transporte)

O Network Load Balancer não analisa o conteúdo da mensagem. Ele atua cegamente encaminhando pacotes com base no IP de origem/destino e na porta (TCP/UDP). Por não ter que descriptografar pacotes SSL/TLS ou ler Cabeçalhos HTTP, ele é capaz de processar milhões de requisições por segundo mantendo latências na ordem dos milissegundos. É a escolha ideal para jogos online ou streaming de dados brutos.

Layer 7 (Camada de Aplicação)

O Application Load Balancer atua no topo da pilha. Ele abre o pacote HTTP/HTTPS, analisa cookies, cabeçalhos, e o URL exato (ex: meusite.com/videos). Com essa "inteligência", ele pode tomar decisões complexas de roteamento (ex: "Se for via smartphone Android, mandar para o servidor Y; se for PC, mandar para servidor X"). É mais lento que o L4, mas indispensável para microsserviços modernos.

3. Autoescalonamento (Auto Scaling)

O Auto Scaling é o serviço responsável por garantir que o sistema tenha a quantidade exata (e necessária) de instâncias executando a aplicação. Ele concretiza o pilar da Elasticidade do NIST.

Políticas de Escalonamento (Gatilhos)

Como o Auto Scaling decide criar ou destruir uma máquina? Ele depende de métricas (via sistemas de monitorização como o CloudWatch na AWS). As estratégias são:

  • Rastreamento de Meta (Target Tracking): Você define um alvo contínuo. "Quero que a utilização média de CPU do meu grupo de servidores fique sempre em torno de 50%". O Auto Scaling adiciona ou remove máquinas sozinho para manter a balança em 50%.
  • Escalonamento por Etapas (Step Scaling): Responde a alertas específicos. Ex: "Se a memória RAM passar de 80%, crie 2 máquinas. Se a RAM passar de 90%, entre em pânico e crie logo 5 máquinas extras".
  • Escalonamento Agendado (Scheduled Scaling): Escalonamento preditivo com base no calendário (cronograma). Você sabe que o sistema do TCE-MA será bombardeado pelos autarcas no dia 31 do mês às 08:00h. Você agenda o Auto Scaling para iniciar 10 máquinas novas às 07:45h.
Resfriamento (Cooldown Period)
Após criar (ou destruir) uma máquina, o Auto Scaling entra num período de resfriamento (ex: 300 segundos). Durante esse tempo, ele pausa novas avaliações. Isso previne que o sistema fique "louco" e crie 50 máquinas de uma vez antes de a primeira máquina nova ter tido a chance de ligar e aliviar a CPU do grupo.

🔗 4. A Dança: Como os dois se conectam?

Em uma arquitetura tolerante a falhas na nuvem, o Load Balancer e o Auto Scaling Group trabalham acoplados.

Se o Health Check do Load Balancer perceber que a "Máquina B" parou de responder, o Load Balancer para de mandar tráfego para ela. Ele avisa o Auto Scaling de que aquela máquina está corrompida. O Auto Scaling a destrói impiedosamente (Termina a instância) e liga uma "Máquina C" limpa e nova. Assim que a Máquina C inicializar, o Auto Scaling a regista no Load Balancer, que retoma a distribuição do tráfego. Essa é a magia dos sistemas auto-recuperáveis (Self-healing).
