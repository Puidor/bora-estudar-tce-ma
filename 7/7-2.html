Data Lakes e Processamento Distribuído

🌊 O que é um Data Lake?

Um Data Lake (Lago de Dados) é um repositório centralizado que permite armazenar todos os dados da organização (estruturados e não estruturados) em qualquer escala e em seu estado BRUTO (Raw). Você não precisa formatar os dados antes de salvá-los; você simplesmente despeja tudo lá.

O Alvo da Prova: A Base de Armazenamento Na nuvem, 99% dos Data Lakes são construídos usando Armazenamento de Objetos (como o Amazon S3, Azure Data Lake Storage ou Google Cloud Storage) devido ao custo baratíssimo e escalabilidade infinita.

🥊 Data Lake vs Data Warehouse

CAI MUITO EM PROVA!
Característica Data Lake (O Lago) Data Warehouse (O Armazém)
Tipos de Dados Estruturados, semi-estruturados e NÃO estruturados (Vídeos, Logs, JSON, PDFs). Apenas dados Estruturados (Tabelas Relacionais, SQL).
Estado do Dado BRUTO (Raw). É mantido exatamente como foi gerado. REFINADO. Já foi limpo, transformado e tratado (ETL).
Processamento / Esquema Schema-on-Read (O esquema é aplicado apenas na hora de ler o dado). Schema-on-Write (Você tem que criar a tabela e o esquema ANTES de salvar o dado).
Usuários Alvo Cientistas de Dados (Machine Learning) e Engenheiros de Dados. Analistas de Negócios (Business Intelligence - BI) e Executivos.

⚡ Processamento Distribuído de Big Data

Você não consegue processar 50 Terabytes num único computador. O Processamento Distribuído fatia o problema em pedaços pequenos, distribui para um *Cluster* (grupo de servidores) e depois junta os resultados. Os dois gigantes desta área são:

Hadoop (MapReduce)

O pioneiro. Ele divide o trabalho (Map) e consolida os resultados (Reduce). É super resiliente, mas é LENTO porque salva os dados parciais no Disco Rígido entre cada etapa do cálculo.

Apache Spark

O substituto moderno. Até 100x mais rápido que o Hadoop pois processa tudo em Memória RAM (In-Memory Processing) utilizando RDDs (Resilient Distributed Datasets).

🚨 Pegadinhas Mortais da Banca (CEBRASPE)

  • O Data Lake "substituiu" o Data Warehouse?
    "Com o advento do Data Lake na computação em nuvem, os Data Warehouses tornaram-se tecnologias obsoletas, devendo ser integralmente substituídos nas empresas modernas."
    Falso! Eles são Complementares! O Data Lake absorve tudo barato. Depois, ferramentas de ETL puxam do Data Lake os dados que realmente importam para o negócio, limpam e salvam no Data Warehouse para que os diretores usem em relatórios rápidos de BI. (A união moderna dos dois chama-se Data Lakehouse).
  • O Problema do "Pântano de Dados" (Data Swamp):
    "Um Data Lake garante automaticamente a qualidade, catálogo e descoberta fácil dos dados apenas pelo ato de ingeri-los na nuvem."
    Falso! Se você apenas despejar dados no Data Lake sem criar um Catálogo de Metadados (Data Catalog) e sem Governança, o seu lago limpo vai transformar-se num Data Swamp (Pântano de Dados): um lixão digital onde ninguém consegue encontrar ou confiar em nenhuma informação.
  • Schema on Write no Data Lake:
    "Para garantir consistência, o Data Lake opera nativamente com o paradigma Schema-on-Write, validando a estrutura do arquivo antes da inserção."
    Falso! Isso é característica de bancos relacionais e Data Warehouses. O Data Lake utiliza Schema-on-Read (O esquema só é inferido e aplicado no momento em que você executa uma query ou lê o arquivo).

1. A Filosofia do Data Lake (Lago de Dados)

Historicamente, se uma organização do setor público (como o TCE-MA) quisesse guardar dados, ela precisava saber exatamente o formato desse dado para criar uma Tabela de Banco de Dados. Isso causava um problema: muitos dados valiosos (como logs de servidores, imagens escaneadas de processos e PDFs não estruturados) eram jogados fora, porque processá-los e estruturá-los era muito caro.

O Data Lake surgiu para resolver o desafio do Big Data. A lógica inverteu-se: Armazene tudo agora de forma bruta e barata; decida como estruturar e analisar apenas quando precisar.

A Arquitetura de Nuvem do Data Lake

Qualquer provedor de nuvem baseia os seus Data Lakes num serviço de Armazenamento de Objetos. Na AWS, a fundação do Data Lake é sempre o Amazon S3. No Azure, é o Azure Data Lake Storage (ADLS), que nada mais é do que o Azure Blob Storage com suporte a um sistema de arquivos hierárquico (HDFS compatível) por cima.

2. Estruturando o Caos: Data Catalog e ETL/ELT

Como mencionado na aba de pegadinhas, um Data Lake sem governança vira um Data Swamp (Pântano). Para evitar isso, usa-se um serviço de Catálogo de Dados (Data Catalog).

Serviços como o AWS Glue usam robôs (Crawlers) que "rastejam" pelo Data Lake, abrem os arquivos CSV, JSON ou Parquet, inferem o que está lá dentro e criam um dicionário de metadados. Sem esse dicionário, ferramentas de análise ficam cegas.

De ETL para ELT

Antigamente (na era dos Data Warehouses locais), a regra era ETL: Extract (Extrair), Transform (Transformar e limpar a memória num servidor intermediário), Load (Carregar no banco de destino).

A mudança do Data Lake (ELT): Na nuvem, o paradigma preferido é o ELT. Extract (Extrair da fonte), Load (Carregar bruto e rápido direto no Data Lake), e só depois Transform (Transformar o dado usando o poder massivo de processamento distribuído da própria nuvem, como o Apache Spark, para criar as visões refinadas).

3. Processamento Distribuído

Depois de armazenar Petabytes de dados, como processá-los? O processamento distribuído (frameworks nativos para Big Data) é a resposta.

A. Hadoop e o paradigma MapReduce

O Apache Hadoop dividia problemas colossais em milhares de computadores normais. O coração dele era o HDFS (sistema de arquivos próprio) e o motor de cálculo MapReduce. O Map filtrou e dividiu os dados; o Reduce somou e consolidou as respostas. O problema do Hadoop é que, entre a fase do Map e a do Reduce, ele precisava salvar o estado no disco rígido (Disk I/O). Para Analytics moderno e Machine Learning, isso revelou-se demasiado lento.

B. Apache Spark (A Revolução)

O Spark não substitui o armazenamento do Hadoop, mas substitui o motor de cálculo (MapReduce). A grande sacada do Spark é o processamento Em Memória (In-Memory) utilizando estruturas imutáveis chamadas RDDs e DataFrames. Ele faz todo o trabalho pesado na Memória RAM dos servidores do cluster e evita ao máximo gravar nos discos rígidos físicos. É o padrão oficial da indústria atual.

☁️ 4. Serviços de Nuvem Associados (Nomes de Prova)

Bancas de nível sênior podem cobrar de que forma você implementa um Spark ou um Hadoop num dos "Big 3" sem ter que instalar tudo manualmente (PaaS/SaaS).

  • Amazon EMR (Elastic MapReduce): A plataforma de Big Data da AWS. Permite que você "alugue" um cluster Hadoop/Spark completo que se liga em minutos. Você analisa os dados que estão no S3 e pode destruir o cluster após a análise (pagando pouco).
  • Amazon Athena: Serviço Serverless genial. Ele permite que você digite comandos SQL padrão e os execute diretamente sobre ficheiros brutos (CSV/JSON/Parquet) que estão largados lá dentro do Amazon S3, sem nem precisar de ter um banco de dados ligado.
  • GCP Dataproc e BigQuery: O Dataproc é o equivalente ao EMR da AWS no Google (para rodar Spark/Hadoop gerido). O BigQuery é o Data Warehouse Serverless massivo do Google, considerado líder da indústria em velocidade analítica.
