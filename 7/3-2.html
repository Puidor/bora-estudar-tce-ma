Serverless e Event-driven

⚡ Serverless (Sem Servidor)

Em arquiteturas Serverless, existem servidores físicos, mas o desenvolvedor não precisa provisionar, escalar, atualizar (SO) ou gerenciar nenhum deles. Você apenas escreve o código e o provedor de nuvem (AWS, Azure) faz o resto.

Escala a Zero Se ninguém acessar seu sistema, a infraestrutura "desliga" automaticamente.
Pay-per-Use Você só paga pelos milissegundos exatos em que seu código esteve executando.
FaaS (Function) Modelo mais comum. Seu código vira uma "Função". Ex: AWS Lambda, Google Cloud Functions.

📨 Arquitetura Event-driven (Orientada a Eventos)

DESACOPLAMENTO!

Sistemas reagem a "mudanças de estado" (Eventos) em vez de esperarem consultas síncronas.

Produtor Gera o Evento Ex: "Cliente finalizou uma compra no site."
➔
Roteador / Broker Fila ou Tópico Ex: Amazon SNS, Kafka, RabbitMQ. Filtra e distribui.
➔
Consumidor Reage ao Evento Ex: Microsserviço de Estoque diminui o item; Serviço de Email envia nota fiscal.

💍 O Casamento: Serverless + Eventos

Serverless e Event-driven nasceram um para o outro. Uma Função Serverless (ex: AWS Lambda) fica "dormindo" (custando R$ 0,00). Quando um Evento acontece (ex: Um PDF é salvo no banco de dados), o roteador acorda a Função instantaneamente, ela processa o PDF e volta a dormir.

Vantagens combinadas:
  • ✅ Assincronismo: O usuário não fica esperando a tela carregar.
  • ✅ Custo-Efetivo: Evita pagar por servidores ligados de madrugada sem uso.
  • ✅ Auto-Escala mágica: Se 1.000 PDFs chegarem num segundo, o provedor liga 1.000 instâncias da função.

🚨 Pegadinhas Mortais da Banca (CEBRASPE)

  • O Fim dos Servidores?
    A banca diz: "No paradigma Serverless, a arquitetura física foi abolida, de modo que o processamento ocorre inteiramente nas interfaces de rede sem uso de hardware em data centers."
    Falso Absoluto! "Sem servidor" é apenas um nome comercial. Os servidores físicos, switches e roteadores continuam existindo nos galpões da AWS/Azure/GCP. A diferença é que a responsabilidade de gerir, atualizar SO e alocar espaço é 100% transferida para o provedor, ficando abstraída/transparente para o cliente.
  • O Custo é sempre menor?
    "Adotar arquitetura Serverless garante invariavelmente uma redução de custos operacionais em relação a instâncias IaaS (Máquinas Virtuais)."
    Falso! Serverless é cobrado por execução/milissegundo. Se você tiver uma aplicação que roda 100% do tempo, 24 horas por dia, num processamento pesado ininterrupto, Serverless vai sair MUITO MAIS CARO do que alugar uma VM com valor fixo mensal (IaaS).
  • Sincronismo em Event-Driven:
    "Na arquitetura Event-driven, o produtor do evento deve aguardar sincronicamente a resposta do consumidor para garantir a integridade da transação."
    Falso! A magia do Event-driven é o Assincronismo e o Desacoplamento. O produtor "joga" o evento no Barramento/Fila (Fire and Forget) e vai fazer outra coisa. Ele não sabe nem se o consumidor existe, quem dirá ficar esperando por ele.

1. Desmistificando o Serverless

O termo Serverless (Computação sem Servidor) não significa o fim do hardware. Ele se refere a um modelo de execução em computação em nuvem em que o provedor atua como o alocador dinâmico de recursos da máquina.

Na computação IaaS tradicional, você aluga uma Máquina Virtual (VM) de 16GB de RAM. Mesmo se o seu site não receber nenhuma visita de madrugada, você pagará por aquela máquina ligada. No Serverless, o provedor só cobra quando um bloco de código (Função) é executado, tarifando a memória usada e os milissegundos que o código demorou para rodar.

FaaS vs BaaS (Divisão Teórica)

A arquitetura Serverless geralmente engloba dois conceitos interligados:

  • FaaS (Function as a Service): É a execução de lógicas de negócio. Você faz o upload do seu código (Node.js, Python), e o provedor lida com a infraestrutura para executar. (Ex: AWS Lambda, Azure Functions).
  • BaaS (Backend as a Service): São serviços de terceiros que substituem componentes de infraestrutura tradicionais por APIs gerenciadas. (Ex: Em vez de instalar um Banco de Dados MySQL numa VM, você usa o Amazon DynamoDB ou o Firebase, onde você não provisiona a máquina do banco, apenas "salva e lê" dados, escalando ao infinito).

2. O Problema do "Cold Start" (Partida Fria)

Este é o calcanhar de Aquiles do Serverless e um termo técnico obrigatório para provas de TI.

Como o provedor "desliga" (escala para zero) os recursos quando ninguém está usando, o que acontece quando o primeiro cliente do dia acessa o seu sistema às 8h da manhã?

A infraestrutura precisa baixar o seu código, alocar um container isolado, iniciar a linguagem (Runtime como Java ou .NET) e aí sim rodar a sua função. Todo esse processo de "acordar a máquina" leva alguns segundos (às vezes 2 ou 3 segundos extras). A esse atraso dá-se o nome de Cold Start (Partida Fria).

Consequência: Se você tem uma aplicação que exige latência crítica e ultrabaixa (ex: sistema de negociação na Bolsa de Valores ou controle de mísseis), Serverless NÃO é recomendado, devido à imprevisibilidade do Cold Start.

Nota: Requisições subsequentes logo após a primeira irão bater no container que já está "quente" (Warm Start), sendo processadas em milissegundos.

3. Arquitetura Orientada a Eventos (EDA)

A Arquitetura Orientada a Eventos (Event-Driven Architecture) promove a produção, detecção, consumo de, e reação a eventos. Um evento pode ser definido como "uma mudança significativa no estado" (Ex: Carrinho de compras pago, Upload de vídeo concluído).

Padrões de Comunicação Event-Driven

  • Padrão Publisher/Subscriber (Pub/Sub): A infraestrutura de mensagens rastreia assinaturas (Subscribers). Quando um evento é publicado, ele é enviado (push) a todos os inscritos naquele "Tópico". Depois que o evento é recebido, ele não pode ser repetido/lido novamente. (Ex: Amazon SNS).
  • Event Streaming (Fluxo de Eventos): Os eventos são gravados em um log (um fluxo contínuo). Ao contrário do Pub/Sub clássico, os clientes podem ler (e reler) eventos passados de qualquer parte do fluxo a qualquer momento, pois os eventos são retidos por um tempo predeterminado (Ex: Apache Kafka, Amazon Kinesis).
  • Event Sourcing (Origem de Eventos): Em vez de salvar apenas o estado atual (Ex: "Saldo: R$ 100"), o banco de dados armazena a sequência inteira de eventos que levou a esse estado (Ex: "Depósito de R$ 50", "Depósito de R$ 50"). Isso permite reconstruir o estado passado da aplicação a qualquer momento ("Viagem no tempo").

🔗 4. O Risco Oculto: Vendor Lock-in

Ao adotar uma arquitetura puramente Serverless e Event-Driven, os desenvolvedores tendem a utilizar fortemente as soluções nativas do provedor (AWS Lambda, AWS SNS, DynamoDB).

O problema arquitetural gerado por isso é o Vendor Lock-in (Aprisionamento Tecnológico). Como o seu código passa a depender de APIs fechadas, ferramentas de monitoramento e gatilhos que só existem na AWS, mudar o sistema inteiro para o Microsoft Azure ou GCP no futuro se tornará uma tarefa dolorosa, cara e de refatoração extrema (diferente de usar Containers/Docker, que rodam de forma igual em qualquer provedor de nuvem).
