Ferramentas de Migração

🧰 As 4 Famílias de Ferramentas

Migrar não é apenas "copiar e colar". Os provedores (focaremos nos nomes da AWS e Azure que dominam provas) criaram ferramentas específicas para cada tipo de ativo que você quer mover.

1. Descoberta (Discovery)

Instala-se agentes no datacenter antigo para monitorar as VMs por semanas e descobrir quem conversa com quem (mapeamento de dependências).

2. Migração de Servidores (VMs)

Faz a cópia exata do Sistema Operacional e das aplicações da máquina física para instâncias na nuvem (IaaS). Executa a estratégia "Rehost".

3. Migração de Bancos de Dados

Ferramentas que migram os dados de forma contínua, permitindo que a base antiga e a nova fiquem sincronizadas até o "corte" final.

4. Transferência Massiva de Dados

Para mover Petabytes/Exabytes. Usa appliances (maletas ou caminhões físicos) quando a internet é muito lenta para enviar tudo online.

A Família Snow (Transferência Física)

Enviar 50 Petabytes de dados pela sua internet corporativa de 1 Gbps levaria cerca de 12 anos. Para resolver isso, os provedores usam transferência física Offline.

A AWS envia uma maleta blindada à prova d'água (Snowball Edge) via Correios. O TI do órgão conecta essa maleta no rack local, copia os dados em alta velocidade, e os Correios levam a maleta fisicamente de volta para a AWS ligar no datacenter de destino.

🚨 Pegadinhas Mortais da Banca (CEBRASPE)

  • O DMS converte esquemas sozinho?
    A banca diz: "O AWS Database Migration Service (DMS) migra os dados de um banco de dados Oracle para um MySQL, convertendo automaticamente as tabelas e o código do banco."
    Falso! O DMS migra apenas os DADOS (as linhas). Se você estiver migrando para um motor diferente (Oracle para MySQL), você tem a obrigação de usar primeiro uma ferramenta de conversão de esquema, como a AWS SCT (Schema Conversion Tool), que cria as tabelas "vazias" no novo formato.
  • Zero Downtime na Migração de VMs:
    "O uso de ferramentas como o AWS Application Migration Service exige a paralisação do servidor de produção on-premises durante dias, até que a cópia na nuvem seja concluída."
    Falso! Essas ferramentas funcionam com Replicação Contínua em Bloco (Block-level replication). A máquina velha continua atendendo os clientes. Apenas no momento final (Cutover), a TI "vira a chave" alterando o DNS, o que gera um downtime de apenas alguns minutos (ou quase zero).
  • Transferência Offline por insegurança:
    "O AWS Snowball é utilizado principalmente porque transferir dados pela internet é considerado inseguro e vulnerável a interceptações."
    Falso! A internet pública, usando VPNs (IPSec) ou TLS, e conexões diretas (Direct Connect) são perfeitamente seguras. O Snowball é utilizado puramente por uma questão de Limitação de Banda de Rede e Tempo de Transferência para volumes na casa dos Terabytes/Petabytes.

1. Descoberta e Avaliação (Discovery Tools)

Antes de migrar, a TI precisa saber o que existe no datacenter velho. Muitas vezes, nem a própria empresa sabe quantos servidores tem, nem se o servidor do "Estoque" precisa conversar com o servidor do "Faturamento".

Ferramentas como o AWS Application Discovery Service ou Azure Migrate (Discovery) resolvem isso. Você instala um "Agente" em todas as máquinas locais e ele passa semanas escutando a rede. No final, ele gera um mapa visual mostrando: "A Máquina A depende da Máquina B. Você não pode migrar a A sem levar a B junto."

2. Migração de Servidores (VMs e Bare Metal)

Quando a estratégia escolhida é o Rehost (Lift and Shift), usamos ferramentas de replicação de máquinas. A ferramenta principal na AWS é o AWS MGN (Application Migration Service), que substituiu o antigo CloudEndure.

Como funciona a Mágica do "Quase Zero Downtime"?

  1. Instala-se um pequeno software (agent) no servidor legado do TCE-MA.
  2. O agente começa a ler as alterações diretamente no nível do bloco do disco rígido (Continuous Block-Level Replication) e espelha esses blocos, em tempo real, para uma área de *staging* na nuvem da AWS.
  3. O servidor do TCE continua trabalhando normalmente, atendendo os cidadãos.
  4. Quando todos os discos já estão 100% espelhados, a equipa de TI marca a janela de manutenção, clica num botão e a ferramenta "converte" aqueles dados brutos numa instância EC2 perfeita rodando na nuvem.
  5. Muda-se o DNS, e o acesso passa a cair na nuvem.

3. Migração de Bancos de Dados

Bancos de Dados são o coração sensível de qualquer sistema. Eles não podem ser desligados facilmente. A ferramenta clássica de prova é o AWS DMS (Database Migration Service).

Homogênea vs Heterogênea

  • Migração Homogênea: Migrar do Oracle On-Premises para o Oracle na Nuvem (Amazon RDS). É simples. O DMS apenas lê as linhas de um lado e escreve do outro.
  • Migração Heterogênea (Atenção redobrada!): A empresa quer fugir das licenças caras e vai migrar do Oracle (legado) para PostgreSQL (nuvem open-source).
    Nesse caso, a arquitetura da tabela é diferente! Primeiro, utiliza-se a ferramenta AWS SCT (Schema Conversion Tool) para traduzir o "esqueleto" do banco e as "Procedures" do formato Oracle para o formato Postgres. Só depois o DMS é ligado para despejar as informações (os dados reais) dentro das tabelas convertidas.
CDC (Change Data Capture) O DMS utiliza a tecnologia CDC para ler os logs de transação do banco velho. Mesmo durante semanas de migração de 10TB de dados, se um cliente fizer uma compra nova no banco velho, o CDC captura essa nova linha e empurra para a nuvem. Mantém os bancos sincronizados 24 horas por dia até ao momento do corte definitivo.

🚚 4. A Física da Transferência de Dados

Se a empresa precisar migrar 100 Petabytes (PB) de vídeos de câmeras de segurança antigas, as soluções de rede clássicas colapsam. O custo de manter Direct Connects apenas para uma migração única seria colossal.

A AWS criou a Família Snow para transferências offline (e as vezes edge computing).

  • AWS Snowcone: Um aparelho bem pequeno (parece um rádio relógio). Suporta dezenas de Terabytes. Leve o suficiente para ser enviado por drone, se necessário.
  • AWS Snowball Edge: Do tamanho de uma mala de viagem pequena. Suporta até ~80 Terabytes. O equipamento é criptografado fisicamente de fábrica (se alguém o roubar do camião dos correios, ele explode os dados logicamente ou é inútil).
  • AWS Snowmobile: A lenda (apareceu no lançamento puxado por um camião de 18 rodas). É literalmente um contêiner de transporte marítimo com um datacenter inteiro de discos dentro, suportando até 100 Petabytes de dados por caminhão. É acompanhado por seguranças armados.
