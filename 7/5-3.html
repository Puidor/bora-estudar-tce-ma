Observabilidade: Logs, Métricas e Traces

👁️ A Diferença Chave

Monitoramento

Responde à pergunta: "O sistema está quebrado?"
Foco no Sintoma. Mostra se um serviço parou de funcionar e aciona alertas (ex: "CPU chegou a 90%").

Observabilidade

Responde à pergunta: "POR QUE o sistema está quebrado?"
Foco na Causa. Capacidade de entender o estado interno de um sistema analisando apenas as suas saídas de dados.

🏛️ Os Três Pilares da Observabilidade

OS ALVOS DE PROVA

Logs

Registros textuais
O que aconteceu? São registos imutáveis com data/hora (timestamp) de eventos discretos. Se uma exceção ocorrer, o erro detalhado (Stack Trace) estará aqui.
Ferramentas: ELK Stack (Elasticsearch, Logstash, Kibana), Splunk, Datadog.

Metrics

Métricas
Qual é a tendência geral? Representações numéricas agregadas medidas ao longo do tempo. (Ex: "A média de tempo de resposta é 200ms", ou "A taxa de erro é 5%"). Ocupam pouca memória e disparam alertas.
Ferramentas: Prometheus, Grafana, AWS CloudWatch.

Traces

Rastreamento
Onde está o gargalo? Representa o ciclo de vida completo de uma única requisição ponta a ponta passando por N microsserviços. Ex: "Gastou 10ms na API Gateway, 50ms no Microsserviço de Pagamento e 2000ms travado no Banco de Dados".
Ferramentas: AWS X-Ray, Jaeger, Zipkin.

🔗 Como funciona o Tracing Distribuído?

Em um sistema de Microsserviços, um simples clique do usuário passa por 10 servidores diferentes. Se demorar, como saber qual servidor foi o culpado?

  1. Quando o utilizador faz o request, o API Gateway gera um Trace ID (ID de Correlação) único (ex: id-1234).
  2. Esse ID é inserido no Cabeçalho HTTP (Header) da requisição.
  3. O Microsserviço "A" recebe a requisição, anota nos seus Logs "Estou a processar o id-1234" e repassa o mesmo Cabeçalho para o Microsserviço "B".
  4. No final, a ferramenta de Tracing (ex: Jaeger) puxa os logs de todas as máquinas e monta um gráfico temporal (Cascata) unindo tudo o que tiver id-1234.

🚨 Pegadinhas Mortais da Banca (CEBRASPE)

  • Métricas Substituem Traces?
    A banca diz: "Em arquiteturas de microsserviços maduras, a correta implementação de métricas (Metrics) torna a implementação de rastreamento (Tracing) obsoleta."
    Falso! Eles têm propósitos diferentes e complementares. A Métrica diz: "O tempo de resposta do Checkout demorou 5 segundos (Alertar a equipa!)". Mas a Métrica não sabe dizer porquê. O Trace é quem vai dizer: "O tempo demorou 5 segundos porque a etapa de validar o cartão de crédito demorou 4.8 segundos no serviço de terceiros."
  • O Conceito de Observabilidade:
    "Observabilidade é o ato mecânico de coletar o uso de CPU, disco e memória RAM utilizando agentes instalados nos servidores."
    Falso! O ato de coletar as métricas brutas (CPU, RAM) define o Monitoramento (Monitoring). A Observabilidade é uma propriedade do sistema (o quão transparente ele é) que utiliza ferramentas de monitoramento para responder a problemas desconhecidos (os "unknown unknowns").
  • Elasticsearch/Kibana são Tracing?
    "O Kibana e o Logstash são as ferramentas padrão da indústria para realizar Tracing Distribuído."
    Falso! Embora o Kibana possa exibir dados de Tracing em algumas configurações, ele (e o ELK stack em geral) é mundialmente famoso como um concentrador de Logs (Registros). Para Tracing nativo, as bancas esperam que você conheça nomes como Jaeger, Zipkin ou AWS X-Ray.

1. O Problema Moderno: O Paradigma dos Microsserviços

Nos sistemas antigos (monolitos), quando a aplicação travava, o desenvolvedor apenas precisava se ligar remotamente por SSH ao servidor único, abrir um ficheiro de texto chamado error.log e procurar a linha com erro. Simples.

Na era moderna (Cloud e Kubernetes), a sua aplicação pode ser dividida em 50 microsserviços, rodando em 500 contêineres diferentes que são ligados e desligados dinamicamente a cada minuto (Auto Scaling). Se um cliente disser "Tentei pagar e deu erro", onde você vai procurar o log? A máquina que atendeu o cliente pode até já ter sido destruída e não existir mais! É aí que nasce a disciplina e engenharia de Observabilidade.

2. A Fundo nos Três Pilares (MELT)

1. Métricas (Metrics)

Métricas são valores numéricos amostrados e agrupados. Elas medem o desempenho ou o estado de um sistema em um momento específico, como uso de CPU, latência ou quantidade de erros 404 HTTP (Ex: "Taxa de erro subiu para 200/segundo").

  • Pró: Ocupam um espaço minúsculo de armazenamento e são perfeitas para disparar alertas SMS/Email (ex: PagerDuty) e criar dashboards executivos.
  • Contra: Não lhe dizem o motivo do erro.

2. Logs (Registros)

Logs são registros detalhados (texto puro ou JSON) de eventos que aconteceram em uma aplicação (Ex: [12:00:05] ERROR: Cannot connect to database at 10.0.1.5). Em sistemas distribuídos, a prática obrigatória é a Centralização de Logs.

Como funciona a Centralização? Os contêineres enviam todos os seus logs de texto (stdout/stderr) não para um arquivo local, mas sim para a rede usando agentes como o Fluentd ou Logstash. Esses agentes enviam o texto para um indexador gigante (ex: Elasticsearch), onde a equipe pode pesquisar palavras-chave como se estivessem no Google.

3. Traces (Rastreamento)

Traces amarram o caos. Enquanto um Log diz "O Serviço C estourou memória", o Trace diz "O Serviço C estourou memória quando processava o ID 1234, que foi originado no Serviço A, passou pelo Serviço B e chegou ao C".

Anatomia do Trace: O Trace completo (toda a jornada) é composto por Spans (os "pulos" individuais em cada serviço). Um Trace contém um Trace ID único, e cada Span diz quanto tempo demorou o seu pedaço de trabalho.

3. OpenTelemetry: A Padronização do Mercado

No passado, cada ferramenta de monitoramento exigia que você programasse o seu código utilizando a biblioteca proprietária deles. Se você quisesse mudar do Splunk para o Datadog, teria que reescrever boa parte do seu código-fonte para trocar os agentes emissores.

A resposta da indústria (e figurinha certa em concursos sofisticados) foi o OpenTelemetry (OTel). Ele é um framework e padrão aberto (incubado pela Cloud Native Computing Foundation - CNCF) que unificou como as Métricas, Logs e Traces devem ser gerados e coletados.

Impacto Arquitetural: Com o OpenTelemetry, a organização utiliza bibliotecas agnósticas (neutras) na sua aplicação. O código apenas "emite o dado". O OpenTelemetry Collector apanha o dado e, a partir daí, pode enviá-lo para qualquer ferramenta de análise paga do mercado, eliminando o Vendor Lock-in de observabilidade.

📊 4. Os Quatro Sinais Dourados (Four Golden Signals)

Um dos livros mais famosos da área é o "Site Reliability Engineering" (Engenharia de Confiabilidade de Sites) do Google (SRE). O Google definiu quais são as 4 métricas absolutas que devem ser monitorizadas em aplicações orientadas ao utilizador:

  • 1. Latência (Latency): O tempo que leva para atender a uma solicitação (mesmo quando a requisição é bem-sucedida ou falha). O atraso.
  • 2. Tráfego (Traffic): A demanda atual sobre o seu sistema, frequentemente medida em solicitações HTTP por segundo.
  • 3. Erros (Errors): A taxa de solicitações que falham (códigos HTTP 500x). É o principal indicador de que algo quebrou.
  • 4. Saturação (Saturation): Quão "cheio" está o seu sistema. Ex: Uso de CPU, Memória ou a Fila de um banco de dados que está perto do limite.
