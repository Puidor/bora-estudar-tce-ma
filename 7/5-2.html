Pipelines CI/CD: Jenkins e GitHub Actions

🛤️ O que é um Pipeline?

Um Pipeline é uma sequência de etapas automatizadas que o seu código percorre desde o momento em que o programador faz o "commit" (envio) até chegar ao servidor de produção. Ele materializa o CI/CD (Integração e Entrega Contínuas).

🥊 O Duelo: Jenkins vs GitHub Actions

CAI MUITO EM PROVA!
Característica Jenkins GitHub Actions
Hospedagem Self-hosted (IaaS/On-prem). Você precisa instalar, gerenciar e atualizar o servidor. SaaS (Gerenciado). Roda direto na nuvem do GitHub, sem servidor para administrar.
Linguagem do Pipeline Baseado em Groovy (Jenkinsfile). Baseado em YAML (.github/workflows/*.yml).
Extensibilidade Baseado em milhares de Plugins de terceiros. Baseado em Actions nativas e da comunidade (Marketplace).
Arquitetura de Execução Master / Agent (Slave). O Master coordena, os Agents rodam o job. Baseado em Runners (VMs do GitHub ou Self-hosted).

☕ Vocabulário Jenkins

Jenkinsfile

É o arquivo de texto que contém a definição do Pipeline. Ele permite a prática de "Pipeline as Code" (Pipeline como Código). Fica salvo no próprio repositório Git junto com o código-fonte da aplicação.

Declarative vs Scripted

O Jenkinsfile pode ser escrito de duas formas: Declarativo (mais moderno, estruturado e fácil de ler) ou Scripted (forma antiga, usa código Groovy puro, mais complexo mas muito flexível).

🚨 Pegadinhas Mortais da Banca (CEBRASPE)

  • O Jenkins não precisa de servidor?
    A banca diz: "O Jenkins é uma solução Serverless que isenta o administrador de gerenciar o servidor de integração contínua."
    Falso! O Jenkins requer um servidor físico ou máquina virtual (IaaS). Você tem que instalar o Java, baixar o Jenkins, atualizar o SO e gerenciar a segurança dele. O GitHub Actions é que atua no formato SaaS/Serverless.
  • Master executa as tarefas (builds)?
    "Na arquitetura do Jenkins, o nó Master é responsável por compilar e testar os códigos de todas as linguagens da empresa simultaneamente."
    Falso! Por boas práticas, o Master serve APENAS para gerenciar o painel web, escalonar os jobs e monitorizar os Agents (Slaves). Quem efetivamente roda o código, instala pacotes e faz o build são os Agents (máquinas auxiliares conectadas ao Master).
  • GitHub Actions só funciona com Git?
    "Os Workflows do GitHub Actions só podem ser disparados quando um desenvolvedor faz um *push* (envio) de código no repositório."
    Falso! Os Workflows (pipelines) do GitHub Actions são Event-driven (orientados a eventos). Eles podem ser acionados por um *push*, mas também por eventos de tempo (Agendamento CRON / schedule), cliques de botões manuais (workflow_dispatch) ou até webhooks de sistemas externos.

1. Fundamentos: Por que precisamos de Pipelines?

Antes do CI/CD, os programadores construíam o código nas suas próprias máquinas e, na sexta-feira, tentavam juntar os códigos de toda a equipe para enviar para o servidor. Se algo falhasse (e sempre falhava), ninguém sabia qual linha exata causou o problema.

Os Pipelines de CI/CD automatizam esse processo. É uma "linha de montagem". Se você escreve código, você envia para o servidor Git. Imediatamente, uma ferramenta (como Jenkins) é notificada. Ela descarrega o seu código e executa uma série de Estágios (Stages): Build (Compilar) -> Test (Testar) -> Deploy (Lançar). Se falhar no teste, a esteira para e o programador recebe um e-mail de alerta.

2. Jenkins (O Gigante Open-Source)

Escrito em Java, o Jenkins é a ferramenta de automação open-source mais famosa do mundo. Sua maior força é o seu ecossistema: ele possui milhares de plugins, podendo integrar-se com quase qualquer tecnologia de software existente.

A Arquitetura Master / Agent

O Jenkins usa uma arquitetura distribuída para escalar.

  • Controller (antigo Master): É o coração do Jenkins. Serve a interface web (UI), distribui as tarefas (Jobs), gere os ficheiros de configuração e recebe as chamadas Webhooks do repositório de código.
  • Agents (antigos Slaves): São máquinas, contêineres ou VMs configuradas especificamente para executar os builds. O Controller manda uma ordem: "Agent do Linux, pegue esse código Python e rode o teste unitário".

Jenkinsfile: Pipeline as Code

No passado, as pessoas clicavam em botões na interface web do Jenkins para configurar uma automação. Hoje, usa-se o Jenkinsfile. Ao colocar a infraestrutura do pipeline em um arquivo de código-fonte, todo o processo se torna versionável.

# Exemplo de Jenkinsfile Declarativo
pipeline {
  agent any
  stages {
    stage('Build') {
      steps { echo 'Compilando...' }
    }
    stage('Test') {
      steps { echo 'Testando...' }
    }
  }
}

3. GitHub Actions (A Nuvem Nativa)

O GitHub Actions é uma plataforma moderna de CI/CD embutida nativamente dentro do GitHub. Diferente do Jenkins, o GitHub Actions é servido no modelo SaaS (Software as a Service).

A Estrutura do GitHub Actions

  • Workflows: O pipeline inteiro. Configurado em arquivos .yml no diretório .github/workflows do seu repositório.
  • Events (Eventos): O que "acorda" o workflow. Pode ser um push na branch principal, a abertura de um pull_request ou um evento agendado.
  • Jobs: Conjunto de etapas (steps) que executam na mesma máquina (Runner). Jobs rodam em paralelo por padrão, a não ser que você configure dependências entre eles.
  • Runners: É o "Agent" do GitHub. O GitHub fornece VMs gerenciadas (Ubuntu, Windows, macOS) para rodar seus jobs gratuitamente (até certo limite). Você também pode hospedar seu próprio Runner (Self-hosted) no seu datacenter.
Diferença Central de Manutenção: O Jenkins exige uma equipa de Operações (Ops) para gerir patches do Java, falhas no servidor do Jenkins e backups. O GitHub Actions elimina 90% do esforço de Ops, já que o GitHub fornece o ambiente pronto e autoescalável, permitindo que a equipa foque estritamente em escrever o ficheiro YAML da automação.
