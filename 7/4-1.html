Redes, Sub-redes, VPNs e Segurança

🌐 VPC e Sub-redes

VPC (Virtual Private Cloud)

Uma rede virtual isolada logicamente dentro do provedor (como um datacenter virtual). Você define o seu próprio intervalo de endereços IP (CIDR).

Sub-rede (Subnet)

Fatia a VPC em pedaços menores. Pode ser Pública (tem rota direta para a internet) ou Privada (não tem ligação direta com o exterior).

🛡️ Security Groups vs NACLs

CAI SEMPRE!
Característica Security Group (SG) NACL (Network ACL)
Onde atua? Nível da Instância (VM) Nível da Sub-rede (Subnet)
Estado (MUITO COBRADO) STATEFUL (Com estado) STATELESS (Sem estado)
Comportamento de Retorno Se a requisição entrou, o tráfego de retorno é permitido automaticamente. O tráfego de retorno precisa de uma regra explícita permitindo a saída.
Tipos de Regras Só suporta regras de Permissão (Allow). Tudo que não for permitido é negado. Suporta regras de Permissão e Negação (Deny). Avaliadas por número de ordem.

🚪 Gateways e VPNs

IGW (Internet Gateway)

A porta para o mundo exterior. Permite comunicação bidirecional (entrada e saída) entre instâncias na VPC e a internet.

NAT Gateway

Permite que máquinas em sub-redes privadas acedam à internet (para baixar atualizações), mas impede que a internet inicie conexões com elas (Unidirecional).

Site-to-Site VPN

Conexão encriptada por IPsec. Liga o Datacenter físico do seu escritório diretamente à VPC da nuvem, como se estivessem na mesma rede local.

🚨 Pegadinhas Mortais da Banca (CEBRASPE)

  • Stateful x Stateless:
    A banca inverte de propósito: "Security Groups atuam como firewalls stateless no nível da instância."
    Falso! Guarde isto para a vida: Security Group = Stateful (Lembram-se do estado). NACL = Stateless (Amnésia).
  • Onde fica o NAT Gateway?
    A banca diz: "Para dar acesso à internet aos servidores de banco de dados, o administrador deve colocar um NAT Gateway na sub-rede privada onde os bancos de dados estão."
    Falso! O NAT Gateway serve as sub-redes privadas, mas ele DEVE ser instanciado dentro de uma Sub-rede PÚBLICA (pois ele precisa de uma rota para o Internet Gateway para poder "sair").
  • Regras de Deny (Negação) no Security Group:
    "O analista configurou o Security Group da aplicação web para negar explicitamente qualquer tráfego vindo do IP de um atacante conhecido."
    Falso! Security Groups não possuem regras de DENY. O que não tem "Allow" é simplesmente barrado. Se quiser bloquear um IP específico explicitamente (regra Deny), você é obrigado a usar uma NACL (Network ACL).

1. A Fundação: VPC e Sub-redes

Qualquer arquitetura séria em nuvem começa na rede. O bloco fundamental é a VPC (Virtual Private Cloud - terminologia da AWS) ou VNet (Virtual Network - terminologia Azure).

A VPC é a sua rede lógica isolada dentro da vasta nuvem pública do provedor. É como se você construísse um muro invisível em torno de uma fatia do datacenter da Amazon ou da Microsoft.

Sub-redes (Subnets)

Ao criar a VPC, você define um bloco de IPs grande (ex: 10.0.0.0/16). Para organizar as máquinas, fatiamos esse bloco grande em sub-redes menores. Existem dois tipos críticos para as arquiteturas em nuvem:

  • Sub-rede Pública: O tráfego desta rede tem uma rota que aponta diretamente para a internet (via Internet Gateway). Aqui colocamos Servidores Web, Bastion Hosts e Load Balancers públicos.
  • Sub-rede Privada: O tráfego interno não tem uma rota direta para o exterior. Ninguém da internet consegue aceder a um IP aqui dentro. Usamos obrigatoriamente para as Bases de Dados e para a camada de Lógica de Negócio (App Servers).

2. A Ponte: Internet Gateway vs NAT Gateway

Como as máquinas (VMs) dentro da sua VPC comunicam com o mundo exterior? Precisamos de "Gateways" (Portões).

Internet Gateway (IGW)

Ele atua nas sub-redes públicas. Providencia comunicação bidirecional (entra e sai). Fornece um "alvo" nas suas tabelas de rotas para tráfego rastreável pela internet e realiza a tradução de endereços de rede (NAT) de um IP privado da sua VM para o IP público associado a ela.

NAT Gateway (Network Address Translation)

Um servidor na sub-rede privada precisa baixar uma atualização do Linux na internet (ex: apt-get update). Mas ele não tem IP público! É aqui que entra o NAT Gateway. A VM envia o pedido para o NAT Gateway. O NAT mascara o IP privado usando o seu próprio IP público e envia para a web. Quando a atualização chega, o NAT devolve para a máquina privada.

Crucial: O tráfego no NAT Gateway é unidirecional. A máquina privada acede à internet, mas a internet jamais pode iniciar uma conexão com a máquina privada através dele.

3. A Barreira: Security Groups e NACLs

A segurança de rede em nuvem funciona em camadas (Defesa em Profundidade). As duas principais são as ACLs de Rede (NACL) e os Grupos de Segurança (Security Groups).

Security Group (A Linha de Frente da VM)

Funciona como um firewall virtual no nível da interface de rede da Instância (Máquina Virtual).

  • Stateful (Com Estado): Se você abrir a porta 80 (HTTP) para receber um pedido do cliente, o fluxo de resposta (outbound) daquele pacote está automaticamente autorizado, independentemente das regras de saída. Ele "lembra" do pacote que entrou.
  • Apenas Regras "Allow": Por omissão (default), ele nega tudo que entra e permite tudo que sai. Para deixar alguém aceder, tem de criar explicitamente uma regra de Permissão (Allow). Não existe regra do tipo "Bloquear o IP X".

NACL (A Barreira do Bairro)

Network Access Control List. Atua no nível da Sub-rede (como um porteiro de condomínio fechado, e não da sua casa especificamente).

  • Stateless (Sem Estado): Não tem memória. Se você permitir que uma requisição HTTP entre pela porta 80 (Inbound), você TEM OBRIGATORIAMENTE que criar uma regra de saída (Outbound) permitindo o retorno pelas portas efêmeras (1024-65535). Se não tiver a regra de saída explícita, a resposta morre.
  • Possui Regras "Deny": É a ferramenta ideal se sua empresa sofreu um ataque DDoS de uma sub-rede russa específica. Você cria uma NACL com o IP do atacante configurado como `DENY` com número de precedência 1 (Regras de número mais baixo são lidas primeiro).

🔐 4. Conectando a Nuvem ao Escritório (VPNs)

Mesmo tendo toda a infraestrutura virtualizada numa VPC, empresas muitas vezes precisam de comunicação criptografada com os seus datacenters físicos locais (on-premises) ou com colaboradores trabalhando em home-office. Para isso usamos Virtual Private Networks (VPNs).

  • Site-to-Site VPN: Uma conexão IPsec estável que une dois roteadores fixos. De um lado o Virtual Private Gateway (VGW) da AWS/Azure, do outro lado o roteador (Customer Gateway) físico do escritório. Tudo que trafega pelo túnel é invisível à internet.
  • Client VPN (Point-to-Site): Para acesso de utilizadores individuais. O programador instala uma aplicação cliente OpenVPN no seu computador de casa. Quando conectada, a sua máquina recebe um IP local da VPC para conseguir acessar os servidores de base de dados seguros.
  • Dica: Direct Connect / ExpressRoute: Quando o túnel IPsec da internet não é suficientemente rápido ou fiável para um banco, o provedor oferece linhas de fibra óptica exclusivas e diretas dedicadas que ligam o datacenter da empresa à nuvem. (AWS Direct Connect / Azure ExpressRoute).
