🔄 IPsec: Modo Transporte vs Túnel

Modo Transporte

Criptografa apenas os DADOS (payload). O cabeçalho IP original continua visível. Usado para comunicação host a host (PC para PC).

Modo Túnel

Criptografa o pacote inteiro (Dados + Cabeçalho IP original) e adiciona um NOVO cabeçalho IP por fora. Usado em VPNs Site-to-Site (Roteador para Roteador).

🚨 Pegadinhas Mortais de Prova

  • SSL vs TLS: A banca diz que "O SSL é o padrão moderno utilizado para proteger transações bancárias".
    Falso! O SSL (Secure Sockets Layer) está morto, depreciado e vulnerável. O padrão mundial atual que protege a internet é o TLS (Transport Layer Security), embora muitas pessoas ainda digam "Certificado SSL" por força do hábito.
  • PGP vs Autoridade Certificadora: "Para utilizar o PGP (Pretty Good Privacy) no e-mail, é obrigatório comprar um certificado numa AC (como a Serasa ou a Verisign)".
    Falso! O PGP utiliza um modelo descentralizado chamado Web of Trust (Teia de Confiança), onde os próprios usuários assinam as chaves uns dos outros, dispensando a necessidade de uma autoridade central (AC).

1. SSL e TLS (A Segurança da Web)

O TLS atua na Camada de Transporte. Ele é acionado por protocolos de aplicação (como o HTTP virando HTTPS) para garantir um "túnel" seguro entre o navegador do usuário e o servidor web.

O TLS usa uma abordagem Híbrida. Ele une o melhor da criptografia assimétrica (para autenticação) com a simétrica (para velocidade). O processo inicial de negociação é chamado de TLS Handshake e cai frequentemente em provas avançadas.

O Handshake TLS Simplificado

  1. Client Hello: O navegador diz: "Quero falar seguro. Suporto AES, ChaCha20 e RSA".
  2. Server Hello + Certificate: O servidor escolhe o AES, e envia o seu Certificado Digital (que contém a Chave Pública do servidor).
  3. Autenticação: O navegador verifica se o certificado foi assinado por uma Autoridade de Confiança (CA). Se sim, garante que não está falando com um site falso (Garante Autenticidade).
  4. Key Exchange: O navegador gera uma chave de sessão (Chave Secreta), criptografa com a Chave Pública do servidor e envia. (Em versões modernas do TLS 1.3, o Diffie-Hellman faz essa troca sem usar o RSA).
  5. Comunicação Cifrada: Daqui para a frente, tudo viaja ultra-rápido criptografado com a Chave Simétrica acordada (Garante Confidencialidade e Integridade).

2. IPsec (Internet Protocol Security)

O IPsec atua na Camada de Rede (L3). Diferente do TLS que precisa que a aplicação (navegador) inicie a segurança, o IPsec é transparente para os usuários e aplicações. Ele cifra o tráfego a nível do Sistema Operacional ou do Roteador. É o protocolo "rei" na criação de VPNs Site-to-Site.

Ele é um suite composto por 3 protocolos menores:

  • 1. IKE (Internet Key Exchange)
    Usa a porta UDP 500. É a "recepção". Ele negocia a associação de segurança (SA) e as chaves usando Diffie-Hellman antes dos dados começarem a trafegar.
  • 2. AH (Authentication Header)
    Garante Autenticidade, Integridade e Proteção contra ataques de repetição (Anti-replay). Mas ATENÇÃO: o AH NÃO criptografa os dados (Zero Confidencialidade). Se usar só o AH, um hacker lendo a rede verá os dados passando em texto puro, mas não poderá alterá-los.
  • 3. ESP (Encapsulating Security Payload)
    O pacote completo. Garante tudo do AH e cifra os dados (garante Confidencialidade). Nas VPNs modernas, o ESP é usado em 99% das vezes, deixando o AH esquecido na teoria.

3. Proteção de E-mails (S/MIME vs PGP)

A camada de Aplicação também tem protocolos específicos de proteção. Os e-mails são enviados em "texto claro" nativamente. Para garantir que um anexo chegue a salvo, usamos a Criptografia de Chave Pública (Assimétrica).

S/MIME

Padrão em ambientes corporativos pesados. Como confia que a Chave Pública pertence mesmo ao emissor? Baseia-se em PKI (Public Key Infrastructure). Requer que uma Autoridade Certificadora (CA) central valide os certificados.

PGP (Pretty Good Privacy)

Padrão "raiz", muito usado por ativistas e programadores open-source. Não usa AC central. Usa a Web of Trust (Teia de Confiança): eu confio na tua chave porque um amigo meu que eu confio assinou a tua chave confirmando que te conhece.

4. SSH (Secure Shell)

O SSH (que roda na porta TCP 22) atua na Camada de Aplicação. Veio para substituir os obsoletos e inseguros Telnet e rlogin, que transmitiam senhas de administrador em texto plano.

O SSH usa criptografia forte para criar um canal seguro por onde um administrador pode fazer login remoto num servidor Linux (linha de comandos). Mas nas provas, o recurso mais cobrado do SSH é o seu poder de criar túneis:

Port Forwarding (Tunelamento SSH):

O SSH pode encapsular (esconder dentro de si) o tráfego de outros protocolos que seriam inseguros. Por exemplo, você pode "enrolar" uma conexão de FTP, VNC ou até HTTP dentro da conexão SSH, garantindo confidencialidade a esses protocolos antigos usando a criptografia do SSH.
