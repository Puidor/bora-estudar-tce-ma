DMZ (Demilitarized Zone)

⚡ Direto ao Ponto

  • O que é? A DMZ (Zona Desmilitarizada) é uma sub-rede isolada que fica entre a internet pública (insegura) e a rede interna da organização (segura).
  • Objetivo: Expor os serviços que precisam ser acessados externamente (ex: Site da empresa, Servidor de E-mail) de forma controlada, garantindo que, se um hacker os invadir, ele continuará isolado e não terá acesso à rede interna (LAN) onde estão os dados críticos.

🏢 Onde coloco os meus servidores?

Ficam NA DMZ (Expostos)

Serviços que precisam "falar" com o público externo na Internet.

  • 🌐 Servidores Web (HTTP/HTTPS)
  • 📧 Servidores de E-mail (SMTP/Mail Relay)
  • 🧭 Servidores DNS Externos
  • 📁 Servidores FTP (Públicos)
  • 🎭 Proxy Reverso

Ficam NA REDE INTERNA (LAN)

Serviços críticos que contém dados sigilosos ou operacionais.

  • 🗄️ Banco de Dados (SGBD)
  • 🧑‍💼 Active Directory (Controlador de Domínio)
  • 💻 Computadores dos Funcionários
  • 📁 Servidores de Arquivos Internos (File Server)

🚦 Regras de Tráfego (A Alma da Prova)

Decore quem pode iniciar conexão com quem.

🌐 ➔ 🏢

Internet para DMZ

PERMITIDO
Apenas nas portas específicas do serviço (Ex: Porta 80 para Web, 25 para SMTP). O resto é bloqueado.

🏢 ➔ 🔒

DMZ para Rede Interna

BLOQUEADO
A DMZ NÃO PODE iniciar livremente conexões para a LAN (exceto furos muito específicos e estritos, como o servidor web acessando o BD interno).

🔒 ➔ 🏢

Rede Interna para DMZ

PERMITIDO
A rede interna (segura) tem permissão total para acessar a DMZ e a Internet (saindo).

🚨 Pegadinhas Mortais de Prova

  • A Ilusão de Segurança Máxima: A banca diz que "Os servidores instalados na DMZ são os mais seguros da empresa, pois estão protegidos pelo firewall de borda".
    Falso! Os servidores da DMZ são os MAIS EXPOSTOS e de maior risco da empresa, afinal, todo o tráfego da internet é canalizado diretamente para eles. Eles estão na "linha de tiro". O local de segurança máxima é a LAN Interna.
  • Banco de Dados na DMZ? "Para acelerar o carregamento do site institucional, o Banco de Dados deve ser alocado junto ao servidor web dentro da DMZ".
    Falso! Regra sagrada de arquitetura: Nunca coloque o Banco de Dados principal na DMZ. Se o servidor Web for invadido (o que é comum), o atacante levaria o banco de dados inteiro de brinde. O BD fica na LAN, protegido pelo firewall interno.

1. A Origem e o Conceito da DMZ

O termo Demilitarized Zone vem do meio militar (a faixa de terra neutra entre as Coreias do Norte e do Sul). Na computação, a ideia é a mesma: uma "terra de ninguém" que serve como um amortecedor entre a internet pública e a rede corporativa privada.

A dor que a DMZ resolve: Antigamente, se você quisesse colocar o seu site no ar, você abria a Porta 80 do seu Firewall e direcionava para o servidor web que estava dentro da sua rede local (ao lado do computador do financeiro). Se um hacker explorasse uma falha no site e tomasse controle do servidor web, ele estaria automaticamente dentro da sua rede interna, com acesso livre a todos os outros computadores. A DMZ isola esse risco.

2. Topologias Físicas: Como Implementar?

As bancas de concurso (principalmente CEBRASPE) adoram cobrar os dois modelos clássicos de implementação de uma DMZ.

A. Firewall de Três Pernas (Three-Homed Firewall)

Usa apenas UM equipamento de Firewall que possui, no mínimo, três interfaces de rede (três cabos conectados a ele).

  • Interface 1: Conectada à Internet (WAN).
  • Interface 2: Conectada à Rede Interna (LAN).
  • Interface 3: Conectada ao Switch da DMZ.

Vantagem: Mais barato e simples de gerenciar (só uma caixa).
Desvantagem (A Pegadinha): Cria um Single Point of Failure (Ponto Único de Falha). Se o firewall pifar, ou se o hacker conseguir invadir as regras de administração desse único firewall, a rede interna e a DMZ caem nas mãos dele simultaneamente.

B. Firewall Duplo (Back-to-Back Firewall)

A abordagem de segurança máxima. Utiliza DOIS firewalls separados.

Internet ➔ [ Firewall 1 ] ➔ DMZ ➔ [ Firewall 2 ] ➔ LAN Interna

A Tática Avançada (Defense in Depth): Em provas difíceis, é recomendada a política de Diversidade de Fornecedores. O Firewall 1 deve ser de uma marca (ex: Cisco) e o Firewall 2 de uma marca completamente diferente (ex: Fortinet ou pfSense).
Por quê? Porque se amanhã descobrirem um bug "Zero-Day" fatal no firewall da Cisco, o hacker quebra o Firewall 1, invade a DMZ, mas esbarra no Firewall 2 (que é de outra marca e não possui aquele bug), salvando a rede interna!

3. Hardening dos Servidores da DMZ

Como a DMZ é a "zona de tiro" da empresa, você não deve simplesmente instalar o Windows e largar o servidor lá. Servidores na DMZ exigem processos estritos de Hardening (Endurecimento).

São eles os chamados Bastion Hosts (Hospedeiros Bastião):

  • Remoção de absolutamente todos os serviços, protocolos e softwares que não são necessários para a função do servidor. (Se é um servidor Web, o FTP deve ser desinstalado).
  • Esses servidores NÃO devem fazer parte do domínio (Active Directory) da rede interna. Eles devem ser máquinas em grupo de trabalho isolado (Workgroup) para que, se invadidos, o hacker não roube as credenciais de AD da empresa.
  • Devem estar rodando os patches mais agressivos de segurança e ferramentas como IDS/IPS locais e FIM (File Integrity Monitoring - para saber se o hacker alterou as páginas html).
