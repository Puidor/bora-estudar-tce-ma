ATDD (Desenvolvimento Guiado por Testes de Aceitação)

⚡ Direto ao Ponto

  • O Conceito Central: O ATDD foca na colaboração e comunicação. Antes de os desenvolvedores começarem a programar, toda a equipe (Negócios, Devs e QA) senta junto para definir os Critérios de Aceitação exatos daquela funcionalidade.
  • Testes como Requisitos: Os critérios de aceitação discutidos pela equipe são transformados em testes automatizados de alto nível. Se o código passar nesses testes, significa que o negócio "aceita" a funcionalidade.
  • O Foco: Prevenir defeitos de comunicação (construir a funcionalidade errada) em vez de apenas caçar bugs no código.

🔄 O Ciclo do ATDD (Os 4 D's)

É a sequência metodológica oficial cobrada em provas mais exigentes.

1

Discuss (Debater)

Cliente, Dev e Tester (os 3 Amigos) reúnem-se para debater o que a funcionalidade deve fazer e tirar todas as dúvidas do negócio.

2

Distill (Destilar)

A equipe extrai a essência do debate e escreve os testes de aceitação num formato que todos (humanos e máquinas) entendam.

3

Develop (Desenvolver)

O desenvolvedor escreve o código (podendo usar o TDD internamente) até que todos os testes de aceitação passem (fiquem verdes).

4

Demo (Demonstrar)

A equipe faz uma demonstração ao cliente do software rodando e passando nos testes, garantindo que o requisito foi cumprido.

⚖️ O Confronto Final: TDD x BDD x ATDD

Como distinguir as três "letrinhas" na hora da prova.

Prática Público-Alvo A Pergunta que Responde Linguagem do Teste
TDD Programadores "Estamos construindo a coisa direito?" (Sem bugs/Falhas) Código da linguagem (Java, C#)
BDD Toda a equipe "Como o sistema deve se comportar nessa situação?" Linguagem natural (Gherkin: Dado/Quando/Então)
ATDD Cliente, QA e Dev "Estamos construindo a coisa certa?" (Valor de negócio) Critérios de aceitação tabulares ou textuais (FitNesse, Robot)
*Nota: Na prática do dia a dia do mercado, BDD e ATDD se sobrepõem muito. Para a prova, lembre-se do ATDD muito atrelado ao conceito de Critérios de Aceitação da História de Usuário.

🚨 Pegadinhas Mortais de Prova

  • A Exclusividade do QA: A banca afirma: "No modelo ATDD, a equipe de QA (Testadores) é a única responsável por escrever e aprovar os testes de aceitação".
    Falso! A essência do ATDD é a criação colaborativa. Os testes são derivados de uma conversa conjunta entre Desenvolvedores, QA e Clientes/Negócios.
  • Substituição do TDD: "A adoção do ATDD substitui completamente a necessidade de aplicar o TDD no nível de código".
    Falso! O ATDD garante que a funcionalidade cumpre a regra de negócio (visão externa). O TDD garante que as classes e funções internas estão bem desenhadas e sem bugs (visão interna). Eles são perfeitamente complementares. O dev usa TDD para cumprir a meta do ATDD.

1. A Origem: O Foco no "Quê" e não no "Como"

No desenvolvimento de software tradicional, muitas vezes o cliente diz: "Eu quero um sistema de carrinho de compras". O desenvolvedor vai para a sua mesa, escreve o código (usando TDD ou não) e 1 mês depois entrega o sistema. Quando o cliente vai usar (Teste de Aceitação), ele descobre que faltou colocar o botão de descontos.

O ATDD (Acceptance Test-Driven Development) tenta matar esse problema pela raiz. Ele exige que as perguntas sejam feitas antes da codificação começar. O cliente e o testador criam exemplos práticos: "E se o carrinho tiver R$ 100 e eu aplicar um cupom expirado?". Esse cenário exato vira um teste de aceitação automatizado.

2. Critérios de Aceitação como Especificações Executáveis

No Ágil (como no Scrum), os requisitos vêm na forma de "Histórias de Usuário". Cada História deve ter um conjunto de Critérios de Aceitação, que são a "check-list" que define se a história está pronta (Done).

A Magia do ATDD: Especificações Executáveis

A grande sacada do ATDD é não deixar os critérios de aceitação num papel esquecido. Eles são codificados em ferramentas especiais que rodam o critério de aceitação contra o código real. Se o teste brilhar em "verde", a funcionalidade foi aprovada pelo computador usando a regra de negócio do cliente.

Isso transforma a documentação (os requisitos) em algo VIVO, que é testado continuamente (Living Documentation).

3. A Integração: ATDD + TDD na mesma Sprint

Como funciona o dia a dia de um programador que aplica ambas as metodologias? Para a prova, visualize um "ciclo dentro do outro":

  • 1. O Ciclo Macro (ATDD): O desenvolvedor recebe o Teste de Aceitação automatizado. Ele roda e, obviamente, o teste Falha (Red), pois não há código.
  • 2. O Ciclo Micro (TDD): Para fazer esse grande Teste de Aceitação passar, o desenvolvedor precisa criar várias classes pequenas no código. Para cada classe, ele aplica o ciclo TDD (Red-Green-Refactor) dezenas de vezes em Testes Unitários ultrarrápidos.
  • 3. O Fechamento Macro: Após horas fazendo TDD, o desenvolvedor finalmente rodará o Teste de Aceitação (Macro) e ele passará a ficar Verde (Green). A funcionalidade de negócio está finalizada e testada de ponta a ponta.

4. Ferramentas Típicas de ATDD (Foco de Prova)

Embora ferramentas de BDD como o Cucumber sejam muito usadas para ATDD devido à linguagem natural Gherkin, existem frameworks que nasceram com um forte viés tabular/técnico (Keyword-driven) clássico do ATDD.

  • FitNesse: Um wiki de colaboração na web e ferramenta de testes de aceitação. Permite que os testadores e clientes insiram casos de teste em tabelas simples na web, e a ferramenta lê essas tabelas e executa-as contra o código (usando uma interface chamada FIT ou SliM).
  • Robot Framework: Extremamente popular e cobrado em concursos. Baseado em Python, utiliza uma abordagem Keyword-Driven (guiada por palavras-chave), permitindo criar testes de aceitação através de tabelas bem fáceis de ler, sendo muito forte também para RPA (Robotic Process Automation).
  • Selenium / Cypress (Automação UI): Muitas vezes o ATDD termina testando a interface visual final que o cliente usa. Ferramentas de automação de navegadores (como Selenium) são usadas em conjunto com frameworks ATDD/BDD para simular os cliques e digitações do usuário final.
