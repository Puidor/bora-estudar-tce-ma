Docker e Orquestração com Kubernetes

⚡ Direto ao Ponto: O Motor vs O Maestro

🐳 Docker (O Motor)

Foca no empacotamento e execução individual. Ele cria a imagem e roda o contêiner numa única máquina.

☸️ Kubernetes (O Maestro)

Foca na orquestração em cluster. Ele espalha, escala, reinicia e monitora milhares de contêineres através de múltiplas máquinas.

🧠 Arquitetura do Kubernetes (K8s)

Sempre dividido em duas grandes áreas físicas/lógicas.

1. Control Plane (Master)

O Cérebro. Toma as decisões globais sobre o cluster.

  • kube-apiserver: A "porta da frente". Todos os comandos passam por ele.
  • etcd: O banco de dados chave-valor que guarda o "estado desejado" do cluster.
  • kube-scheduler: Decide em qual "Nó" um novo Pod deve rodar.
  • kube-controller-manager: Fiscaliza se o estado atual é igual ao estado desejado (ex: mantendo as 3 réplicas exigidas).

2. Worker Nodes (Nós Operários)

As máquinas onde as aplicações realmente rodam.

  • kubelet: O agente/capataz. Garante que os contêineres estão rodando corretamente dentro de um Pod e avisa o Master.
  • kube-proxy: Gerencia as regras de rede (networking) e roteamento de tráfego para os Pods.
  • Container Runtime: O motor de execução de fato (ex: Docker, containerd).

📦 Objetos do Kubernetes (Core Decoreba)

1. Pod

A menor unidade criável no K8s. É uma "capsula" que empacota um ou mais contêineres. Compartilham o mesmo IP interno.

2. ReplicaSet

Garante que um número fixo de cópias (réplicas) de um Pod esteja sempre rodando. Se um cair, ele levanta outro.

3. Deployment

Objeto superior que gerencia ReplicaSets. Permite fazer atualizações seguras (Rolling Updates) de uma versão velha para uma nova.

4. Service

Um IP estático e roteador interno. Serve como "Porta de entrada" fixa, distribuindo o tráfego para os Pods (que nascem e morrem com IPs dinâmicos o tempo todo).

🚨 Pegadinhas Mortais de Prova

  • A Unidade Básica do K8s: A banca vai jurar que "A menor unidade gerenciável dentro do Kubernetes é o Contêiner Docker".
    Falso! Absolutamente Falso! O Kubernetes NÃO gerencia contêineres diretamente. A sua menor unidade de controle é o POD. O K8s cria um Pod, e o contêiner roda lá dentro.
  • Docker vs Virtual Machine: "Contêineres Docker são máquinas virtuais completas, por isso precisam ter um Sistema Operacional próprio instalado (Guest OS)".
    Falso! Diferente das VMs, os contêineres compartilham o mesmo Kernel (núcleo) do sistema operacional hospedeiro. Eles não possuem SO próprio emulado, o que os torna muito mais leves e rápidos que as VMs tradicionais.

1. Fundamentos do Docker

A frase de terror das equipes de desenvolvimento sempre foi: "Mas na minha máquina funciona!". O código funcionava no notebook do desenvolvedor porque ele tinha as versões exatas do Java e de bibliotecas específicas, mas quando ia para o servidor, quebrava.

O Docker resolveu isso criando a Conteinerização. Ele empacota o código junto com todas as bibliotecas, dependências e configurações necessárias num pacote isolado. A promessa é a portabilidade: se rodar no Docker do desenvolvedor, vai rodar exatamente da mesma forma no servidor de produção.

A Mágica do Isolamento no Linux

Como os contêineres não instalam um SO completo, como eles ficam isolados uns dos outros?

  • cgroups (Control Groups): Limita os recursos físicos. O Docker diz ao Kernel: "Este contêiner só pode usar 500MB de RAM e 10% da CPU".
  • Namespaces: Cria o isolamento lógico. Engana o contêiner fazendo-o acreditar que ele tem o seu próprio disco rígido, sua própria rede e seus próprios processos, impedindo-o de enxergar arquivos de outros contêineres na mesma máquina.

Vocabulário Docker (Core Decoreba)

  • Dockerfile: É o arquivo de texto (receita de bolo) que contém os comandos para criar uma imagem.
  • Image (Imagem): É o pacote final gerado. Possui uma arquitetura de "Camadas" (Layers) e é somente-leitura (imutável).
  • Container: É a imagem quando colocada em execução na memória. Ele recebe uma camada extra de gravação temporária.
  • Registry: Repositório central onde armazenamos e baixamos imagens (ex: Docker Hub).

2. A Necessidade da Orquestração

Gerenciar 5 contêineres num servidor usando o comando `docker run` é fácil. Mas num cenário de Big Data ou Microsserviços de um e-commerce, você precisa gerenciar 5.000 contêineres espalhados em 50 servidores diferentes.

Se o Servidor 10 queimar, quem reinicia automaticamente os 100 contêineres que estavam nele, movendo-os para o Servidor 11? Quem cria o balanceamento de carga entre eles? A resposta é o Orquestrador. O mercado tinha o Docker Swarm e o Apache Mesos, mas o Kubernetes (K8s) (criado pelo Google e doado à CNCF) venceu a guerra e tornou-se o padrão absoluto.

3. Detalhando a Arquitetura do Kubernetes

O Kubernetes é baseado numa arquitetura declarativa de Estado Desejado. Você não dá comandos passo a passo. Você diz: "Quero 3 instâncias do site rodando sempre". O K8s se encarrega de monitorar a realidade e consertá-la caso algum nó caia.

Control Plane (Master Node)

O coração do sistema. As bancas cobram os seus componentes essenciais:

  • Kube-APIServer: O único componente com o qual você (ou outros componentes) fala diretamente (via CLI kubectl ou REST API). Valida e encaminha solicitações.
  • etcd: Um banco de dados chave-valor altamente disponível e confiável. É a única fonte da verdade (Source of Truth) de todo o cluster. Todo o estado do K8s fica guardado aqui.
  • Kube-Scheduler: Observa a criação de novos Pods (que ainda não têm um nó assinalado) e decide em qual máquina operária (Node) ele deverá rodar, analisando CPU e RAM disponíveis.
  • Kube-Controller-Manager: Contém os loops de controle que vigiam o estado do cluster, comparando o "estado atual" com o "estado desejado" e tomando ações corretivas.

Data Plane (Worker Nodes)

  • Kubelet: O agente instalado em cada máquina. Ele se comunica com o Master, recebe as instruções ("rode estes contêineres") e usa o Docker/containerd para iniciá-los, monitorando a saúde deles.
  • Kube-Proxy: Regras de rede. É ele quem permite que as regras de balanceamento de carga e IPs dos "Services" funcionem fisicamente no sistema operacional Linux.

4. Objetos e Escalabilidade

A interação com o K8s ocorre através da declaração de arquivos YAML que descrevem "Objetos".

A Tríade Produtiva

  • A. Deployment (A Regra) Descreve o aplicativo. Ex: "Rode a Imagem v1.0 do meu site com 5 réplicas". O Deployment cria automaticamente um ReplicaSet para garantir as 5 cópias vivas (Pods). Se você alterar o YAML para "v2.0", o Deployment gerencia uma atualização progressiva (Rolling Update), matando os pods velhos um a um sem tirar o site do ar.
  • B. Pods (A Execução) Os 5 Pods nascem e começam a processar os dados. Se um der erro fatal de memória, o K8s mata-o e levanta outro num segundo, com um endereço IP Interno completamente novo.
  • C. Service (A Porta de Entrada) Como o usuário acessa o sistema se o IP dos Pods muda a toda hora? O Service cria um Endereço Fixo e um Balanceador de Carga nativo. O usuário bate no "Service", e o Service distribui o tráfego para os Pods atuais, ignorando os antigos. É o escudo da abstração de rede.
