Fase de Análise (Requisitos)

Foco no O QUÊ o sistema deve fazer. (Entender o problema do cliente, regras de negócio).

Fase de Projeto (Design)

Foco em COMO o sistema será construído tecnicamente. (Arquitetura, Banco de Dados, Interfaces, Padrões).

🏆 O Santo Graal do Design de Software

Se você decorar apenas uma coisa sobre este tópico, que seja isto:

ALTA COESÃO
e
BAIXO ACOPLAMENTO

Coesão (Força Interna)

Mede o quão focada uma classe/módulo é. Um módulo com Alta Coesão faz apenas uma coisa, e faz muito bem (Princípio da Responsabilidade Única).

Queremos ALTA!

Acoplamento (Dependência Externa)

Mede o nível de dependência entre dois módulos. Um Baixo Acoplamento significa que você pode alterar o Módulo A sem quebrar o Módulo B.

Queremos BAIXO!

🧩 Conceitos Estruturais

Abstração

Focar nas características essenciais de um objeto, ignorando os detalhes complexos ou irrelevantes. (Ex: Para o motorista, um carro é um volante e pedais; ele abstrai a injeção eletrônica).

Ocultação de Informação

Módulos devem esconder seus detalhes internos (algoritmos, dados) de outros módulos. Eles comunicam-se apenas através de Interfaces bem definidas.

Refatoração (Refactoring)

Reorganizar a estrutura interna do código para torná-lo mais limpo e manutenível, SEM alterar o seu comportamento externo (o software continua a fazer a mesma coisa).

🚨 Pegadinhas Mortais de Concurso

  • A Inversão Diabólica: A banca vai jurar que "O objetivo de um bom projeto de software é garantir o Baixo Acoplamento e a Baixa Coesão".
    Falso! A coesão (foco interno) deve ser sempre ALTA. O acoplamento (dependência externa) deve ser BAIXO.
  • Refatoração vs Reengenharia: "A refatoração altera a funcionalidade do software para adicionar novos requisitos."
    Falso! Refatoração é como arrumar o guarda-roupas: as roupas (funcionalidades) continuam lá e servem do mesmo jeito, só estão mais fáceis de achar. Se mudar o comportamento, não é refatoração.

1. A Importância do Projeto (Design)

Na Engenharia de Software clássica (como definida por Pressman ou Sommerville), a etapa de Projeto (Design) é o momento em que os requisitos do negócio são traduzidos em uma representação técnica do software.

Se a fase de Engenharia de Requisitos descobre "o problema", a fase de Projeto é a criação da solução. Sem um bom projeto, o sistema pode até funcionar no início, mas será um "código espaguete", impossível de atualizar, testar ou dar manutenção no futuro.

Os 4 Níveis do Projeto de Software:
  • Projeto de Dados/Classes: Como os dados e as estruturas de classes serão moldados (ex: Modelagem de Banco de Dados, Diagrama de Classes).
  • Projeto Arquitetural: A visão macro do sistema. Como os grandes módulos se relacionam (ex: Arquitetura MVC, Microsserviços).
  • Projeto de Interface: Como o software interage com os utilizadores (UI/UX) ou com outros sistemas (APIs).
  • Projeto de Componentes: A visão micro. A lógica interna (algoritmos) que cada módulo utilizará para funcionar.

2. Coesão e Acoplamento (A Fundo)

Estes são os dois indicadores absolutos de qualidade de um projeto arquitetural. As bancas gostam de cobrar os *tipos* (níveis) de coesão e acoplamento.

A Escala de Coesão

A coesão avalia se as funções dentro de um módulo têm a ver umas com as outras. Queremos a mais alta possível.

  • Funcional (A Melhor): O módulo realiza uma, e APENAS uma tarefa muito bem definida (ex: CalcularImpostoDeRenda()).
  • Sequencial: A saída de uma função do módulo serve como entrada para a próxima função dentro do mesmo módulo.
  • Comunicacional: Todas as operações do módulo atuam sobre a mesma estrutura de dados (mesmo input).
  • Coincidente (A Pior): As funções estão no mesmo módulo por puro acaso. Não têm nenhuma relação umas com as outras (uma classe "Utils" mal feita que envia e-mail, calcula raiz quadrada e desliga o PC).

A Escala de Acoplamento

O acoplamento avalia o quão "colado" um módulo está no outro. Queremos o mais baixo possível.

  • De Dados (O Melhor / Mais Fraco): Os módulos apenas trocam dados simples através de parâmetros bem definidos (ex: envio de um CPF como String para a função).
  • De Controle: Um módulo passa uma "flag" (ex: um booleano) que controla o que o outro módulo vai executar internamente.
  • De Conteúdo (O Pior / Mais Forte): Um módulo altera secretamente ou acessa diretamente os dados internos privados de outro módulo. Se o módulo A mudar a sua estrutura, o módulo B explode.

3. Modularidade e Ocultação da Informação

Modularidade

"Dividir para conquistar". É a quebra do software em componentes (módulos) menores e separadamente nomeáveis, que são integrados para satisfazer os requisitos. Atenção à teoria: A modularidade tem um ponto de equilíbrio ótimo. Se não houver módulos (tudo num ficheiro gigante), é o caos. Mas se você quebrar um software em 10 mil módulos minúsculos de 1 linha de código, o custo de integrá-los e chamá-los vai anular os benefícios, degradando a performance.

Information Hiding (Ocultação de Informação)

Princípio proposto por David Parnas. Estabelece que os módulos devem ser especificados de forma que a informação essencial (como um algoritmo matemático específico ou a estrutura de uma tabela no banco de dados) contida num módulo seja inacessível aos outros módulos.

Benefício Prático: Reduz o Acoplamento. Se a "Classe Pagamento" esconde como calcula a taxa, amanhã a taxa pode mudar de 5% para 7% alterando o código *apenas* na Classe Pagamento, sem quebrar as outras 50 classes do sistema que a invocam.

4. Padrões de Projeto (Design Patterns - Menção Honrosa)

Dentro de técnicas de projeto de software, a banca costuma citar os Design Patterns da Gang of Four (GoF). São soluções já testadas e catalogadas para problemas recorrentes de design de software orientado a objetos.

Embora não exista um tópico exclusivo de Padrões GoF no seu edital, conhecer as três grandes categorias pode render pontos numa questão mais difícil:

  • Criacionais: Lidam com a melhor forma de criar objetos, ocultando a lógica da instanciação. (Ex: Singleton - garante que a classe só tem uma instância na memória; Factory Method).
  • Estruturais: Lidam com a composição de classes e objetos, usando herança e interfaces. (Ex: Adapter - faz uma interface velha funcionar num sistema novo; Facade).
  • Comportamentais: Lidam com a comunicação e a atribuição de responsabilidades entre os objetos. (Ex: Observer - avisa a todos os assinantes quando há uma mudança; Strategy).
