Histórias do Usuário (User Stories)

📝 O Template Clássico

Uma História de Usuário é uma descrição curta e simples de uma funcionalidade, contada a partir da perspectiva de quem deseja a nova capacidade (o usuário).

Como um(a) [ Ator / Perfil de Usuário ]

Eu quero [ Ação / Funcionalidade ]

Para que [ Valor de Negócio / Benefício ]

🃏 Os 3 C's (Ron Jeffries)

Toda boa história de usuário é composta por 3 elementos fundamentais:

  • Card
    Cartão

    A história deve ser pequena o suficiente para caber num post-it. É um lembrete do requisito, não um documento detalhado.

  • Conv.
    Conversa

    Os detalhes técnicos e regras de negócio surgem da discussão entre desenvolvedores, PO e clientes. O cartão inicia a conversa.

  • Conf.
    Confirmação

    São os Critérios de Aceitação. Como sabemos que a história está pronta? Os testes automatizados ou manuais provam isso.

✨ O Acrônimo INVEST

Criado por Bill Wake, define as 6 características de uma boa história de usuário:

  • Independent (Independente)
  • Negotiable (Negociável)
  • Valuable (Valiosa para o Negócio)
  • Estimable (Estimável)
  • Small (Pequena)
  • Testable (Testável)

🚨 Pegadinhas Mortais de Prova

  • História de Usuário = Caso de Uso? A banca afirma que "Histórias de Usuário são a versão moderna e idêntica aos Casos de Uso (UML)".
    Falso! Casos de Uso descrevem o comportamento do sistema de forma exaustiva, com fluxos principais, alternativos e de exceção. Histórias de Usuário são propositalmente incompletas; são apenas "promessas para uma conversa". Elas não detalham o fluxo passo a passo no documento.
  • Pode ser Escrita pelo Programador? "Uma User Story deve ser escrita com detalhes técnicos, como nomes de tabelas, para facilitar a vida do desenvolvedor".
    Falso! Histórias de usuário focam no "O Quê" e no "Por que" (valor de negócio). Elas NÃO descrevem o "Como" (tecnologia). Devem ser compreensíveis pelo cliente final.

1. A Origem e Filosofia das User Stories

A prática de usar Histórias de Usuário originou-se com a metodologia Extreme Programming (XP) em 1998, embora hoje seja amplamente associada ao Scrum (que utiliza as histórias para preencher o Product Backlog).

No modelo tradicional em cascata, os requisitos eram congelados em documentos de 300 páginas ("Especificação de Requisitos do Sistema"). Ninguém os lia e, quando a equipe começava a programar, o documento já estava obsoleto.

A Mudança de Paradigma:

As histórias de usuário mudam o foco de escrever sobre os requisitos para conversar sobre eles. O cartão escrito ("Como um cliente...") é o estopim para a colaboração. A equipe confia no entendimento compartilhado gerado pela conversa diária, não num papel.

2. Desconstruindo o INVEST (Foco nas Bancas)

O acrônimo INVEST (cunhado por Bill Wake) é a régua pela qual o Product Owner (PO) e a equipe medem se uma história está pronta para ser puxada para uma Sprint (se ela atende à Definition of Ready).

  • Independent (Independente): As histórias devem ser tão desvinculadas umas das outras quanto possível. Isso permite que a equipe puxe a História C para a Sprint sem ser obrigada a fazer a História A e B primeiro.
  • Negotiable (Negociável): Uma história NÃO é um contrato fixo. Os detalhes podem (e devem) ser alterados ou ajustados durante a conversa com a equipe de desenvolvimento.
  • Valuable (Valiosa): A história DEVE entregar valor para o usuário final ou cliente. Uma história tipo "Configurar banco de dados Oracle" não tem valor explícito pro cliente; o valor real é "Como cliente, quero buscar meus pedidos antigos para ver o histórico".
  • Estimable (Estimável): A equipe técnica deve ter clareza suficiente do escopo (através da Conversa) para conseguir estimar o tamanho/complexidade da história (usando Story Points, por exemplo).
  • Small (Pequena): Deve caber dentro de uma iteração/Sprint. Se a história leva 3 meses para ser feita, ela é grande demais e deve ser "fatiada" (Dividida).
  • Testable (Testável): Se você não consegue dizer como a história será testada, ela não está bem escrita. É aqui que entram os Critérios de Aceitação.

3. A Hierarquia Ágil: Épicos, Temas e Histórias

Como gerenciar requisitos gigantescos que não cabem em uma Sprint? As metodologias ágeis resolvem isso estruturando os requisitos em níveis de abstração.

🏔️

1. Tema

É o nível mais alto. Um grupo de grandes iniciativas relacionadas ou uma área estratégica do produto. (Ex: "Segurança do Sistema").

🐘

2. Épico (Epic)

Uma grande história de usuário que não cabe numa única Sprint. É um "monstro" que precisa ser fatiado antes de entrar no desenvolvimento. (Ex: "Como usuário, quero pagar minhas compras").

📝

3. User Story

O Épico "Pagamento" é quebrado em Histórias Pequenas e INVESTíveis. (Ex: "Pagar com PIX", "Pagar com Cartão de Crédito"). Estas vão para a Sprint.

4. Critérios de Aceitação (O 3º 'C' - Confirmation)

A frase "Como usuário, quero fazer login para ver meus dados" é muito vaga por si só. Como a equipe sabe que a tarefa de desenvolver o Login está pronta? Através dos Critérios de Aceitação.

Eles são os limites da história. São as regras de negócio escritas em forma de checklist ou no formato Dado-Quando-Então (Gherkin/BDD).

Exemplo de Critérios de Aceitação para a "História de Login":
  • O sistema não deve permitir senhas com menos de 8 caracteres.
  • Se a senha estiver errada 3 vezes, a conta deve ser bloqueada por 15 minutos.
  • A senha deve ser mascarada com asteriscos (***) durante a digitação.

Na prova: Se o software atende a todos os critérios de aceitação listados no verso do cartão, a história é considerada CONCLUÍDA (Done).
