Monitoramento e Observabilidade

⚡ A Diferença Crucial (Cai Muito!)

📊 Monitoramento

Foco no SINTOMA. Mostra o estado atual do sistema através de dashboards.
Responde a pergunta: "O sistema está quebrado?"

🕵️‍♂️ Observabilidade

Foco na CAUSA RAIZ. É a capacidade de entender o estado interno do sistema apenas olhando para suas saídas externas.
Responde a pergunta: "POR QUE o sistema quebrou?"

🏛️ Os 3 Pilares da Observabilidade (MEL)

O tripé que sustenta a resolução de problemas em arquiteturas complexas.

📈

Métricas

Dados numéricos medidos em intervalos de tempo. Usados para disparar alertas quando limites são ultrapassados.

Ex: "Uso de CPU está em 95%"
📝

Eventos / Logs

Registros textuais imutáveis de eventos específicos que ocorreram no sistema, com um timestamp (data e hora).

Ex: "Erro 500 às 14:02: User Not Found"
🔗

Traces (Rastros)

Rastreamento Distribuído. Acompanha o caminho exato de UMA requisição do usuário através de dezenas de microsserviços.

Ex: "A API de Login atrasou a API de Compras"

🧰 Sopa de Letrinhas (Ferramentas de Mercado)

Pilar / Função Ferramentas Padrão Ouro O que fazem?
Métricas & Alertas Prometheus Coleta métricas numéricas puxando dados (pull model). É o padrão para Kubernetes.
Visualização Grafana Cria os painéis (dashboards) bonitos e gráficos lendo os dados do Prometheus ou outros bancos.
Centralização de Logs ELK Stack (Elasticsearch, Logstash, Kibana) Coleta logs textuais de todos os servidores, indexa tudo para buscas ultra-rápidas e gera visões.
Distributed Tracing Jaeger / Zipkin Rastreia o caminho e a latência de uma requisição que passa por múltiplos microsserviços.

🚨 Pegadinhas Mortais de Prova

  • A Ilusão da Substituição: A banca vai jurar que "A Observabilidade é uma tecnologia mais moderna que substituiu a necessidade de fazer o Monitoramento tradicional".
    Falso! Um não substitui o outro; eles são complementares. O Monitoramento é a base necessária para você descobrir que tem um problema. A Observabilidade é o próximo passo, que te permite "cavar fundo" nos dados do monitoramento para achar o porquê do problema.
  • Logs resolvem tudo? "Para garantir total observabilidade em uma arquitetura de microsserviços, basta centralizar todos os arquivos de log num sistema robusto como o ELK Stack".
    Falso! Apenas Logs não garantem observabilidade em microsserviços. Sem Traces Distribuídos (Trace ID), você tem milhões de linhas de log soltas, mas é impossível saber qual linha de log do "Serviço A" se relaciona com a linha de erro do "Serviço B" para uma mesma requisição do usuário.

1. A Evolução: Por que a Observabilidade nasceu?

No passado, as aplicações eram "Monolíticas" (um único grande bloco de código rodando em um único servidor). Se o sistema caía, o Administrador de Sistemas olhava para os painéis de Monitoramento. Se o gráfico da CPU batesse 100%, ele reiniciava o servidor. O problema estava resolvido. O monitoramento foca nos "Known Unknowns" (Desconhecidos Conhecidos) – coisas que já sabemos que podem falhar, como CPU, RAM e Disco.

Hoje, as aplicações são em Nuvem, desenhadas com arquiteturas de Microsserviços. Uma simples compra online pode envolver o Serviço de Carrinho, o de Estoque, o de Pagamento e o de Nota Fiscal, espalhados em dezenas de contêineres que nascem e morrem a toda hora (Kubernetes).

Se a compra falhar, não adianta olhar a CPU. O problema pode ser um erro de código no microsserviço 14 que causou um timeout no microsserviço 3. É para investigar os "Unknown Unknowns" (Desconhecidos Desconhecidos) que a Observabilidade surgiu. Ela exige que a aplicação exporte dados suficientes para que possamos investigar problemas novos e imprevistos.

2. Dissecando os Pilares (Métricas, Logs, Traces)

A. Métricas (Metrics)

São representações numéricas de dados extraídos em intervalos de tempo (séries temporais). Elas são leves de armazenar, fáceis de processar matematicamente e excelentes para painéis gerais e alertas.

Métricas de Ouro (Four Golden Signals do Google SRE): Latência (Tempo de resposta), Tráfego (Demandas por segundo), Erros (Taxa de falhas) e Saturação (Quão cheio está o sistema).

B. Logs

Um log é um registro imutável com data/hora de eventos que aconteceram ao longo do tempo. Enquanto a métrica diz "Tivemos 50 erros 500 no último minuto", o log diz "Às 15:42, o Usuário X tentou consultar o Banco Y e falhou porque a Senha estava Errada". É o detalhe minucioso e pesado para armazenamento.

Prática comum: Logs Estruturados (gravar os logs em formato JSON) para que ferramentas como Elasticsearch consigam pesquisar os dados rapidamente.

C. Traces Distribuídos

Um trace representa a vida inteira de uma transação ou requisição enquanto ela viaja por um sistema distribuído.

Cada vez que a requisição bate em um microsserviço diferente, gera-se um Span (uma unidade de trabalho com início e fim). Todos os Spans de uma mesma requisição de usuário recebem o mesmo Trace ID. Assim, o engenheiro de DevOps consegue visualizar um "Gráfico de Gantt" mostrando exatamente onde a requisição gastou mais tempo.

3. A Filosofia SRE (Site Reliability Engineering)

SRE é a abordagem prática que o Google inventou para aplicar a cultura DevOps em operações de TI reais em grande escala. "SRE é o que acontece quando você pede a um engenheiro de software para projetar uma equipe de operações."

Para a prova, o foco supremo do SRE são os seus indicadores de disponibilidade. O SRE adota a premissa de que 100% de confiabilidade é o alvo errado (pois para tentar atingir 100%, você teria que parar de inovar e congelar o sistema).

O Vocabulário SRE (SLI, SLO, SLA)
  • SLI (Service Level Indicator): A Medida Real atual. O que eu estou de fato medindo no mundo real? (Ex: A latência das requisições web).
  • SLO (Service Level Objective): A Meta Interna de qualidade que a equipe se propõe a atingir, baseada no SLI. (Ex: "99% das requisições web devem retornar em menos de 200 milissegundos num período de 30 dias"). Se a meta não for batida, a equipe para de lançar novas features e foca só em corrigir estabilidade.
  • SLA (Service Level Agreement): O Acordo Contratual/Comercial. A meta assinada com o cliente que envolve punições financeiras (multas) se não for cumprida. (Ex: O contrato diz 98% de disponibilidade. Notem que o SLO interno da equipe é sempre mais alto e rigoroso do que o SLA externo do contrato, atuando como margem de segurança).

Error Budget (Orçamento de Erro)

Se o nosso SLO de disponibilidade é de 99.9%, significa que temos um Orçamento de Erro de 0.1% no mês (cerca de 43 minutos de indisponibilidade permitida). O Error Budget é uma métrica usada pelas equipes de Dev e Ops para "gastar". Enquanto houver orçamento, a equipe pode lançar novidades e arriscar. Se o orçamento estourar, os deploys de inovação são congelados.
