Projeto Centrado no Usuário (UCD)

⚡ Direto ao Ponto

  • O que é? O Projeto Centrado no Usuário (UCD - User-Centered Design) é uma abordagem iterativa que foca no entendimento profundo de quem usará o produto (suas necessidades, limitações, ambiente e tarefas).
  • Não é só UI (Tela): Envolve entender o problema do usuário antes mesmo de pensar em como a tela será desenhada.
  • O Pilar Mestre (Empatia): O princípio número um do UCD é a empatia. O desenvolvedor não é o usuário. Você precisa sair do prédio e conversar com as pessoas reais que usarão o sistema.

🔄 O Ciclo Iterativo (Norma ISO 9241-210)

Este é o fluxo oficial cobrado nas provas. Ele se repete até que os requisitos do usuário sejam plenamente satisfeitos.

1

Entender o Contexto

Quem vai usar? Onde? Para quê? (Ex: Operários usando o sistema num galpão barulhento com luvas).

2

Especificar Requisitos

Identificar as necessidades do usuário e os objetivos de negócio que o sistema deve cumprir.

3

Criar Soluções (Design)

Passar da teoria para a prática. Criar wireframes, mockups e protótipos interativos.

4

Avaliar o Design

Testar o protótipo com os usuários reais. Se não estiver bom, o ciclo volta ao passo 1 ou 3.

🛠️ Ferramentas Clássicas do UCD

Ferramenta O que é? Objetivo / Quando usar
Personas Personagens fictícios baseados em dados reais que representam os diferentes perfis de usuários. Gerar empatia. Fazer a equipe pensar "Como o João usaria essa funcionalidade?".
Mapa da Jornada do Usuário Representação visual do caminho que o usuário percorre para alcançar um objetivo, incluindo emoções. Identificar "pontos de atrito" (pain points) ou frustrações no processo atual.
Testes de Usabilidade Observar usuários reais tentando realizar tarefas num protótipo ou sistema. Fase final do ciclo UCD. Validar se a solução criada é realmente fácil de usar.

🚨 Pegadinhas Mortais de Prova

  • A Ilusão da Participação Final: A banca afirma que "No UCD, a participação do usuário restringe-se à fase de testes de homologação, para garantir que o software funcione sem erros".
    Falso! No Projeto Centrado no Usuário, o envolvimento do usuário é ativo e contínuo desde o primeiro dia (fase de concepção e entendimento do contexto) até a última etapa.
  • Usabilidade x Acessibilidade:
    👉 Usabilidade: Quão fácil e satisfatório é usar o sistema.
    👉 Acessibilidade: Garantir que pessoas com deficiências (visuais, motoras, etc.) *consigam* usar o sistema.
    Pegadinha: Acessibilidade é um pré-requisito. Se o sistema não é acessível, a usabilidade para aquela pessoa é nula.

1. A Filosofia do UCD (User-Centered Design)

A Engenharia de Software tradicionalmente focava-se no "Sistema" e na "Tecnologia". Os desenvolvedores criavam soluções geniais, mas os sistemas muitas vezes fracassavam porque o usuário final não conseguia operá-los ou achava o processo demasiadamente complexo.

O UCD (Projeto Centrado no Usuário) inverte a lógica. O foco primário não é a tecnologia de ponta, mas sim o ser humano que irá utilizar a tecnologia. É uma filosofia multidisciplinar que engloba a psicologia cognitiva, o design de interação (IxD) e a ergonomia.

Os Três Pilares da Usabilidade (ISO 9241-11)

Para a norma internacional, um sistema possui usabilidade se atingir três objetivos em um contexto específico de uso:

  • Eficácia (Effectiveness): O usuário consegue atingir o objetivo? (Ex: Conseguiu finalizar a compra?)
  • Eficiência (Efficiency): O usuário atinge o objetivo com o menor gasto de recursos/esforço possível? (Ex: Precisou de 2 cliques ou de 15 cliques?)
  • Satisfação (Satisfaction): O usuário sentiu-se confortável e teve atitudes positivas ao usar o sistema?

2. Ferramentas de Empatia: Personas

"Projetar para todos é não projetar para ninguém". O conceito de Personas, popularizado por Alan Cooper, é uma técnica poderosa de elicitação de requisitos.

O que é uma Persona?

Não é um "público-alvo" genérico (ex: "Homens de 20 a 30 anos"). A persona é a personificação de uma pesquisa. Você cria um perfil específico (ex: "Roberto, 28 anos, ansioso, acessa o app de pé no ônibus usando 4G ruim").

Objetivo (Foco de Prova): A persona tira o desenvolvimento do campo do "achismo". Quando a equipe discute uma funcionalidade, a pergunta deixa de ser "Eu gostei deste botão?" e passa a ser "O Roberto conseguiria clicar neste botão enquanto está em pé no ônibus?". Aumenta drasticamente a empatia da equipe.

3. A Jornada do Usuário (Customer Journey Map)

Muitos sistemas falham não porque a tela é feia, mas porque o processo/fluxo é longo e doloroso. O Mapa da Jornada do Usuário é uma técnica narrativa que mapeia, passo a passo, a experiência do usuário ao longo do tempo, desde o momento em que ele descobre a sua necessidade até a sua resolução.

Pontos Críticos (Touchpoints e Pain Points)

  • Touchpoints (Pontos de Contato): Qualquer momento em que o usuário interage com o sistema ou com a empresa (ex: receber um e-mail, clicar num anúncio, usar a barra de busca do site).
  • Pain Points (Pontos de Dor): São as quebras de expectativa. Os momentos de frustração, demora ou confusão. A engenharia de requisitos moderna foca-se nestes pontos de dor para criar Histórias de Usuário que proponham soluções para aliviá-los.

4. Heurísticas de Jakob Nielsen

Em provas de alto nível, quando o tema aborda Avaliação Heurística ou Usabilidade, as 10 Heurísticas de Nielsen são a base teórica. Você não precisa decorar as dez palavras exatas, mas precisa entender a sua lógica. Destacam-se para concursos:

  • 1. Visibilidade do status do sistema: O usuário sempre deve saber o que está acontecendo (ex: barra de progresso durante o download).
  • 3. Controle e Liberdade do Usuário: O usuário frequentemente clica em botões por engano. Deve haver sempre uma "saída de emergência" clara (ex: botões "Desfazer / Ctrl+Z").
  • 5. Prevenção de Erros: Melhor que mensagens de erro bonitas, é o design que impede o erro de ocorrer (ex: desativar o botão de "Enviar" até que o e-mail tenha o formato correto).
  • 6. Reconhecimento em vez de Recordação: Minimizar a carga de memória do usuário. Interfaces boas sugerem opções, o usuário não deve precisar decorar códigos.
  • 8. Estética e Design Minimalista: Diálogos não devem conter informações irrelevantes. Menos é mais (Redução do ruído visual).
