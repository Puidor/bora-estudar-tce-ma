Build, Testes e Deploy Automatizados

⚡ A Tríade da Automação (O Fim do "Na minha máquina funciona")

🔨

1. Build

Pega no código-fonte, descarrega as bibliotecas externas (dependências), compila tudo e gera um pacote executável fechado (ex: .JAR ou imagem Docker).

🧪

2. Teste

Executa as rotinas de verificação sem cliques humanos. Avalia regras de negócio, performance e segurança. Se 1 falhar, todo o processo é abortado.

🚀

3. Deploy

Pega no pacote aprovado (Build) e instala-o nos servidores, banco de dados ou nuvem, substituindo a versão antiga de forma automática e silenciosa.

🧰 Sopa de Letrinhas das Bancas (Mapeamento de Ferramentas)

Fase da Esteira Ferramentas de Mercado Objetivo Técnico
Automação de Build Maven, Gradle, npm, Webpack, MSBuild Resolver dependências, compilar e empacotar (JAR, WAR).
Automação de Testes JUnit, Selenium, Jest, SonarQube Verificar unidade, integração, UI e análise estática (SAST).
Orquestração (O "Chefe") Jenkins, GitLab CI, GitHub Actions, Bamboo Ligar as fases. Chama o Maven, depois o JUnit, depois faz o Deploy.
Automação de Infra/Deploy Ansible, Terraform, Docker, Kubernetes Infraestrutura como Código (IaC) e orquestração de containers.

🚨 Pegadinhas Mortais de Prova

  • Build é apenas Compilar o Código? A banca afirma: "A fase de build consiste unicamente na compilação do código-fonte para binário, não incluindo bibliotecas externas".
    Falso! Ferramentas de automação de Build (como o Maven) fazem muito mais. Elas resolvem e baixam dependências (bibliotecas na internet), limpam pastas antigas, compilam, e geram o empacotamento final (artefato), como um arquivo `.jar`.
  • Deploy manual evita falhas? "O deploy manual é recomendado em ambientes críticos bancários, pois o analista pode inspecionar o servidor, o que torna o processo menos propenso a erros do que scripts cegos".
    Falso! É exatamente o contrário! O erro humano é a maior causa de quedas de sistemas (esquecer de rodar um script de banco, copiar a pasta errada). O deploy automatizado garante 100% de repetibilidade e previsibilidade. A máquina não esquece passos.

1. Automação de Build e Imutabilidade

No passado, a geração da versão do software dependia da máquina do desenvolvedor ("Na minha máquina compila!"). O desenvolvedor compilava o código na sua IDE (Eclipse/Visual Studio), gerava o pacote e mandava por e-mail para o administrador do servidor. Isso gerava o famoso problema de diferenças de ambiente (versão do Java diferente, bibliotecas diferentes).

A Automação de Build centraliza este processo usando arquivos de configuração como o pom.xml (Maven) ou package.json (NPM). O servidor de CI (Jenkins) baixa o código "puro" do Git, lê esse arquivo e baixa exatamente as mesmas dependências sempre, gerando um Artefato Imutável.

Princípio de Prova (Imutabilidade): O artefato (ex: a imagem Docker) gerado na fase de Build é único. É exatamente o mesmo artefato físico que passará pela fase de Testes, de Homologação e de Produção. Você nunca deve compilar o código de novo no meio do pipeline, pois isso poderia gerar um binário diferente.

2. Automação de Testes na Esteira (Fail Fast)

Quando automatizamos testes dentro do pipeline CI/CD, o objetivo primário é o Feedback Loop Curto. A filosofia DevOps utiliza o conceito de Fail Fast (Falhe Rápido).

  • Se um desenvolvedor introduziu um bug crítico, não queremos descobrir isso daqui a uma semana no Teste de Aceitação manual.
  • A automação roda milhares de testes unitários (JUnit) nos primeiros 2 minutos após o envio do código.
  • Se houver falha, a automação para imediatamente a esteira (conhecido como Andon Cord na cultura Lean), impedindo que um software quebrado avance para a fase de empacotamento ou deploy.

3. Estratégias de Deploy Automatizado (Foco FGV/CEBRASPE)

Automatizar o Deploy (a implantação) é o ato de colocar o software no servidor sem intervenção manual, usando scripts ou ferramentas como Ansible, Terraform ou Kubernetes. No entanto, as bancas adoram cobrar as Estratégias Avançadas que permitem lançar uma versão nova sem que o usuário perceba que o servidor foi reiniciado (Zero Downtime).

A. Blue-Green Deployment

A equipe mantém dois ambientes de produção idênticos. O ambiente Azul (ativo, com os usuários) e o Verde (ocioso). O deploy automatizado joga a versão nova no ambiente Verde. Após testar tudo lá dentro, um "Roteador" (Load Balancer) apenas redireciona o tráfego de usuários do Azul para o Verde. Se der problema crítico, basta virar a chave de volta para o Azul imediatamente (Rollback instantâneo).

B. Canary Release (Deploy Canário)

Baseado na prática dos mineradores de levar um canário para as minas de carvão. Em vez de mudar 100% dos usuários para a versão nova de uma vez, o deploy automatizado direciona apenas 5% do tráfego para os servidores com a versão nova. Se não houver erros na monitoração durante algumas horas, o sistema aumenta automaticamente para 20%, 50% e finalmente 100%. Reduz o "raio de explosão" caso haja um bug oculto.

4. O Grande Confronto Teórico: Deploy vs Release

Embora sejam sinônimos no dia a dia, para as provas de Engenharia de Software Moderna (DevOps), existe uma fronteira clara que deve ser respeitada:

Deploy (Implantação)

É um ato Técnico. É o processo automatizado de colocar o código/binário no servidor de Produção para que ele rode na memória. No entanto, esse código pode estar "invisível" para o usuário final.

Release (Lançamento)

É um ato de Negócio. É o momento em que a funcionalidade é ativada e disponibilizada ao público alvo, gerando valor. É quando a campanha de marketing acontece.

A Mágica do Feature Toggle (Feature Flags)

As empresas conseguem separar o Deploy do Release usando "Feature Toggles". A automação de Deploy joga o código do "Novo Botão de Pix" em produção na segunda-feira. Porém, o código tem um "IF" associado a um painel de controle. O botão fica invisível (apagado). Na sexta-feira, o Diretor de Marketing clica num botão e a funcionalidade acende para todos os usuários (Release).
