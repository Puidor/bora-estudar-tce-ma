Produto Mínimo Viável (MVP)

⚡ Direto ao Ponto

  • O que é? A versão mais simples de um novo produto que permite à equipe coletar a máxima quantidade de aprendizado validado sobre os clientes com o menor esforço possível.
  • O Objetivo: Testar uma hipótese de negócio. Responder à pergunta: "As pessoas realmente querem pagar ou usar isso que estamos construindo?" antes de gastar milhões.
  • Lançamento Real: Diferente de um protótipo descartável de papel, o MVP é frequentemente um produto real, lançado para um subconjunto restrito de usuários reais (early adopters).

🚗 Como construir um MVP? (A Analogia)

Se a necessidade do cliente é "ir do Ponto A ao Ponto B mais rápido", como você entrega valor?

O jeito ERRADO (Cascata)

⚙️ ➔ 🛞 ➔ 🚘 ➔ 🚙

Você entrega 1 roda, depois 2 rodas, depois a carcaça. O cliente não consegue usar nada até o final do projeto. Não há entrega antecipada de valor.

O jeito CERTO (MVP)

🛹 ➔ 🛴 ➔ 🚲 ➔ 🚙

Você entrega um Skate (MVP). Ele já resolve a dor de "ir do ponto A ao B" (é Viável). Com o feedback, você evolui para Patinete, Bicicleta e finalmente o Carro.

⚖️ O Confronto: PoC x Protótipo x MVP

Este é o ponto onde 90% dos candidatos erram.

Técnica O que valida? (A Pergunta) Público-Alvo
PoC (Prova de Conceito) "É tecnicamente possível fazer isso?" Valida a tecnologia/viabilidade. Equipe Técnica (Interno).
Protótipo "Como isso vai se parecer ou funcionar?" Valida Design, Usabilidade e UX. Stakeholders ou Grupo pequeno de testes.
MVP "Alguém vai comprar/usar isso?" Valida a Hipótese de Negócio. Usuários Finais reais (Early Adopters).

🚨 Pegadinhas Mortais de Prova

  • MVP é software com defeito? A banca afirma que "Para garantir a velocidade de lançamento do MVP, a equipe deve ignorar a qualidade do código, testes e a usabilidade (Mínimo)".
    Falso! O MVP corta no Escopo/Funcionalidades (foca só na principal), mas NÃO corta na qualidade. O software lançado deve funcionar bem, ser confiável e agradável para a funcionalidade restrita a que se propõe. (Lembre-se: é Mínimo, mas é Viável).
  • MVP é a primeira fase do projeto Cascata? "No modelo tradicional (Waterfall), o MVP corresponde à fase final de testes de aceitação".
    Falso! O MVP é um conceito indissociável de metodologias ágeis e iterativas (Lean/Scrum). No Cascata clássico, você constrói o sistema 100% antes de ver se o cliente o quer.

1. A Origem: O Livro "A Startup Enxuta"

Para provas que envolvem gestão de produtos (Product Management), o termo MVP está intimamente ligado ao autor Eric Ries e ao conceito de Lean Startup.

A premissa é simples: as empresas gastam milhões e anos de desenvolvimento para criar o "produto perfeito" cheio de dezenas de funcionalidades (features). Quando lançam o produto no mercado, descobrem que o cliente não se importa com a maioria daquelas funcionalidades, e o projeto fracassa.

O Ciclo Construir-Medir-Aprender (Build-Measure-Learn)

O MVP existe para rodar esse ciclo o mais rápido possível:
1. Construir a versão essencial do produto (o MVP).
2. Entregá-lo aos clientes e Medir o seu comportamento real (dados, não "achismos").
3. Aprender com os dados: o cliente realmente pagou por isso? Se sim, a equipe itera e adiciona a funcionalidade 2. Se não, a equipe Pivota (muda radicalmente a estratégia de negócio).

2. A "Fatia" Vertical de Qualidade

Um conceito muito abordado por autores de Engenharia de Software é a "Pirâmide do Produto". Nela, um produto completo é dividido em várias fatias horizontais de qualidade: Funcional, Confiável, Utilizável (Usável) e Emocional (Agradável).

A Abordagem Incorreta

Muitas empresas acham que o MVP deve ser apenas "Funcional". Entregam um sistema feio, com bugs constantes, difícil de usar e que cai o tempo todo.

A Abordagem Correta (Fatia Vertical)

Um verdadeiro MVP deve entregar uma pequena fatia funcional (menos funcionalidades), mas essa fatia deve englobar todas as camadas de qualidade. O "Skate" da analogia não tem motor, mas ele deve ser perfeitamente pintado, ter as rodinhas ajeitadas e funcionar sem quebrar no meio da calçada. Assim o cliente tem uma boa experiência com aquela única funcionalidade principal.

3. Formatos Não-Codificados de MVP

Surpresa para muitos desenvolvedores: o MVP ideal às vezes nem tem código de software desenvolvido! Se o objetivo é testar a hipótese de negócio, você pode fazê-lo de outras formas:

  • O MVP Concierge: A empresa finge que tem um software complexo. Você preenche um formulário e no dia seguinte a empresa envia o resultado. Na verdade, por trás da tela, há seres humanos fazendo todo o trabalho manualmente (em Excel) no lugar de uma Inteligência Artificial. Serve para ver se o cliente vê valor no resultado antes de gastar 1 ano codificando a IA.
  • O MVP Mágico de Oz: Semelhante ao Concierge, mas aqui o usuário realmente acha que está lidando com um sistema 100% automatizado, enquanto a equipe humana trabalha freneticamente nos bastidores para "simular" o software em tempo real (como o sapateiro escondido atrás da cortina no Mágico de Oz).
  • O MVP "Fumaça" (Smoke Test / Landing Page): A equipe cria um site de vendas impecável para o software com um botão "Compre Agora". Quando o usuário clica, aparece a mensagem: "Ainda estamos em desenvolvimento. Deixe seu e-mail". O objetivo é medir a conversão: se 10.000 pessoas clicarem, a hipótese está validada, a equipe pode começar a programar.

4. O MVP no Scrum e Mapeamento de Produto

No framework Scrum, o MVP é frequentemente o alvo de um dos primeiros "Releases" (lançamentos) de um produto recém-criado.

O Product Owner (PO) olha para o Product Backlog (que pode ter 200 histórias de usuário) e traça uma linha horizontal de corte. Tudo o que está acima da linha compõe o "Mínimo" necessário para testar o núcleo do negócio; esse será o escopo do MVP. O que está abaixo da linha ficará para as próximas Sprints (versão 2.0).

"O MVP não é o produto final com a maioria das funcionalidades cortadas. É o produto núcleo com todo o ruído de fundo removido para garantir foco na validação de aprendizado."
