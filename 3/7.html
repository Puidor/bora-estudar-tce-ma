⚡ A Tríade da Medição (Cai muito!)

Em provas, as bancas adoram testar se você sabe a diferença exata entre estes três conceitos de Pressman:

Medida (Measure)

É o valor bruto quantitativo. Uma indicação quantitativa simples.

Ex: 5.000 linhas de código.

Métrica (Metric)

A relação matemática entre duas medidas. Avalia um atributo do sistema.

Ex: 5 erros por cada mil linhas.

Indicador (Indicator)

É uma métrica projetada para tomada de decisão. Diz se a meta foi atingida.

Ex: A qualidade do software caiu.

📐 Análise de Pontos de Função (APF)

O método oficial do Governo Federal para medir tamanho de software. Baseia-se na visão do usuário, e não nas linhas de código (LOC).

Os 5 Elementos da APF

Funções de DADOS (Armazenamento)

  • ALI (Arquivo Lógico Interno): Mantido dentro da fronteira da aplicação (Ex: Tabela de Clientes do seu próprio banco de dados).
  • AIE (Arquivo de Interface Externa): Apenas lido/referenciado pela aplicação, mas mantido por um sistema externo (Ex: Consultar tabela de CEP dos Correios).

Funções de TRANSAÇÃO (Movimento)

  • EE (Entrada Externa): Processa dados vindos de fora para alterar/inserir algo dentro (Ex: Tela de cadastro de produto).
  • SE (Saída Externa): Envia dados para fora com cálculo ou lógica extra (Ex: Relatório mensal com somatórios).
  • CE (Consulta Externa): Envia dados para fora sem cálculo (recuperação simples). (Ex: Pesquisar nome de um cliente).

⚖️ O Confronto: LOC vs Pontos de Função

Característica LOC (Linhas de Código) APF (Pontos de Função)
Dependência de Linguagem Totalmente dependente Independente da tecnologia
Momento de Medição Só pode ser medida com precisão no final (após codificar). Pode ser medida no início (fase de requisitos).
Foco Visão do Desenvolvedor. Visão do Usuário (Valor de negócio).

🚨 Pegadinhas Mortais de Prova

  • LOC (Lines of Code) é boa para comparar equipes? A banca afirma: "A métrica LOC é ideal para comparar a produtividade entre uma equipe que programa em Assembly e outra em Python".
    Falso! Uma tela feita em Python pode exigir 20 linhas, enquanto em Assembly exigirá 2.000 linhas. A equipe de Assembly não é 100x mais produtiva. A métrica LOC penaliza linguagens modernas de alto nível.
  • Pontos de Função Complexos? "Os Pontos de Função avaliam a complexidade algorítmica interna do sistema (ex: cálculos matemáticos avançados no backend)".
    Falso! APF foca no que o usuário vê e faz (Entradas, Saídas, Consultas). Um botão que faz um cálculo matemático assustador de 10 mil linhas, mas que só gera uma saída simples na tela, pode gerar poucos pontos de função, pois a APF ignora complexidade puramente interna de código.

1. A Necessidade das Métricas

"Você não pode gerenciar o que não pode medir". As métricas de software servem para estimar custos de projetos, medir a produtividade da equipe, avaliar a qualidade do código e prever prazos de entrega.

Métricas Diretas vs Indiretas

  • Métricas Diretas: Podem ser medidas imediatamente e sem subjetividade.
    Exemplos: Linhas de código (LOC), velocidade de execução, tamanho do arquivo executável, número de defeitos relatados.
  • Métricas Indiretas: Medem atributos intangíveis da qualidade do software. São derivadas e calculadas.
    Exemplos: Usabilidade, confiabilidade, manutenibilidade, complexidade, Pontos de Função.

2. Complexidade Ciclomática (McCabe)

Criada por Thomas McCabe, a Complexidade Ciclomática é uma métrica de software que fornece uma medida quantitativa da complexidade lógica de um programa. As bancas cobram a sua aplicação e a sua fórmula.

Ela mede o número de caminhos lógicos independentes através de um programa. Quanto maior a complexidade, mais difícil é manter o código e mais testes unitários serão necessários para garantir 100% de cobertura.

A Fórmula Matemática: V(G) = E - N + 2

Baseado na Teoria dos Grafos:

  • V(G): Complexidade Ciclomática do Grafo.
  • E (Edges): Número de arestas (as setas/ligações).
  • N (Nodes): Número de nós (os blocos de comando/condições).
  • + 2: Constante padrão da fórmula.

*Macete de Prova: Se houver apenas 1 única estrutura de controle `IF/ELSE` (um nó condicional) no código, a complexidade será 2 (um caminho para o IF, um caminho para o ELSE).

3. Métricas Orientadas a Objetos (CK Metrics)

Chidamber e Kemerer (CK) propuseram a suíte de métricas mais famosa para avaliar o design de sistemas Orientados a Objetos (OO). Em concursos de alto nível, os nomes das siglas e o que indicam se estiverem "altas" são cobrados.

  • WMC (Weighted Methods per Class): Soma da complexidade dos métodos de uma classe. Se o WMC for alto, a classe é complexa, mais propensa a falhas e fere o princípio da responsabilidade única.
  • CBO (Coupling Between Object classes): Mede o número de outras classes a que uma classe está acoplada (depende). Cai muito! Um CBO alto é péssimo (alta dependência, difícil de reusar e testar). Lembre-se: queremos sempre Baixo Acoplamento.
  • DIT (Depth of Inheritance Tree): A profundidade da árvore de herança. Se for muito alta, há maior reuso de código, mas o sistema fica super complexo para entender ("quem herdou de quem?").
  • LCOM (Lack of Cohesion in Methods): Falta de Coesão. Mede quão independentes são os métodos dentro de uma mesma classe. Se o LCOM for alto, a classe não é coesa e provavelmente deveria ser dividida em duas.

4. Aprofundando em Pontos de Função (APF)

A APF é a métrica padrão para contratos de software no governo. O processo de medição (definido pelo IFPUG) é complexo, mas as provas focam nas definições dos Tipos de Função.

A. Funções de Dados (Arquivos lógicos)

  • ALI (Arquivo Lógico Interno): Grupo lógico de dados mantido por dentro da fronteira da aplicação. (Ex: A tabela de "Funcionários" onde o seu sistema faz Insert/Update/Delete).
  • AIE (Arquivo de Interface Externa): Grupo lógico de dados referenciado pela sua aplicação, mas mantido dentro da fronteira de outra aplicação externa. (Ex: O seu sistema consulta a cotação do Dólar no sistema do Banco Central).

B. Funções de Transação (O que o usuário faz)

  • EE (Entrada Externa): Processa dados que vêm de fora para dentro. O objetivo primário é manter um ALI (fazer Insert, Update ou Delete) ou alterar o comportamento do sistema.
  • SE (Saída Externa): Envia dados para fora. O objetivo primário é apresentar informações ao usuário, contendo obrigatoriamente lógica de processamento extra ou cálculos matemáticos (ex: Um relatório financeiro com totais e subtotais).
  • CE (Consulta Externa): Envia dados para fora, mas SEM conter lógica de processamento extra. O objetivo primário é a simples recuperação (SELECT) de dados que já estão guardados. (Ex: Digitar um nome e a tela mostrar os dados do cliente).
Passo Final da Contagem: Depois de identificar as funções, analisa-se a complexidade de cada uma delas (Baixa, Média ou Alta) com base na quantidade de campos (DER) e arquivos referenciados (ALR). Cada nível de complexidade atribui um peso numérico. A soma gera os Pontos de Função Não Ajustados.
