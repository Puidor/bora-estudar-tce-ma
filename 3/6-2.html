⚖️ Regra de Ouro: O que automatizar?

✅ IDEAL PARA AUTOMAÇÃO

  • Testes de Regressão: Aqueles que rodam toda vez que uma linha de código muda (CI/CD).
  • Testes Repetitivos: Tarefas tediosas e propensas a erro humano (ex: preencher o mesmo formulário 50 vezes com dados diferentes).
  • Testes de Carga/Estresse: É impossível simular 5.000 usuários clicando ao mesmo tempo manualmente.

❌ NÃO DEVE SER AUTOMATIZADO

  • Testes de Usabilidade (UX): A máquina não sabe se o botão vermelho "causa uma sensação de alerta" ou se a interface é "amigável". Exige humano.
  • Testes Exploratórios: Quando o testador age como um "hacker", clicando em lugares aleatórios para descobrir falhas imprevistas. A máquina só testa o script fixo.
  • Requisitos instáveis: Telas que mudam de layout todos os dias (o custo de manutenção do teste seria maior que o benefício).

🚨 Pegadinhas Mortais de Prova

  • A Ilusão do "100% Automatizado": A banca afirma que "No desenvolvimento Ágil e DevOps, a meta é atingir 100% de automação, abolindo o teste manual".
    Falso! Testes exploratórios, testes de UX e auditorias de segurança criativas sempre exigirão validação manual humana.
  • Selenium testa Banco de Dados? "O Selenium é uma ferramenta completa que permite testar a consistência de tabelas SQL no banco de dados e APIs REST".
    Falso! O Selenium atua exclusivamente no navegador (Web UI). Para testar API usaríamos o Postman ou RestAssured.

1. A Essência da Automação e o ROI

Por que não testar tudo à mão? Porque o teste manual não escala. Se um sistema tem 500 funcionalidades e um desenvolvedor altera uma linha de código, o testador manual teria que repassar horas testando as outras 499 para garantir que nada foi "quebrado" (Teste de Regressão).

A Automação de Testes é o uso de software (separado do software sendo testado) para controlar a execução de testes, comparar os resultados reais com os esperados e gerar relatórios.

O Paradoxo do ROI (Return on Investment)

Criar um teste automatizado custa muito mais caro e demora muito mais tempo do que executar o teste manualmente a primeira vez. O ROI da automação só se justifica pela repetição contínua ao longo do tempo. Se uma tela for ser usada (e testada) apenas uma vez e for descartada, não vale a pena automatizá-la.

2. Selenium (O Rei da UI Web)

Se a prova falar sobre "Testes de Interface (UI)", "Automatizar cliques no navegador", "Testes ponta-a-ponta (E2E) web", a resposta em 99% dos casos é o Selenium.

Ele não é apenas uma ferramenta, mas uma suíte. O componente mais famoso e cobrado é o Selenium WebDriver. Ele aceita comandos em várias linguagens (Java, Python, C#) e as traduz em ações diretas no navegador (Chrome, Firefox), exatamente como um humano faria (clicar, rolar a tela, ler um texto, preencher input).

Nota Moderna (Cypress): Bancas mais atualizadas podem cobrar o Cypress como a principal alternativa moderna ao Selenium. O Cypress roda dentro do próprio navegador junto com a aplicação web, oferecendo testes de frontend muito mais rápidos e estáveis, mas é focado estritamente em JavaScript/TypeScript.

3. Ferramentas Unitárias e a Família xUnit

Para suportar a prática do TDD, surgiram os frameworks xUnit. Eles formam a base da pirâmide de testes. O mais cobrado é o JUnit (para Java).

Anotaçoes Críticas do JUnit

As bancas adoram colocar trechos de código e perguntar o que as anotações (annotations) fazem na ordem de execução:

  • @Test: Indica que aquele método é efetivamente um caso de teste que deve ser rodado.
  • @BeforeEach / @Before: Roda antes de cada método @Test. Usado para "limpar a mesa" e preparar os dados (setup).
  • @AfterEach / @After: Roda após cada teste, geralmente para fechar conexões ou apagar dados sujos de teste (teardown).
  • @BeforeAll / @BeforeClass: Roda apenas UMA vez no início de tudo (antes do primeiro teste iniciar).

Asserções (Assertions)

O coração do teste automatizado é o Assert. É ele quem faz a máquina comparar o resultado que a função devolveu com o resultado que esperávamos. (Ex: assertEquals(4, calculadora.somar(2, 2))). Se o resultado for 5, a asserção falha e o teste fica vermelho.

4. Testes Não Funcionais e Integração Contínua

JMeter (Teste de Carga)

O Apache JMeter não testa "se" o botão funciona, ele testa se o botão continua funcionando quando 10.000 pessoas clicam nele. Ele usa o conceito de Thread Groups, onde cada thread simula um usuário virtual (VU). Ele consegue disparar milhares de requisições HTTP, consultas JDBC (Banco de Dados) e chamadas SOAP/REST simultaneamente, medindo o Tempo de Resposta e a Vazão (Throughput).

Postman / SoapUI (Teste de APIs)

Encaixam-se no nível de "Testes de Integração". As arquiteturas modernas são feitas de Microsserviços e APIs REST. O Postman permite disparar requisições automatizadas (GET, POST) contra os servidores e validar se o JSON retornado tem o status HTTP 200 (OK) e os dados corretos, muito antes de existir uma tela visual pronta.

A Orquestração Final: CI (Continuous Integration)

Ter testes automatizados na máquina do desenvolvedor não serve de muito para o projeto todo. As ferramentas de automação (JUnit, Selenium) são engatadas num orquestrador de CI, como o Jenkins ou GitLab CI.

A regra ágil é: sempre que um desenvolvedor enviar código para o repositório, o Jenkins "acorda", compila o sistema, roda 100% dos testes do JUnit e do Selenium automaticamente. Se 1 único teste falhar, o Jenkins bloqueia a atualização (Build Broken) e avisa a equipe. Isso garante a qualidade inabalável do código na esteira DevOps.
