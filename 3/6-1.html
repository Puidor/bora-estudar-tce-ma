Tipos e Níveis de Teste

🔺 A Pirâmide de Testes

TESTES DE ACEITAÇÃO / UI

Lentos, Caros, Foco no Usuário Final (Ponta a Ponta / E2E).

Quantidade: Poucos

TESTES DE INTEGRAÇÃO

Comunicação entre módulos, APIs, Banco de Dados.

Quantidade: Moderada

TESTES UNITÁRIOS

Rápidos, Baratos, Isolados. Foco no Código (Funções/Classes).

Quantidade: MUITOS (A Base de Tudo)

⚖️ Níveis de Teste Funcional

Nível O que testa? Quem faz?
1. Unitário A menor parte do software (um método ou função) de forma isolada (usando Mocks/Stubs). Caixa Branca. Desenvolvedor
2. Integração A comunicação entre dois ou mais componentes que já passaram no teste unitário. Se as interfaces se "falam" bem. Dev / QA Automático
3. Funcional (Sistema) O sistema inteiro. Se ele cumpre os requisitos funcionais (O que ele deveria fazer). Caixa Preta. QA / Tester
4. Aceitação A validação final. O sistema atende às necessidades reais do negócio? (UAT - User Acceptance Test). Cliente / Usuário

⚙️ Testes Não Funcionais (Atributos de Qualidade)

⏱️

Desempenho

Mede a rapidez e a capacidade de resposta do sistema sob uma carga normal de trabalho. Quanto tempo a tela demora a carregar?

🏋️

Carga (Load)

Testa o sistema no limite máximo ESPERADO de uso. Ex: O sistema foi feito para suportar 1.000 usuários simultâneos; colocamos 1.000 usuários.

🛡️

Vulnerabilidade

Testa a segurança. Procura proativamente por brechas, portas abertas, injeção de SQL ou bibliotecas desatualizadas (SAST / DAST).

🚨 Pegadinhas Mortais de Prova

  • Validação vs Verificação (Sommerville/Pressman):
    👉 Verificação: Estamos construindo o produto corretamente? O código está livre de bugs? (Testes Unitários / Integração).
    👉 Validação: Estamos construindo o produto certo? Atende ao que o cliente queria no mundo real? (Testes de Aceitação).
  • Teste de Carga vs Teste de Estresse: A banca vai jurar que são sinônimos.
    Falso! Carga vai até o máximo que foi planejado suportar. Estresse ultrapassa propositalmente esse limite (ex: envia 5.000 usuários para um sistema feito para 1.000) para ver como o sistema cai e se ele se recupera graciosamente ou corrompe o banco de dados.

1. Caixas de Teste (Visão Interna vs Externa)

A forma como o testador enxerga o sistema dita a abordagem do teste. As bancas usam o conceito de "Caixas" para classificar isso:

  • Caixa Branca (White Box / Estrutural): O testador vê o código-fonte e conhece a lógica interna. Ele desenha testes para passar por todos os caminhos do código (If/Else, Loops).
    Nível Típico: Teste Unitário. Quem faz: Desenvolvedor.
  • Caixa Preta (Black Box / Funcional): O testador NÃO vê o código. O software é uma caixa fechada. O teste foca nas Entradas e nas Saídas esperadas (Requisitos Funcionais). O testador clica num botão e vê se a tela de sucesso aparece, não importa o SQL que rodou por trás.
    Nível Típico: Testes de Sistema e Aceitação.
  • Caixa Cinza (Grey Box): Uma mistura. O testador não tem acesso total ao código-fonte, mas conhece parte das estruturas de dados internas ou algoritmos (ex: ele sabe que os dados vão para uma tabela X e usa scripts SQL para validar o resultado de um clique na tela).

2. Estratégias de Teste de Integração

Quando temos vários módulos (já testados de forma unitária), precisamos juntá-los para ver se as "costuras" (interfaces) funcionam. A forma como juntamos esses módulos gera questões teóricas clássicas:

  • Big Bang: Todos os módulos são integrados de uma só vez e o sistema inteiro é testado. É péssimo para sistemas grandes, porque se der erro, é extremamente difícil descobrir onde está a falha no meio de todo aquele código.
  • Top-Down (De Cima para Baixo): Começa testando o módulo principal (a tela/interface de usuário) e vai descendo para as funções menores. Exige o uso de STUBS (Dublês) para simular os módulos de baixo que ainda não foram integrados.
  • Bottom-Up (De Baixo para Cima): Começa testando os módulos mais básicos e fundamentais (ex: acesso a banco de dados) e vai subindo. Exige o uso de DRIVERS (Programas chamadores) para simular a tela/módulo superior que ainda não existe, chamando as funções de baixo.
  • Sandwich (Híbrido): Combina Top-down e Bottom-up simultaneamente, testando do topo e da base até se encontrarem no meio.

3. Testes de Aceitação: Alpha e Beta

O Teste de Aceitação é o momento da verdade. O software está pronto para ser entregue? Para softwares comerciais (de prateleira, como um jogo ou sistema operacional), o Teste de Aceitação é geralmente dividido em duas fases, amplamente cobradas:

Teste Alpha (α)

Realizado por um grupo restrito de potenciais usuários, mas dentro do ambiente do desenvolvedor. O desenvolvedor "olha por cima do ombro" do usuário, anota as falhas e ajusta o software imediatamente no ambiente controlado.

Teste Beta (β)

Realizado por um grupo muito maior de usuários reais, no próprio ambiente do cliente, sem a presença física do desenvolvedor. Se o sistema der falha lá, o usuário reporta (envia log de crash). O objetivo é expor o software à "selva" do mundo real e hardwares diversos.

4. Testes Contínuos (Regressão e Fumaça)

No desenvolvimento Ágil e no DevOps, alterar o código é uma rotina diária. Mas como garantir que consertar um bug não quebrou outra coisa que já funcionava?

Teste de Regressão

Sempre que uma nova funcionalidade é adicionada ou um bug é corrigido, um conjunto de testes antigos é rodado novamente (geralmente de forma automática via CI/CD). O objetivo é garantir que o software não "regrediu" (não quebrou o que estava funcionando).

Teste de Fumaça (Smoke Test)

Originário da eletrônica ("se você ligar na tomada e não sair fumaça, o circuito base funciona"). É um subconjunto de testes muito rápido que verifica apenas as funcionalidades mais críticas da aplicação logo após um novo Build. Se o Smoke Test falhar, a equipe rejeita a versão imediatamente, poupando tempo de testes profundos num código que está fundamentalmente quebrado.
