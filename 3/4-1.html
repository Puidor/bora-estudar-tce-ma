Elicitação de Requisitos e UX

⚡ Direto ao Ponto

  • Elicitação não é Levantar: "Levantar" implica que os requisitos estão prontos num canto só esperando serem recolhidos. "Elicitar" significa descobrir, provocar e negociar, pois muitas vezes o cliente não sabe o que quer.
  • A União com UX: A Engenharia de Requisitos tradicional foca no Software. A Experiência do Usuário (UX) foca no Ser Humano. Unir ambos garante que o software construído seja útil, usável e desejável.

⚖️ Tipos de Requisitos (Core Decoreba)

Requisitos Funcionais (RF)

Descrevem O QUE o sistema deve fazer. As funcionalidades e serviços que o sistema fornece ao usuário.

Exemplos:
- "O sistema deve emitir um relatório de vendas."
- "O usuário deve conseguir calcular o frete."

Req. Não Funcionais (RNF)

Descrevem COMO o sistema deve se comportar. São as restrições de qualidade, performance e segurança.

Exemplos:
- "O relatório deve ser gerado em até 2 segundos."
- "O sistema deve ter alta Usabilidade."

👥 Técnicas de UX na Elicitação

🧑‍💼

Personas

Personagens semi-fictícios que representam o seu usuário ideal. Ajuda a equipe a ter empatia e foco em "Para quem" estamos construindo o software.

🗺️

Jornada do Usuário

Mapeamento visual de todos os passos que o usuário dá para atingir um objetivo, identificando suas dores e oportunidades de melhoria.

🔍

Observação (Etnografia)

Ir ao ambiente do usuário e assistir como ele trabalha calado. Descobre requisitos ocultos que o usuário esquece de falar na entrevista.

🚨 Pegadinhas Mortais de Prova

  • A Ilusão UI = UX: A banca afirma: "A principal função da Experiência do Usuário (UX) é criar interfaces gráficas bonitas e atrativas (User Interface - UI)".
    Falso! UI (Interface) é o que o usuário VÊ (telas, botões, cores). UX (Experiência) é o que o usuário SENTE e VIVE ao longo de todo o processo (facilidade de uso, performance, valor entregue). UI é apenas uma parte da UX.
  • Usabilidade é Requisito Funcional? "O requisito de que o sistema deve ser fácil de usar e permitir o cadastro em até 3 cliques é um requisito funcional".
    Falso! Usabilidade é um atributo de qualidade, portanto, é sempre um Requisito Não Funcional.

1. A Engenharia de Requisitos

Requisitos são as descrições dos serviços fornecidos pelo sistema e de suas restrições operacionais. A Engenharia de Requisitos engloba todas as tarefas para entender, documentar e gerenciar essas necessidades.

O Conceito de Elicitação

Na literatura clássica (Pressman, Sommerville), não usamos mais o termo "Levantamento de Requisitos". A palavra Elicitação é usada para demonstrar que os requisitos não estão simplesmente prontos na cabeça do cliente esperando para serem "levantados".

Eles precisam ser descobertos, provocados e construídos através da interação. O cliente frequentemente não sabe o que quer, sabe o que não quer, ou tem necessidades conflitantes com outros departamentos da mesma empresa.

2. Tipos de Requisitos (Classificação)

A classificação dos requisitos é um dos assuntos mais cobrados em provas básicas e intermediárias de TI.

  • Requisitos Funcionais (RF):
    Declarações de serviços que o sistema deve fornecer, de como o sistema deve reagir a entradas específicas e de como deve se comportar em determinadas situações.
    Exemplo: "O sistema deve calcular o valor do frete com base no CEP inserido".
  • Requisitos Não Funcionais (RNF):
    Restrições sobre os serviços e funções oferecidos pelo sistema. Geralmente aplicam-se ao sistema como um todo, não a funcionalidades individuais. Incluem:
    • Usabilidade: Facilidade de uso, tempo de aprendizado.
    • Desempenho/Performance: Tempo de resposta (ex: "em até 1 segundo").
    • Confiabilidade/Disponibilidade: Tolerância a falhas, tempo no ar (ex: "99,99% de uptime").
    • Segurança: Criptografia, controle de acesso.
  • Regras de Negócio (RN):
    São restrições ou políticas da empresa que afetam o funcionamento do sistema, independentemente de haver um software ou não.
    Exemplo: "Nenhum cliente pode ter limite de crédito maior que 30% do seu salário comprovado".

3. A União: Elicitação Orientada ao Usuário (UX)

Historicamente, a engenharia de requisitos focava em preencher planilhas e formulários pesados com o que os "Gerentes" queriam. O problema? Os gerentes compram o software, mas são os "Operadores" que o usam no dia a dia.

A integração de UX (Experiência do Usuário) na elicitação traz o Design Centrado no Usuário (UCD - User-Centered Design). Significa descobrir os requisitos conversando e observando quem realmente vai clicar nos botões, garantindo que as funcionalidades criadas resolvam as dores reais (e não imaginárias) do usuário final.

Diferencial de Prova: UX engloba a usabilidade, acessibilidade, utilidade e conveniência do produto. Quando usamos UX na elicitação, o foco muda de "O que o sistema faz?" para "O que o usuário consegue fazer/atingir usando o sistema?".

4. Técnicas de Elicitação (Tradicionais e UX)

As bancas exigem que você saiba quando usar qual técnica. Cada técnica tem um propósito específico.

A. Técnicas Clássicas

  • Entrevistas: Muito boas para entender a visão geral. Podem ser fechadas (roteiro fixo) ou abertas (exploratórias).
  • Questionários: Ideais quando os usuários estão geograficamente dispersos e são em grande quantidade (centenas de pessoas).
  • Análise de Documentos: Ler os manuais e formulários de papel que a empresa usa hoje. Excelente para descobrir regras de negócio ocultas.
  • JAD (Joint Application Design): Sessões intensivas (workshops) de vários dias com todos os envolvidos (desenvolvedores, clientes, diretores) trancados numa sala para definir requisitos juntos e resolver conflitos rapidamente.

B. Técnicas Baseadas em UX

  • Observação / Etnografia: O analista vai ao local de trabalho e assiste o usuário trabalhando, em silêncio. A melhor técnica para descobrir requisitos ocultos (coisas que o usuário faz no automático e esquece de mencionar na entrevista).
  • Personas: Criação de perfis fictícios ("João, 45 anos, caminhoneiro que tem dificuldade com telas pequenas") para guiar o design e a priorização de funcionalidades.
  • Jornada do Usuário (User Journey Map): Um diagrama que ilustra o passo a passo do usuário desde o momento que ele tem um problema até o momento que ele resolve usando o sistema. Identifica pontos de atrito (pain points).
