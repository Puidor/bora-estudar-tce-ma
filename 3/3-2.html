TDD (Desenvolvimento Guiado por Testes)

⚡ Direto ao Ponto

  • A Inversão Clássica: O TDD (Test-Driven Development) vira o modelo tradicional de ponta-cabeça. Em vez de escrever o código e depois testá-lo, o programador escreve PRIMEIRO um teste automatizado que falha, e só depois escreve o código de produção.
  • O Nível do Teste: O TDD atua no nível mais baixo da pirâmide de testes. Estamos falando estritamente de Testes Unitários (de Unidade).
  • Quem faz? É uma prática técnica executada pelo desenvolvedor, e não pela equipe de Qualidade (QA) ou pelo cliente. Nasceu na metodologia Extreme Programming (XP).

🔄 O Ciclo Sagrado (Red - Green - Refactor)

Este é o fluxo de trabalho obrigatório que você tem que memorizar para a prova.

1

RED (Vermelho)

Escreva um teste unitário que FALHE. O teste deve definir uma funcionalidade que ainda não existe no código de produção.

2

GREEN (Verde)

Escreva o código de produção mais simples possível (mesmo que mal feito ou "feio") apenas para fazer o teste passar (ficar verde).

3

REFACTOR

Volte ao código que escreveu no passo 2 e melhore a sua qualidade/arquitetura (refatoração), garantindo que o teste continua verde. Remova duplicações.

🚨 Pegadinhas Mortais de Prova

  • TDD dispensa os outros testes? A banca afirma: "A adoção plena do TDD anula a necessidade de execução de Testes de Integração ou de Aceitação".
    Falso! O TDD garante que as *unidades* (blocos individuais de código) funcionam, mas não garante que elas se comuniquem bem entre si (Integração) nem que a funcionalidade agrade ao usuário (Aceitação/BDD).
  • TDD gera código lento? "A prática do TDD aumenta excessivamente o tempo total de desenvolvimento de um projeto de software, não sendo recomendada para metodologias ágeis".
    Falso! No início a curva de aprendizado atrasa a codificação inicial, mas a longo prazo (que é o que importa), o TDD reduz drasticamente o tempo gasto na correção de *bugs* (debugging), tornando o ciclo de entrega de valor mais rápido e seguro no Ágil.

1. A Origem: XP e Kent Beck

O TDD (Desenvolvimento Guiado por Testes) foi "redescoberto" ou popularizado por Kent Beck no final dos anos 90, como uma das práticas essenciais de engenharia da metodologia Extreme Programming (XP).

Antes do TDD, as equipes usavam a abordagem "Test-Last": os programadores escreviam centenas de linhas de código e, no final da semana, tentavam escrever testes para esse código (muitas vezes os testes se tornavam difíceis de encaixar porque o código estava muito acoplado). Com a abordagem "Test-First" do TDD, o teste lidera e molda a escrita do código.

2. O "Segredo" do TDD: É uma ferramenta de Design!

Esta é a principal teoria avançada que diferencia um candidato que entende o conceito de um candidato que apenas decorou a sigla. As bancas mais exigentes costumam explorar isso.

Test-Driven Design (Design Guiado por Testes)

Apesar do nome "Test", o TDD é fundamentalmente uma técnica de Design de Software. Quando um desenvolvedor é obrigado a pensar no teste *antes* do código, ele é forçado a pensar em como a classe será utilizada, nas suas interfaces e nas suas dependências.

Consequência direta no código: O TDD força o surgimento de um código com Alta Coesão e Baixo Acoplamento (mais modular e testável).

3. A Fase da "Refatoração" (Refactoring)

A etapa final do ciclo Red-Green-Refactor é frequentemente negligenciada por equipes inexperientes, mas é vital para a saúde do software.

Refatorar significa melhorar a estrutura interna do código (torná-lo mais legível, remover duplicação, aplicar Design Patterns) sem alterar o seu comportamento visível externamente.

No TDD, a refatoração é feita com uma rede de segurança formidável: como existe um conjunto de testes automatizados prontos, o desenvolvedor tem a coragem de alterar profundamente o código. Se a alteração "quebrar" alguma coisa, os testes ficam imediatamente vermelhos e o programador percebe o erro na hora, podendo retroceder. Sem os testes do TDD, a refatoração seria um "salto no escuro".

4. Ferramentas e Isolamento (Dublês de Teste / Mocks)

O TDD baseia-se em Testes Unitários. A principal regra de um Teste Unitário é que ele deve testar uma única classe ou método em Isolamento.

Se a classe GestorDePedidos precisa gravar algo no Banco de Dados, o teste unitário NÃO PODE conectar-se ao Banco de Dados real (porque deixaria de ser um teste de unidade e passaria a ser de integração, ficando muito lento).

O Uso de "Mocks" e "Stubs"

Para resolver isso, as bibliotecas de TDD usam "Dublês de Teste" (Test Doubles), conhecidos comercialmente como Mocks ou Stubs. O Mock finge ser o Banco de Dados. Ele diz ao código testado: "Pode avançar, finja que eu já guardei o dado com sucesso". Isso mantém o teste ultrarrápido e focado apenas na lógica interna da classe a ser testada.

Ferramentas de Mercado (Frameworks xUnit)

As bancas de concurso podem citar os frameworks xUnit, que são as bibliotecas de programação usadas para executar o ciclo do TDD nas diferentes linguagens:

  • JUnit: O mais famoso, padrão para a linguagem Java.
  • NUnit / xUnit.net: Padrão para o mundo Microsoft (C# e .NET).
  • PyTest / unittest: Utilizado na linguagem Python.
  • Jest / Mocha: Padrões em ecossistemas baseados em JavaScript/TypeScript.
