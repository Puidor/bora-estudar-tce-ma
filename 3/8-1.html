Pipelines de CI/CD

⚡ Direto ao Ponto

  • O Pipeline: É uma "esteira de montagem" virtual. Uma sequência de scripts e ferramentas automatizadas que pegam o código-fonte escrito pelo desenvolvedor e o transportam, passo a passo, até os servidores de produção.
  • O Objetivo: Eliminar o erro humano, reduzir o tempo entre a criação de uma funcionalidade e sua disponibilização ao cliente (Time to Market) e permitir lançamentos frequentes e seguros.

⚔️ O Confronto: CI vs CD vs CD

CAI MUITO!
Sigla / Nome O que faz no Pipeline? Gatilho (Trigger)
CI
Continuous Integration 		Desenvolvedores enviam o código para o repositório central várias vezes ao dia. O servidor compila (Build) e roda Testes Unitários automaticamente. Automático a cada Commit
CD
Continuous Delivery 		O código testado é empacotado e fica "pronto para produção" num ambiente de homologação. Mas a ida para Produção exige aprovação humana. Manual (Aprovação / Clique)
CD
Continuous Deployment 		O nível máximo. Se passar em todos os testes, o código vai para o servidor de Produção (para o cliente final) sem nenhuma intervenção humana. 100% Automático

🚨 Pegadinhas Mortais de Prova

  • A Confusão da Ferramenta: A banca vai jurar que "O CI/CD é um framework de software vendido por fornecedores Cloud que substitui o Agile".
    Falso! CI/CD não é um software, é uma PRÁTICA (ou conjunto de práticas) da cultura DevOps. Ele se apoia em ferramentas (como Jenkins, GitLab CI), mas é essencialmente um método de trabalho.
  • Deployment Anula Delivery? "A Implantação Contínua (Continuous Deployment) substitui a Entrega Contínua (Continuous Delivery)".
    Falso! A Implantação Contínua é uma extensão natural da Entrega Contínua. Para você ter o *Deployment* (automatizar até o final), você *precisa* ter o *Delivery* funcionando (ter os pacotes sempre prontos e testados). Uma engloba a outra.

1. A Gênese: Integração Contínua (CI)

Historicamente (no modelo Cascata), a "fase de integração" era o pior momento do projeto ("Integration Hell"). O Desenvolvedor A trabalhava isolado por 3 meses, o Desenvolvedor B também. Quando juntavam os códigos, nada funcionava.

A Integração Contínua (CI) é a prática ágil (nascida no Extreme Programming - XP) onde os desenvolvedores submetem seus códigos para um repositório central (ex: Git) frequentemente (várias vezes ao dia).

A Mágica da Automação (O Build Server):

Sempre que um desenvolvedor faz um "push" (envia o código), um servidor de CI (como Jenkins ou GitHub Actions) "acorda". Ele faz o download desse código, compila (Build) para ver se há erros de sintaxe, e imediatamente roda uma bateria de Testes Unitários Automatizados. Se um único teste falhar, o servidor "quebra a esteira" (Build Failed) e notifica o desenvolvedor na hora.

2. Entrega Contínua (Continuous Delivery - CD)

A Entrega Contínua (Delivery) começa exatamente onde a Integração Contínua (CI) termina.

O CI garantiu que o código compila e passa nos testes básicos. O *Continuous Delivery* pega esse código, empacota e realiza o deploy dele em ambientes de homologação ou teste de aceite (Staging). Ali rodam os testes mais pesados (Integração, Segurança/SonarQube, Carga, Aceitação do Usuário).

A Palavra-Chave para Provas: Aprovação Humana

No *Continuous Delivery*, a aplicação está 100% testada e tecnicamente pronta para ir para as mãos do cliente (Produção). Mas ela NÃO vai automaticamente.

Existe uma decisão de negócios envolvida. O Gerente de TI ou o Product Owner avalia e dá um clique manual (Aprovação) para liberar a versão para o público. A equipe pode liberar várias vezes por dia, ou segurar para lançar apenas na sexta-feira.

3. Implantação Contínua (Continuous Deployment - CD)

Este é o "Nirvana" do DevOps (praticado por empresas como Amazon, Netflix e Google).

O Continuous Deployment (Implantação Contínua) remove completamente a necessidade de aprovação humana do pipeline. A regra é puramente cibernética: Se o código que o desenvolvedor enviou (commit) passar limpo por 100% da bateria de testes automatizados, ele é publicado nos servidores de Produção reais instantaneamente.

Por que fazer isso?
  • Feedback Imediato: O desenvolvedor faz a feature de manhã e na hora do almoço clientes reais já estão usando e gerando métricas.
  • Risco Menor: Parece contraintuitivo, mas enviar 10 pequenas alterações para produção por dia é muito menos arriscado do que juntar tudo e enviar 300 alterações gigantescas num único fim de semana (onde, se quebrar, ninguém sabe qual das 300 causou o erro).

4. Benefícios e Desafios (Visão da Banca)

As provas não cobram apenas o "O Quê", mas também os "Porquês". Fique atento a estas justificativas teóricas:

  • Vantagem: Redução do MTTR (Mean Time To Recovery)
    Em ambientes de CD rigoroso, se um bug grave entra em produção, a esteira é tão rápida e automatizada que a equipe consegue codificar a correção (patch) e enviá-la para produção em minutos, e não em semanas.
  • Desafio: Cultura de Testes Inabalável
    Você não pode ter Continuous Deployment (automático) se os seus testes automatizados não forem perfeitos. Se o teste não pega o bug, a esteira envia o bug diretamente para o cliente. A maturidade exigida (como o uso forte do TDD abordado no item 3.2 do seu edital) é o maior obstáculo para as empresas.
