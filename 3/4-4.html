Prototipação

⚡ Direto ao Ponto

  • O que é? É a criação de um modelo inicial e simplificado do software para que os usuários possam visualizá-lo e interagir com ele antes do desenvolvimento real.
  • Para que serve? Ajuda a resolver o grande problema da Engenharia de Requisitos: os clientes muitas vezes não sabem o que querem até verem algo pronto. O protótipo tangibiliza a ideia.

⚔️ O Confronto: Descartável vs Evolutivo

CAI MUITO!
Característica Descartável (Throwaway) Evolutivo (Evolutionary)
Objetivo Descobrir ou validar requisitos mal compreendidos rapidamente. Construir um núcleo sólido que será refinado até virar o software final.
Destino do Código É JOGADO NO LIXO. É APROVEITADO.
Qualidade Técnica Péssima. Ignora arquitetura, segurança e performance (foco é só a tela). Alta. Requer arquitetura robusta desde o primeiro dia.
Ferramentas Figma, Balsamiq, Papel e Caneta, HTML cru. Linguagem oficial do projeto (Java, C#, React).

📱 Os Níveis de Fidelidade

✏️

Baixa Fidelidade (Wireframe)

Rascunho rápido. Esboço feito no papel ou quadro branco. Foco apenas na disposição das informações e fluxo de telas. Sem cores, sem imagens reais.

🎨

Média Fidelidade (Mockup)

Feito no computador. Já tem o visual mais próximo do real (cores, botões). Pode permitir cliques simples para navegar entre as telas, mas não tem lógica de negócio por trás.

🚀

Alta Fidelidade

Muito próximo do software final. Responde a interações reais, validação de campos de formulário e muitas vezes tem algum banco de dados falso rodando por trás. Demora muito para ser feito.

🚨 Pegadinhas Mortais de Prova

  • A Ilusão da Substituição: A banca afirma: "A prototipação de alta fidelidade é tão eficaz que substitui completamente a necessidade de se fazer a documentação e elicitação de requisitos".
    Falso! A prototipação é uma técnica de apoio para ajudar a descobrir ou validar requisitos. Ela não substitui as entrevistas, as histórias de usuário ou a engenharia de requisitos.
  • O "Golpe" do Protótipo Descartável: "Como o protótipo descartável já funciona bem visualmente, a equipe ágil deve aproveitar o seu código para acelerar a entrega da Sprint."
    Falso! Esta é a armadilha número um! O código do protótipo descartável (Rapid Prototyping) é feito sem arquitetura, sem segurança e sem testes (o famoso "código espaguete" rápido). Aproveitá-lo em produção é criar uma dívida técnica irreversível. Ele deve ser estritamente jogado fora.

1. A Essência da Prototipação na Engenharia

Na Engenharia de Requisitos (conforme Sommerville e Pressman), descobrir o que o cliente realmente precisa é extremamente difícil. Se você apenas perguntar, o cliente descreverá o sistema com base em uma percepção abstrata. Quando o sistema final é entregue, frequentemente surge a clássica frase: "Ficou igual ao que eu pedi, mas não era bem isso que eu queria".

A prototipação resolve este abismo de comunicação. Ela fornece um artefato tangível com o qual o usuário pode interagir precocemente.

Vantagens Essenciais:
  • Melhora massivamente a comunicação entre Desenvolvedores e Usuários.
  • Ajuda a descobrir requisitos ocultos (coisas que o usuário esqueceu de falar).
  • Previne que a equipe de TI gaste meses programando a coisa errada (redução de retrabalho).

2. Ciclo de Vida: Descartável (Rapid) vs Evolutivo

Em provas de alto nível, você precisa entender o ciclo de vida e os cenários ideais para o uso de cada modelo de protótipo.

Prototipação Descartável (Throwaway Prototyping)

Foca-se nas partes do sistema que são mal compreendidas (alto risco).

  • O Método: Construir rápido, mostrar ao usuário, anotar o feedback, deletar o código e começar o projeto de verdade usando o conhecimento adquirido.
  • O Perigo: O cliente vê a tela funcionando perfeitamente e diz: "Ótimo! Pode colocar em produção amanhã". O desenvolvedor precisa explicar que aquilo é só uma "casca vazia" e terá que ser refeito do zero, causando frustração no cliente inexperiente.

Prototipação Evolutiva (Evolutionary Prototyping)

Foca-se em entregar partes funcionais do sistema desde o dia um.

  • O Método: Começa pelos requisitos que já são bem compreendidos. A equipe cria um núcleo sólido de código. Ao apresentar ao cliente, as melhorias não são feitas num documento, mas sim codificadas em cima do próprio protótipo, que vai crescendo, evoluindo, até se transformar na versão final do sistema.
  • Cenário ideal: Extrema aderência aos modelos ágeis e iterativos-incrementais.

3. Wireframe, Mockup e Protótipo (A Escada de UX/UI)

No mercado atual e nas questões mais modernas sobre Interface de Usuário (UI) e Design Centrado no Usuário, os termos "Wireframe", "Mockup" e "Protótipo" ganharam distinções exatas:

  • 1. Wireframe (A Planta Baixa):
    Baixa fidelidade. Foca apenas na estrutura, disposição do conteúdo e hierarquia da informação (onde vai ficar a imagem, onde vai ficar o texto). Não há preocupação com cores, tipografia ou estilo. Usado logo nos primeiros dias de concepção.
  • 2. Mockup (A Maquete Visual):
    Média/Alta fidelidade visual. É um modelo estático que demonstra o design real da aplicação. Aqui as cores, a fonte oficial, as sombras e as imagens reais são aplicadas. Serve para aprovar a identidade visual com o cliente, mas geralmente não é clicável.
  • 3. Protótipo (A Simulação Interativa):
    Alta fidelidade interativa. É o Mockup ganhando vida. O usuário pode clicar nos botões, os menus se abrem, as transições de tela funcionam. Serve para testar o fluxo de uso (Experiência do Usuário - UX) antes de programar de fato o Backend. (Ferramentas: Figma, Adobe XD).
