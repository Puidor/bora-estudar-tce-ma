Critérios de Aceitação

⚡ O que são?

  • São as condições específicas que um produto de software (ou História de Usuário) deve cumprir para ser aceito por um usuário, cliente ou sistema.
  • A Fronteira: Eles definem os limites do requisito. Respondem à pergunta: "Como sabemos quando a equipe terminou de desenvolver esta funcionalidade?".
  • Formato Binário: Um critério de aceitação não pode ser ambíguo. Ele tem que ser Passou ou Falhou (Pass/Fail).

⚔️ O Confronto: Acceptance Criteria vs Definition of Done (DoD)

CAI MUITO!
Característica Critérios de Aceitação (AC) Definition of Done (DoD)

📝 Formatos de Escrita

1. Formato de Regras (Checklist)

Mais simples. Uma lista de pontos (bullet points) que devem ser verificados.

☑ O usuário não pode enviar o campo E-mail vazio.

☑ O sistema deve enviar um e-mail de confirmação.

☑ Se o e-mail for inválido, exibir mensagem vermelha.

2. Formato de Cenário (Gherkin / BDD)

Padrão da indústria Ágil moderna. Facilita a automação de testes.

Dado que estou na página de login

Quando preencho um e-mail sem '@'

Então o sistema exibe "E-mail inválido"

🚨 Pegadinhas Mortais de Prova

  • A Responsabilidade Exclusiva: A banca afirma: "A escrita dos critérios de aceitação é uma responsabilidade exclusiva do Product Owner (PO), não cabendo à equipe de desenvolvimento interferir".
    Falso! Embora o PO seja o principal responsável por garantir que eles existam e representem o negócio, os critérios devem ser debatidos e criados colaborativamente (lembre-se da regra dos Três Amigos: PO, Dev, QA) durante o Refinamento do Backlog.
  • Detalhes Técnicos: "Os critérios de aceitação devem incluir o nome da tabela do banco de dados e a linguagem de programação a ser utilizada".
    Falso! Os critérios devem focar na intenção do usuário e no comportamento do sistema (O Quê). Não devem ditar o design técnico interno (O Como).

1. A Importância na Engenharia de Requisitos

Na Engenharia de Software Tradicional, a validação do software era feita no final do ciclo de vida, comparando o produto com um documento de requisitos de 500 páginas. Nas metodologias ágeis (Scrum, XP), os requisitos são ágeis e conversacionais (Histórias de Usuário).

No entanto, uma História de Usuário (ex: "Como cliente, quero pagar com cartão") é propositalmente vaga. É apenas uma "promessa de conversa". Como o programador sabe se deve aceitar Visa, Mastercard ou American Express? É aqui que entram os Critérios de Aceitação.

Benefícios Primários:
  • Remover Ambiguidade: Garantem que o cliente e o desenvolvedor têm exatamente a mesma visão sobre a funcionalidade.
  • Definir Limites: Evitam o Scope Creep (aumento de escopo descontrolado). O programador não fará mais do que o estritamente definido nos critérios.
  • Base para Testes: Formam o alicerce para a equipe de QA (Quality Assurance) criar os seus scripts/roteiros de testes manuais ou automatizados (ATDD).

2. Definition of Done (DoD) vs Acceptance Criteria (AC)

Este é o tópico teórico mais explorado nos concursos de TI. É essencial entender a natureza hierárquica e a abrangência de cada um.

Definition of Done (DoD) - A Qualidade Global

A "Definição de Pronto" é um acordo compartilhado por toda a equipe Scrum sobre o que significa um trabalho estar finalizado com qualidade. Ela é universal para a Sprint. Quer a equipe esteja desenvolvendo a funcionalidade de Login ou a funcionalidade de Carrinho de Compras, as regras do DoD aplicam-se a ambas.

  • Ex: "Todo o código tem que ter testes unitários com 80% de cobertura."
  • Ex: "O código tem que passar no pipeline de CI/CD sem erros."
  • Ex: "A documentação da API tem que ser atualizada no Swagger."

Acceptance Criteria (AC) - O Escopo Local

Os Critérios de Aceitação são os requisitos de negócio inerentes a uma única História de Usuário específica. Eles não se aplicam a outras histórias.

  • Para a História de Login: "A senha deve ter letras maiúsculas e minúsculas."
  • Para a História de Carrinho: "O carrinho deve esvaziar automaticamente após 24 horas de inatividade."

"Para uma História ser considerada DONE, ela tem que cumprir todos os seus Critérios de Aceitação E também cumprir a Definition of Done da equipe."

3. Características de um Bom Critério de Aceitação

As bancas de concurso podem avaliar se um determinado cenário apresentado na prova seria considerado um bom critério de aceitação. Lembre-se destas regras:

  • Não ambíguo e Testável: Um critério como "O sistema tem que ser rápido" é péssimo (o que é rápido?). Um bom critério diz: "A pesquisa de produtos tem que retornar o resultado em menos de 2 segundos". Tem que poder ser testado como Verdadeiro ou Falso.
  • Perspectiva do Usuário: Deve ser focado no resultado esperado por quem usa o software, e não em como o código foi implementado.
  • Cobre Casos Positivos e Negativos: Não deve focar apenas no "Caminho Feliz". Deve incluir cenários de erro (Ex: "Se o usuário inserir um cartão de crédito expirado, o sistema exibe a mensagem de Erro X").
  • Escrito Antes do Desenvolvimento: Tal como nas práticas TDD e ATDD, os critérios de aceitação devem estar claramente definidos antes do desenvolvedor escrever a primeira linha de código (geralmente definidos na reunião de Planejamento da Sprint ou no Refinamento).
