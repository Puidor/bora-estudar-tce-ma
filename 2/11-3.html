Otimização de Performance (Tuning)

🚀 Os Três Pilares do Tuning

Como resolver o problema de consultas que demoram horas para rodar.

📇

1. Indexação (Índices)

Funciona como o índice de um livro. Evita que o banco tenha que ler a tabela inteira linha por linha (Full Table Scan). Aumenta a velocidade de Leitura, mas penaliza a Escrita.

🔪

2. Particionamento

Dividir para Conquistar. Quebra uma tabela gigante de 1 bilhão de linhas em "pedaços" menores (ex: por Ano ou por Região), espalhando a carga de I/O em discos diferentes.

👁️

3. Visão Materializada

Pré-computação. Executa consultas super complexas (com vários JOINs e agregações) de madrugada e salva o resultado fisicamente no disco para consulta instantânea de dia.

🚨 Pegadinhas de Performance (FGV / CEBRASPE)

  • A Ilusão do Índice: A banca afirma: "Para otimizar o banco de dados, a melhor prática é criar índices para TODAS as colunas de todas as tabelas".
    Falso! Índices aceleram a Leitura (SELECT), mas penalizam gravemente a Escrita (INSERT, UPDATE, DELETE). Cada vez que uma linha é inserida, todos os índices da tabela precisam ser reorganizados na memória e no disco.
  • View Simples vs Materializada: "Uma View (Visão) melhora a performance de consultas complexas".
    Atenção! Uma View simples (lógica) não guarda dados, é apenas uma "máscara" do comando SELECT. Ela roda em tempo real e não melhora a performance. Apenas a View MATERIALIZADA melhora a performance, porque ela roda o SELECT e guarda o resultado fisicamente como uma tabela real no disco.

1. Indexação (O Fim do Full Table Scan)

Quando executamos um SELECT * FROM Clientes WHERE Nome = 'João' numa tabela sem índices, o banco de dados é forçado a ler bloco por bloco do disco rígido até ao fim da tabela. Isso chama-se Full Table Scan (FTS) e é o maior vilão da performance de um SGBD.

Para resolver isso, criamos Índices. Um índice é uma estrutura de dados extra que armazena os valores da coluna indexada ordenados, contendo ponteiros físicos diretos para o local exato onde o resto da linha está no disco.

O Rei: B-Tree (Árvore B)

É o tipo de índice padrão em 99% dos bancos relacionais (Oracle, Postgres, SQL Server). Estruturado como uma árvore balanceada. Ele é perfeito para pesquisas exatas (=) e pesquisas por intervalos (BETWEEN, >, <).

O Especialista: Bitmap

Usa matrizes de bits (0 e 1) em vez de ponteiros de árvore. É imbatível em Data Warehouses quando queremos fazer consultas complexas cruzando colunas de baixa cardinalidade (ex: WHERE Sexo = 'F' AND UF = 'SP' AND Estado_Civil = 'Casado'). A operação lógica AND/OR com zeros e uns no processador é quase instantânea.

Perigo OLTP: Se tentar usar índice Bitmap num sistema de caixa de supermercado (muitos INSERTS), o banco trava. O Bitmap exige bloqueio (Lock) de páginas inteiras de dados durante uma atualização.

2. Particionamento e Sharding (Dividir para Conquistar)

Mesmo com índices, uma tabela com 5 bilhões de linhas torna-se instável para gerir (para fazer backup, por exemplo). A solução estrutural é o Particionamento. O SGBD quebra a tabela gigante em pedaços menores internamente, mas para o utilizador/programador, ela continua a parecer uma tabela única.

Particionamento Vertical

Divide a tabela por colunas. Se a "Tabela Clientes" tem 50 colunas, mas 90% das consultas só pedem Nome, CPF e E-mail, podemos separar as outras 47 colunas (Foto, Biografia, etc) numa partição física diferente. A consulta fica mais "fina" e rápida de ser carregada do disco para a memória.

Particionamento Horizontal

Divide a tabela por linhas com base numa regra (chave de partição). Exemplo (Particionamento por Range/Intervalo): Linhas de 2021 ficam na Partição 1, linhas de 2022 na Partição 2. Se a query for SELECT * WHERE ANO = 2022, o banco ignora completamente o disco da Partição 1 (Partition Pruning), poupando I/O.

Sharding (Fragmentação):
É a evolução do particionamento horizontal para arquiteturas distribuídas (muito comum em Bancos NoSQL). Em vez de guardar as partições horizontais num único super servidor, o Sharding espalha as partições (Shards) por diferentes Servidores Físicos distintos espalhados pelo mundo.

3. Query Tuning e o Otimizador

Um índice só ajuda se o Otimizador de Consultas (Query Optimizer) do SGBD decidir usá-lo. O Otimizador é o "cérebro" do banco que analisa a sua consulta SQL e cria um Plano de Execução (Execution Plan) baseado nas estatísticas da tabela.

Erros de Código que inutilizam Índices (Cai em Prova!)

Muitas vezes o índice existe, mas o programador escreve um SQL mau que cega o banco, forçando um Full Table Scan. Evite:

  • Funções na Cláusula WHERE: WHERE UPPER(Nome) = 'JOÃO'. Como você aplicou a função UPPER(), o banco não consegue procurar no índice direto, ele precisa converter linha por linha antes de comparar.
    Solução: Criar um Índice Baseado em Função (Function-based Index).
  • Curinga (Wildcard) no início do LIKE: WHERE Nome LIKE '%Silva'. O índice B-Tree é organizado alfabeticamente pela primeira letra. Como você colocou o % no começo, ele não sabe por qual letra começar e varre a tabela toda.
  • Ocultamento por Conversão de Tipo (Casting): A coluna CPF é VARCHAR, mas você faz WHERE CPF = 123456 (número). O banco fará conversão implícita de todos os CPFs para número antes de comparar, ignorando o índice.

4. Visões Materializadas e Caching

A Visão Materializada (Materialized View)

Em Business Intelligence (BI), é comum ter dashboards que calculam o "Total de Vendas por Estado do último ano", varrendo dezenas de milhões de linhas com funções SUM() e vários JOINs. Se cada diretor atualizar a tela, o banco pode "cair".

A Visão Materializada resolve isso. Ela executa essa consulta pesada uma vez (geralmente agendada de madrugada via Job) e salva a tabela resultante fisicamente no disco. Quando o diretor abre o dashboard, o banco lê uma tabela pronta de 27 linhas (os Estados), o que demora 1 milissegundo. O desafio é o Refresh (mantê-la atualizada sem onerar o banco).

Caching em Memória (Redis / Memcached)

A arquitetura de otimização em larga escala moderna (usada por Google, Netflix, X) adiciona uma camada antes do Banco de Dados Relacional.

Se a aplicação pede os detalhes do produto "Smartphone XPTO", o sistema não vai ao banco de dados. Ele pergunta primeiro ao Cache em Memória (ex: Redis). Se o Redis tiver essa informação na RAM (Cache Hit), devolve em microssegundos. Só vai ao SGBD se der Cache Miss.
