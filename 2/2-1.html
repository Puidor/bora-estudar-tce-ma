Banco de Dados: Conceitos Básicos

⚡ Direto ao Ponto: BD vs SGBD

Banco de Dados (BD)

É a coleção de dados em si, logicamente coerente, armazenada fisicamente nos discos (os arquivos de dados).

SGBD

É o Software (Sistema Gerenciador). Ele intermedeia o usuário/aplicação e o banco de dados.
Ex: Oracle, MySQL, PostgreSQL.

🏗️ Arquitetura de 3 Esquemas (ANSI/SPARC)

1. Nível EXTERNO (Visões)

Como o usuário final enxerga os dados. Esconde o resto do banco de dados do usuário por questões de segurança e simplicidade.

2. Nível LÓGICO (Conceitual)

Descreve QUAIS dados são armazenados e as RELAÇÕES entre eles (Tabelas, Colunas, Tipos de dados). Oculta os detalhes físicos de armazenamento.

3. Nível FÍSICO (Interno)

Descreve COMO os dados são realmente armazenados no disco (Caminhos de arquivos, blocos, índices, compressão).

O objetivo desta arquitetura é garantir a Independência de Dados.

⚖️ O Confronto: Esquema vs Instância

Caem muito as palavras "Intensão" e "Extensão".

Conceito Esquema (Schema) Instância (Estado)
Sinônimo em Prova Intensão / Metadados Extensão / Dados Reais
O que é? A descrição do banco (Tabela 'Cliente' tem as colunas 'Nome' e 'CPF'). Os dados armazenados num momento específico (O cliente 'João', CPF '123').
Frequência de Mudança Raramente sofre alterações (Estático). Muda a cada segundo (Dinâmico).

🚨 Pegadinhas de Prova

  • Terminologia SQL vs BD Clássico: A banca pode dizer que o SGBD fornece o software DML para definição da estrutura do banco.
    Falso! Quem define estrutura (Esquema) é a DDL (Linguagem de Definição de Dados). A DML (Manipulação) atua nas Instâncias (Consultas, Inserções).
  • A Independência Física vs Lógica: "Ao adicionar um novo índice no disco rígido para melhorar a velocidade, o desenvolvedor está usufruindo da independência lógica de dados."
    Falso! Mudar a estrutura de armazenamento em disco (adicionar um índice, trocar de HD para SSD) sem alterar o Nível Lógico (Tabelas) é o conceito de Independência FÍSICA de dados.

1. Sistema de Bancos de Dados

Para entender as questões teóricas, precisamos ser preciosistas com o vocabulário criado por autores como Navathe e Elmasri.

  • Banco de Dados (BD): É uma coleção de dados logicamente coerente que possui um significado implícito. Uma planilha telefônica salva num bloco de notas é, conceitualmente, um banco de dados. Na computação, são os arquivos onde os bits estão guardados.
  • SGBD (Sistema Gerenciador de Banco de Dados): É um conjunto de programas (software) que permite aos usuários criar, manter e controlar o acesso ao banco de dados. Ex: MySQL, Oracle, SQL Server.
  • Sistema de Banco de Dados: É o pacote completo. A união do Banco de Dados (os dados em si) + o SGBD (o software) + a Aplicação (o programa do usuário).
Metadados (Catálogo do Sistema): O SGBD não guarda apenas os dados dos usuários; ele guarda os dados sobre os dados. O catálogo do sistema armazena a descrição da estrutura do banco, permitindo que o SGBD seja um software de propósito geral (ele sabe como ler os dados porque lê o catálogo primeiro). Isso é chamado de natureza autodescritiva do SGBD.

2. Arquitetura ANSI/SPARC de 3 Esquemas

Historicamente, antes dos SGBDs modernos, se a empresa trocasse de disco rígido, o programador tinha que reescrever todo o código do sistema (pois o programa lia diretamente os setores físicos do disco).

A arquitetura de três esquemas foi proposta para separar as aplicações dos usuários do banco de dados físico.

Os Três Níveis:

  • 1. Nível Interno (Físico): Tem um esquema interno que descreve a estrutura física de armazenamento. Utiliza um modelo de dados físico e detalha os caminhos de dados e os índices (B-Tree, Hashing).
  • 2. Nível Lógico (Conceitual): Tem um esquema conceitual que descreve a estrutura de todo o banco de dados para uma comunidade de usuários. Oculta detalhes do armazenamento físico. Foca nas entidades, relacionamentos e restrições.
  • 3. Nível Externo (Visões): Inclui vários esquemas externos (ou Views do usuário). Cada esquema externo descreve a parte do banco de dados que um grupo de usuários específico tem interesse e oculta o resto do banco de dados daquele grupo. (Ex: O RH só vê a tabela de salários, o Vendedor só vê a tabela de produtos).

A Independência de Dados

A capacidade de modificar um esquema em um nível sem precisar alterar o esquema do nível superior.

  • Independência Lógica de Dados: Mudar o Esquema Conceitual (ex: adicionar uma coluna nova na tabela de Clientes) sem precisar alterar os Esquemas Externos ou os programas de aplicação.
  • Independência Física de Dados: Mudar o Esquema Interno (ex: criar um índice novo para acelerar a busca, mudar de HD para SSD) sem precisar alterar o Esquema Conceitual. É a mais fácil de ser alcançada.

3. Por que usar um SGBD? (Vantagens)

Em contrapartida ao antigo sistema de "arquivos de texto", o SGBD oferece funcionalidades obrigatórias em ambientes corporativos:

  • Controle de Redundância: Evita que a mesma informação (ex: telefone do cliente) seja salva em 10 planilhas diferentes. A redundância não é eliminada 100%, mas passa a ser controlada.
  • Restrição de Acesso: Fornece um subsistema de segurança (Contas de usuários, senhas, privilégios).
  • Controle de Concorrência: Permite que dezenas de pessoas acessem e modifiquem a mesma tabela de banco simultaneamente sem corromper os dados, usando bloqueios (Locks) transacionais.
  • Fornecer Backup e Recuperação: O SGBD garante que, no caso de um crash de hardware no meio de uma transação de transferência de dinheiro, o banco recupere o seu estado consistente anterior (Propriedades ACID, exploradas no item 2.2).
  • Impor Restrições de Integridade: Regras no banco que impedem que o usuário salve um CPF com 5 dígitos, ou associe uma venda a um "ID de Cliente" que não existe.

Atores do Cenário (Em Provas)

Administrador de Banco de Dados (DBA): O profissional responsável por autorizar acessos ao BD, coordenar a capacidade dos servidores (Tuning/Desempenho), fazer backups e monitorar a disponibilidade do SGBD. (Atenção: O DBA gerencia o ambiente do SGBD, ele não é o desenvolvedor da aplicação que desenha a lógica do negócio, nem necessariamente o Analista de Dados que tira insights).
