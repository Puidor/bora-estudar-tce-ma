Modelagem Dimensional

⚡ Direto ao Ponto

  • O que é? É uma técnica de design lógico de banco de dados (criada por Ralph Kimball) projetada especificamente para Data Warehouses (OLAP).
  • Objetivo: Otimizar a leitura rápida e agregação de dados, sacrificando a normalização rígida para reduzir a complexidade e a quantidade de JOINs.

🧩 A Anatomia: Fatos vs Dimensões

🔢

Tabela de Fatos

Armazena as métricas (valores quantitativos) do negócio. Responde "o quanto". Costuma ser gigantesca (milhões de linhas).

  • Contém chaves estrangeiras (FKs) para as dimensões.
  • Exemplos de colunas: Valor_Venda, Qtde_Vendida, Desconto.
🏷️

Tabelas de Dimensão

Armazena o contexto (atributos descritivos) dos fatos. Responde "quem, onde, quando e o quê".

  • Geralmente possuem poucas linhas, mas muitas colunas "largas" (textos).
  • Exemplos: Dim_Cliente (Nome, Idade), Dim_Tempo, Dim_Loja.

⚖️ O Confronto: Star Schema vs Snowflake Schema

Característica Esquema Estrela (Star) Esquema Floco de Neve (Snowflake)
Nível de Normalização Desnormalizado Normalizado
Velocidade de Consulta (Query) Muito Rápido (Poucos JOINs) Mais lento (Muitos JOINs para remontar a tabela)
Redundância de Dados Alta (O estado se repete a cada cliente na mesma tabela) Baixa (O estado vira uma tabela separada)

🚨 Pegadinhas de Prova (Chaves e Tipos)

  • A Chave Surrogate (Artificial): A banca vai dizer que a Tabela de Dimensão deve usar o CPF do cliente (chave natural) como chave primária (PK).
    Falso! A recomendação absoluta em Kimball é NÃO usar chaves de negócio (Natural Keys) como PK no Data Warehouse. Deve-se criar uma Surrogate Key (uma chave artificial, um ID inteiro gerado pelo banco) para isolar o DW de mudanças no sistema original.
  • Medidas Semi-Aditivas: Fatos podem ser Aditivos (podem ser somados em qualquer dimensão, ex: Valor_Venda), Não-aditivos (ex: Margem de Lucro em % - não faz sentido somar 10% com 15%) e Semi-aditivos. O exemplo clássico de prova do semi-aditivo é o Saldo Bancário (você pode somar o saldo de todos os clientes numa agência, mas não pode somar o saldo do cliente ao longo dos 30 dias do mês, senão o valor não faz sentido).

1. Fundamentos: OLTP vs OLAP

O modelo entidade-relacionamento tradicional (3FN) é perfeito para sistemas operacionais OLTP (Online Transaction Processing), como o sistema de caixa de um supermercado, pois evita redundâncias e anomalias de atualização em operações rápidas.

No entanto, para sistemas de suporte à decisão OLAP (Online Analytical Processing), o objetivo não é inserir uma venda rápida, mas sim consultar o histórico de 10 anos de vendas agrupado por região e produto. Se o Data Warehouse usasse a 3FN, essa consulta exigiria 15 ou 20 JOINs simultâneos, travando o banco de dados. A solução de Ralph Kimball foi a Modelagem Dimensional: sacrificar a normalização em favor do desempenho absurdo de leitura.

2. Evolução dos Esquemas (Schemas)

Esquema Estrela (Star Schema)

A abordagem clássica. O diagrama visual lembra uma estrela: a enorme Tabela de Fatos fica no centro, conectada por uma relação 1:N diretamente às Tabelas de Dimensão nas pontas.

As tabelas de dimensão no esquema estrela são fortemente desnormalizadas. Exemplo: A tabela Dim_Cliente terá colunas para Nome, Cidade, Estado e País, repetindo a palavra "Brasil" para milhares de clientes, ocupando mais espaço em disco.

Esquema Floco de Neve (Snowflake Schema)

É um modelo Estrela onde as Dimensões foram normalizadas. A tabela Dim_Cliente agora tem apenas um ID_Cidade, que aponta para uma tabela Dim_Cidade, que tem um ID_Estado, que aponta para Dim_Estado.

A estrutura ramifica-se parecendo um floco de neve. Poupa espaço em disco (não repete "Brasil" mil vezes), mas o desempenho cai porque a query precisa de múltiplos JOINs nas ramificações.

Esquema Constelação (Galaxy Schema / Fact Constellation)

Usado em cenários mais complexos. Ocorre quando temos duas ou mais Tabelas de Fatos que compartilham as mesmas Dimensões. Exemplo: A Tabela Fato_Vendas e a Tabela Fato_Estoque ambas se conectam à tabela Dim_Produto (Dimensão Conformada).

3. SCD: Slowly Changing Dimensions (Cai Sempre!)

O que acontece no Data Warehouse quando um atributo da dimensão muda na vida real? (Exemplo: O Cliente João casou-se, ou mudou de endereço de SP para o RJ). Como rastreamos esse histórico? Existem várias técnicas, mas 3 tipos despencam em concursos:

  • Tipo 1 (Sobrescrita / Overwrite): O valor antigo é apagado e o novo é gravado por cima.
    Efeito: Não guarda nenhum histórico. É fácil de implementar, mas todas as análises passadas do João agora aparecerão como se ele sempre morasse no RJ.
  • Tipo 2 (Nova Linha / New Row): A Mais Cobrada
    Cria-se uma nova linha (novo registro) para o João, com uma nova Surrogate Key e o endereço do RJ. A linha antiga (com o endereço de SP) permanece intocada.
    Efeito: Preserva o histórico perfeito e ilimitado. Normalmente usam-se colunas de "Data_Inicio", "Data_Fim" ou "Flag_Ativo" para saber qual é a linha atual.
  • Tipo 3 (Nova Coluna / New Attribute): Adiciona-se uma nova coluna na tabela (ex: Endereco_Anterior). O endereço de SP vai para essa coluna, e a coluna principal Endereco_Atual recebe o RJ.
    Efeito: Guarda histórico, mas é muito limitado (só consegue rastrear a mudança anterior imediata).

4. Fatos Factless (Tabela de Fatos Sem Fatos)

A teoria diz que a Tabela de Fatos guarda medidas numéricas. Mas existe uma "exceção" clássica de prova: a Factless Fact Table.

É uma tabela de fatos que não possui NENHUMA medida quantitativa (numérica); ela contém apenas as Chaves Estrangeiras apontando para as Dimensões.

Para que serve? Serve para registrar eventos ou cobertura.
Exemplo 1: Um sistema de controle de presença universitária. A tabela de fato registra apenas ID_Data, ID_Aluno e ID_Disciplina. Não há valor financeiro ou métrica, o mero cruzamento dessas chaves já atesta o "Fato" de que o aluno estava presente naquele dia.
