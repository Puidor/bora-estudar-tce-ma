Árvore B+ (B+ Tree)

Estrutura balanceada. Os dados (ponteiros) ficam APENAS nas folhas, que são ligadas em lista.

Ideal para: Buscas por Intervalo (BETWEEN)
Hash

Usa uma função matemática para calcular o endereço exato do registro na memória (bucket).

Ideal para: Correspondência Exata (ID = 5)
Bitmap

Mapeia valores usando sequências de 0 e 1. Supercompacto e excelente para cruzamentos (AND/OR).

Ideal para: Baixa Cardinalidade (Sexo: M/F)

📊 Orientado a Linha vs Orientado a Coluna

O divisor de águas entre Bancos Tradicionais e Bancos Analíticos de Big Data.

Característica Orientado a Linhas (Row-Store) Orientado a Colunas (Column-Store)
Como grava no disco? Grava todos os atributos de um registro (uma linha) juntos. Grava todos os valores de uma mesma coluna juntos.
Melhor cenário de uso OLTP (Transações Diárias) OLAP (Data Warehouse / Analytics)
Vantagem Principal Inserções (INSERTs) rápidas de novos registros. Leitura hiper-rápida para agregações (SUM, AVG) e alta taxa de compressão de dados.
Exemplos MySQL, PostgreSQL, Oracle Amazon Redshift, Apache Cassandra, HBase

🚨 Pegadinhas de Prova Clássicas

  • Árvore B vs. Árvore B+: A banca vai dizer que na Árvore B+ os dados (ou ponteiros de dados) são armazenados em todos os nós da árvore, facilitando a busca.
    Falso! Quem armazena dados em todos os nós (raiz e intermediários) é a Árvore B normal. A grande sacada da Árvore B+ (usada no MySQL, Oracle) é guardar os dados EXCLUSIVAMENTE nos nós folha, interligando esses nós como uma lista encadeada para buscas sequenciais rápidas.
  • Índice Hash e Buscas em Intervalo: "Para buscar todos os funcionários com salário entre 5.000 e 10.000, o índice Hash é o mais eficiente."
    Falso! O índice Hash destroi a ordenação lógica dos dados. Ele é o mais rápido (O(1)) para buscar valores EXATOS (ex: Salário = 5.000). Para intervalos (BETWEEN, >, <), a Árvore B+ é infinitamente superior.

1. Organização de Arquivos Físicos

Antes de falarmos de índices, precisamos entender como o SGBD joga os registros brutos dentro do disco rígido. Existem três formas principais de organização de arquivos:

  • A. Arquivo Heap (Pilha / Não-ordenado):
    Os registros são gravados no disco na exata ordem em que chegam, sem nenhuma ordenação lógica.
    Vantagem: Inserção (INSERT) muito rápida. Basta gravar no final do arquivo.
    Desvantagem: Busca terrível. Para achar o João, você precisa ler o disco do começo ao fim (Busca Linear).
  • B. Arquivo Sequencial (Ordenado):
    Os registros são mantidos fisicamente ordenados no disco com base em um campo chave (ex: ordenados pelo ID do cliente).
    Vantagem: Permite fazer busca binária. Excelente para ler intervalos (ex: "Clientes com ID de 100 a 200").
    Desvantagem: A inserção é pesada, pois exige realocar registros para manter a ordem física no disco.
  • C. Arquivo Hash (Espalhamento):
    A posição física do registro é decidida por uma função matemática (Função Hash) aplicada sobre a chave.
    Vantagem: Recuperação exata quase instantânea.
    Desvantagem: Inviável para buscar faixas de valores e sofre com "colisões" se a função Hash for ruim.

2. Estruturas de Índices

Para resolver o problema da lentidão na busca em arquivos Heap, usamos Índices. Funcionam exatamente como o sumário de um livro: em vez de ler o livro todo, você olha no sumário e vai direto para a página.

Tipos Básicos de Índices:

  • Índice Primário: Criado sobre um arquivo que já está ordenado sequencialmente (Campo Chave). O arquivo só pode ter UM índice primário (já que não pode estar ordenado fisicamente de duas formas diferentes).
  • Índice Secundário: Criado sobre campos onde o arquivo físico não está ordenado. O SGBD cria uma estrutura auxiliar separada (como a Árvore B+) que guarda os ponteiros para os registros espalhados no disco. Você pode ter vários numa mesma tabela.

A Estrutura Rei: Árvore B+ (B+ Tree)

A Árvore B+ é a estrutura de dados mais importante para banco de dados relacional. Ela é uma árvore de pesquisa balanceada.

As Leis da Árvore B+ para Provas:
  • Nós internos apenas guiam: Os nós que ficam na raiz e no meio da árvore contêm apenas chaves de roteamento. Não contêm os dados reais.
  • Folhas contêm o ouro: Os dados (ou os ponteiros exatos para o disco) ficam exclusivamente no nível mais baixo (nós folha).
  • Encadeamento lateral: Os nós folha são ligados uns aos outros horizontalmente (como uma lista duplamente encadeada). Isso faz com que, após achar o início de um intervalo, você possa "escorregar" para o lado lendo os próximos registros em altíssima velocidade.

Índices Bitmap

Usado em Engenharia de Dados (Data Warehouses). Ideal para campos de baixa cardinalidade (poucos valores diferentes, como "Gênero" ou "Estado Civil"). Ele cria um vetor de bits (0 ou 1) para cada valor possível. É muito rápido para fazer operações lógicas complexas (AND, OR, NOT) em bancos de dados gigantes.

3. Armazenamento Físico: Linha vs Coluna

A revolução do Big Data e Data Warehousing mudou a forma como gravamos os blocos no disco.

Armazenamento Orientado a Linhas (Row-Store)

Padrão dos bancos relacionais (PostgreSQL, Oracle). Todos os atributos de um registro (ex: ID, Nome, Salário, CPF do funcionário João) são guardados sequencialmente no mesmo bloco de disco.
Vantagem: Excelente para inserir um funcionário novo (OLTP), pois o disco grava tudo de uma vez.
Desvantagem: Péssimo para análises (OLAP). Se eu quiser saber "A média de Salários", o disco terá que ler os blocos inteiros, carregando ID, Nome e CPF à toa para a memória.

Armazenamento Orientado a Colunas (Column-Store)

Padrão em Engenharia de Dados (Amazon Redshift, Apache Cassandra, Apache Parquet). Os valores da mesma coluna de diferentes registros (ex: apenas a coluna de Salários) são guardados de forma contígua no mesmo bloco de disco.
Vantagem de Leitura: A operação "Soma dos Salários" puxa do disco *apenas* os blocos com os salários. É absurdamente rápido.
Vantagem de Compressão: Como uma coluna geralmente tem dados do mesmo tipo (só números, ou muito repetidos), a compressão (Zip) dos blocos é altíssima. Você reduz 1 Terabyte para 100 Gigabytes facilmente.
