⚖️ O Confronto: ETL vs ELT

É o tema mais cobrado em Integração de Dados.

Característica ETL (Extract, Transform, Load) ELT (Extract, Load, Transform)
Onde ocorre a Transformação? Num Servidor Intermediário (Motor de ETL). Dentro do Destino (BD/Data Warehouse).
Dado que chega ao Destino Apenas o dado já tratado, limpo e estruturado. O dado bruto chega primeiro, sendo tratado depois.
Volume de Dados (Big Data) Limitado pela capacidade do servidor intermediário. Altamente escalável (Usa o poder computacional da Nuvem).
Destino Típico Data Warehouses Tradicionais (On-premise). Data Lakes e Cloud Data Warehouses (Snowflake, BigQuery).

🚨 Pegadinhas de Prova

  • Batch vs Streaming: A banca vai dizer que "A ingestão em tempo real (Streaming) substituiu completamente a necessidade de ingestão Batch nas empresas modernas".
    Falso! A ingestão Batch (em lotes) ainda é o padrão ouro para processamentos pesados noturnos (fechamento de folha de pagamento, consolidação de vendas), pois é mais barata e consome menos recursos de rede durante o dia.
  • Data Lake vs Data Warehouse: A banca tentará te convencer que um Data Lake armazena apenas dados estruturados em SQL.
    Falso! O Data Lake é um grande reservatório de dados brutos e não estruturados (vídeos, logs, json). Quem armazena dados altamente estruturados e limpos para BI é o Data Warehouse.

1. Coleta e Ingestão de Dados

A etapa de coleta (ou ingestão) é o processo de capturar dados de diversas fontes externas (Bancos de dados de ERPs, APIs web, logs de servidores, sensores IoT) e trazê-los para o ambiente de engenharia de dados.

Padrões de Ingestão:

  • Processamento em Lote (Batch): Os dados são coletados periodicamente em grandes pacotes (ex: toda meia-noite extrai-se as vendas do dia). É ideal para volumes massivos onde o atraso (latência) de horas não é um problema. Ferramentas: Apache Airflow, AWS Glue.
  • Processamento em Tempo Real (Streaming): Os dados são capturados e processados no exato momento em que são gerados (milissegundos). Essencial para detecção de fraudes em cartões de crédito e monitoramento de IoT. Ferramentas: Apache Kafka, Apache Flink, AWS Kinesis.

2. Tratamento de Dados (Data Quality)

Um velho ditado da computação diz: "Garbage In, Garbage Out" (Se entra lixo, sai lixo). O tratamento visa melhorar a qualidade dos dados antes que cheguem aos diretores que tomam as decisões. Envolve:

  • Limpeza (Cleansing): Remoção de dados duplicados, correção de erros de digitação e tratamento de valores nulos (Null).
  • Normalização: Padronização de formatos. (Ex: Mudar datas de MM/DD/YYYY para YYYY-MM-DD).
  • Enriquecimento: Cruzar o dado com bases externas. (Ex: Pegar o CEP do cliente e enriquecer a tabela com a Latitude e Longitude).
  • Anonimização/Mascaramento: Ocultar dados sensíveis (PII) para cumprir regras da LGPD, como trocar CPFs por hashes irreconhecíveis.

3. Integração (A Guerra ETL vs ELT)

A Integração consolida dados de várias origens num único destino coerente.

ETL (Extrair, Transformar, Carregar)

Abordagem clássica dos anos 90/2000. O dado sai do sistema original, entra num "motor de transformação" (Servidor ETL) onde sofre todo o tratamento pesado, e só depois de limpo é carregado no Data Warehouse (Destino).
Problema: O servidor ETL torna-se um gargalo se o volume for da ordem de Petabytes.

ELT (Extrair, Carregar, Transformar)

Abordagem moderna de Big Data. Você extrai o dado e o despeja bruto (Load) imediatamente no destino (ex: um Data Lake na nuvem). Depois, você usa o imenso poder de processamento do próprio destino para transformar os dados através de scripts SQL ou Spark.
Vantagem Ouro: Reduz o tempo de ingestão e preserva os dados em seu estado original/bruto para uso futuro da equipe de Data Science.

4. Armazenamento e Recuperação

Paradigmas de Armazenamento Centralizado

  • Data Warehouse (DW): Repositório de dados estruturados, limpos e otimizados para consultas rápidas de Business Intelligence (BI). Usa Schema-on-write.
  • Data Lake: O "Lago". Armazena volumes massivos de dados brutos (estruturados, semiestruturados e não estruturados - imagens, vídeos, logs). Usa Schema-on-read. Muito usado por Cientistas de Dados.
  • Data Lakehouse: O conceito mais moderno! Junta o melhor dos dois mundos. O armazenamento barato e flexível do Data Lake com a confiabilidade e suporte a transações ACID do Data Warehouse.

Recuperação (Consumo)

A recuperação é a "ponta da lança". Após o pipeline terminar, os dados precisam ser entregues aos usuários finais. Isso é feito através de:

  • Dashboards (Painéis de BI): Ferramentas como PowerBI ou Tableau executam consultas analíticas (OLAP).
  • APIs: Sistemas externos recuperam dados em formato JSON em tempo real.
  • Modelos de Machine Learning: Algoritmos que consomem dados do Lake para treinar IAs.
