📊 Centralizado vs. Distribuído vs. Paralelo

Como os componentes e dados do SGBD estão distribuídos fisicamente.

Modelo Características de Prova Principal Vantagem / Desafio
Centralizado Tudo roda em um único servidor físico central. Fácil administração, mas cria um ponto único de falha.
Paralelo Múltiplos processadores/discos no mesmo local físico para acelerar consultas. Altíssimo desempenho e escalabilidade vertical (Scale-up).
Distribuído Dados geograficamente separados em vários servidores interligados por rede. Altamente tolerante a falhas; desafio de controle de concorrência global.

🚨 Pegadinhas de Prova

  • Shared Nothing vs Shared Disk: A banca vai dizer que na arquitetura "Shared Disk", cada computador tem sua própria memória e seu próprio disco rígido individual.
    Falso! Quem possui discos isolados por nó é a arquitetura Shared Nothing (Nada Compartilhado). No Shared Disk, todos os nós acessam uma única Storage centralizada externa de forma compartilhada.
  • SGBD Distribuído Homogêneo vs Heterogêneo: A banca tentará afirmar que um sistema distribuído homogêneo usa diferentes softwares gerenciadores (ex: Oracle em um nó, PostgreSQL em outro).
    Falso! Sistema Homogêneo usa o exato mesmo software SGBD em todas as máquinas físicas. Usar softwares diferentes é a definição de SGBD Distribuído Heterogêneo (ou Federado).

1. Arquiteturas de SGBDs Paralelos

SGBDs paralelos foram criados para superar os limites físicos de um único computador, distribuindo as operações de leitura/escrita e processamento em vários processadores locais de alta velocidade. É a base teórica de processamento de Big Data e sistemas analíticos modernos.

As Três Abordagens Principais:

  • A. Memória Compartilhada (Shared Memory):
    Todos os processadores compartilham diretamente uma única memória RAM global do sistema e todos os discos de armazenamento disponíveis.
    O Gargalo: Conforme adicionamos mais processadores, a barra de barramento (bus) de comunicação com a memória RAM fica congestionada. Não escala de forma infinita (limite de escalabilidade vertical).
  • B. Disco Compartilhado (Shared Disk):
    Cada processador possui sua memória RAM local dedicada (evitando congestionamento de barramento de RAM), mas todos os nós continuam compartilhando o acesso direto à mesma matriz de discos físicos através de uma rede SAN de alta velocidade.
    Exemplo de mercado: Oracle Real Application Clusters (RAC).
  • C. Nada Compartilhado (Shared Nothing):
    É a joia da coroa para Engenharia de Big Data! Cada nó do sistema é uma máquina autônoma completa, com seu próprio processador, sua própria memória RAM e seus próprios discos físicos locais. Os nós se comunicam apenas através de uma rede de dados de alta velocidade.
    Vantagem de Ouro: Altamente escalável horizontalmente de forma linear (podemos juntar milhares de computadores comuns em cluster). É a arquitetura que roda por trás de sistemas de processamento massivo em paralelo (MPP) como o Hadoop/HDFS, Amazon Redshift, Snowflake e Cassandra.

2. Bancos de Dados Distribuídos (SGBDD)

Diferente do banco paralelo (onde as máquinas estão na mesma sala conectadas por cabos físicos de altíssima velocidade), um SGBD Distribuído gerencia bases de dados que estão espalhadas em locais geograficamente distantes (ex: uma filial em São Paulo e outra em Tóquio), interconectadas por redes WAN públicas ou privadas.

Classificação dos SGBDDs:

  • Homogêneos: Utilizam o mesmo software SGBD em todos os locais físicos (ex: todos rodam Oracle Database). Facilita o controle de concorrência global e as transações distribuídas.
  • Heterogêneos (ou Federados): Diferentes filiais usam softwares diferentes de banco de dados (ex: uma usa SQL Server, outra usa MySQL). O SGBD distribuído precisa atuar como um "tradutor" e unificador dos dados através de uma camada de middleware (federação).

Conceitos Avançados de Distribuição (Importantes!):

  • Fragmentação de Dados: Dividir uma tabela gigante em fatias para guardar pedaços em diferentes computadores locais.
    • Horizontal: Corta a tabela por linhas (tuplas). Ex: Linhas de clientes do RJ ficam no servidor do RJ, clientes de SP ficam em SP.
    • Vertical: Corta a tabela por colunas (atributos). Ex: Coluna de "dados bancários" fica no servidor restrito da auditoria, coluna de "nome de usuário" fica no servidor de uso comum.
  • Replicação de Dados: Guardar cópias idênticas da mesma tabela em múltiplos servidores distribuídos para melhorar a velocidade de leitura e garantir que o sistema continue de pé se uma filial sofrer um apagão de energia.

3. Arquitetura Cliente-Servidor (Níveis)

A arquitetura tradicional de acesso a bancos de dados é baseada no modelo cliente-servidor, que evoluiu em camadas (tiers):

  • Dois Níveis (2-Tier): O cliente (aplicação instalada no PC do usuário) se conecta diretamente ao servidor de banco de dados.
    Vulnerabilidade/Problema: O cliente precisa do driver de banco (ODBC/JDBC) instalado na própria máquina e tem acesso direto às portas lógicas do BD. Ruim para escala de internet.
  • Três Níveis (3-Tier): A arquitetura padrão web moderna. O cliente (navegador do usuário) fala apenas com uma camada intermediária: o Servidor de Aplicação (App Server). O Servidor de Aplicação roda a lógica de negócios e é o único elemento que se conecta e faz requisições ao servidor de Banco de Dados final. Aumenta massivamente a segurança e a escalabilidade.
