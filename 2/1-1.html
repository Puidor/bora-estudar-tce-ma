Dados Estruturados e Não Estruturados

🗂️ A Tríade da Estruturação

Estruturados

Altamente organizados em linhas e colunas. Possuem um modelo pré-definido e rígido (Esquema).

Ex: Bancos Relacionais (SQL), Planilhas Excel.

Semi-estruturados

Sem estrutura tabular rígida, mas usam marcadores (tags) ou chaves que separam e organizam a hierarquia.

Ex: JSON, XML, HTML, YAML.

Não Estruturados

Caos digital. Sem formato específico ou organização interna nativa rastreável de forma simples por BDs tradicionais.

Ex: Vídeos, Áudios, PDFs, Imagens, E-mails.

⚖️ O Paradigma do Esquema (Schema)

Como o banco de dados lida com a estrutura da informação.

Abordagem Schema-on-Write (Escrita) Schema-on-Read (Leitura)
Definição A estrutura (tabela, tipos) deve existir ANTES da inserção do dado. O dado é gravado "bruto". A estrutura só é aplicada NA HORA DA LEITURA/CONSULTA.
Tipo de Dado Estruturados Não Estruturados / Semi
Ambiente Típico Data Warehouse (SGBD Relacional) Data Lake (Hadoop, S3, NoSQL)

🚨 Pegadinhas de Prova (FGV, CEBRASPE)

  • A Ilusão do ERP: A banca diz que "A maior parte dos dados gerados no mundo hoje são estruturados, devido ao uso intensivo de sistemas ERP e CRM pelas empresas".
    Falso! Estima-se que mais de 80% a 90% dos dados gerados hoje em dia são NÃO ESTRUTURADOS (posts no Twitter/Instagram, áudios no WhatsApp, logs de IoT, vídeos, e-mails).
  • Estrutura Interna vs Estrutura Racional: "Um arquivo PDF possui páginas e parágrafos definidos, logo ele é considerado um dado estruturado".
    Falso! Ele possui estrutura visual, mas para o Banco de Dados, ele é um bloco de bits sem significado relacional. Portanto, textos longos e PDFs são sempre Dados Não Estruturados.

1. Dados Estruturados

São os dados mais "comportados" da Engenharia de Dados. O dado estruturado é aquele que reside num campo fixo e delimitado, geralmente organizado na forma de tabelas (linhas e colunas).

A sua principal característica é exigir um Esquema Rígido (Schema) pré-definido. Antes de inserir a idade de um cliente, o banco de dados já precisa ter uma coluna chamada "Idade" configurada para aceitar apenas números inteiros. Se você tentar gravar o texto "Vinte", o banco de dados irá rejeitar o dado.

  • Armazenamento Típico: Bancos de Dados Relacionais (SGBDR) como Oracle, PostgreSQL, MySQL, SQL Server. E também Data Warehouses.
  • Acesso e Busca: Muito rápidos e eficientes. Utiliza-se a linguagem padrão SQL (Structured Query Language).
  • Vantagem: Integridade alta, pouca redundância, facilidade extrema de geração de relatórios tabulares.

2. Dados Semi-estruturados

É o "meio-termo". Os dados semi-estruturados não possuem uma estrutura tabular rígida de banco de dados relacional, mas eles carregam metadados internos (tags, marcadores ou chaves) que descrevem a hierarquia e separam a informação, permitindo que a máquina os compreenda e processe.

Eles são conhecidos como dados autodescritivos. Diferente de uma tabela que obriga todos os registros a terem a mesma quantidade de colunas, um arquivo semi-estruturado permite que o "Cliente A" tenha um campo de telefone, mas o "Cliente B" não, sem que isso "quebre" a estrutura.

Exemplos Clássicos para Concurso:
  • • JSON (Padrão ouro em APIs REST e Bancos NoSQL como o MongoDB).
  • • XML (Padrão histórico para integrações via SOAP).
  • • HTML (Marcação de sites).

3. Dados Não Estruturados

Representam a vasta maioria da informação que flui no mundo moderno (fenômeno do Big Data). Eles não possuem um esquema pré-definido nem marcadores claros que delimitem a informação de forma padronizada.

Atenção: eles podem até ter alguma estrutura interna (um áudio mp3 tem cabeçalhos de codec), mas eles são "não estruturados" do ponto de vista de busca automatizada e indexação de dados. Você não consegue usar um simples `SELECT * FROM Audio WHERE Voz = 'João'` num banco de dados tradicional.

Características e Desafios

  • Exemplos: Arquivos de texto rico (.docx, .pdf), e-mails, vídeos, imagens satelitais, logs de sensores IoT soltos, áudios.
  • Armazenamento: Normalmente "despejados" como objetos (Blob) em Data Lakes (ex: Hadoop HDFS ou Amazon S3).
  • Abordagem de Leitura: Usam o paradigma Schema-on-read. A estrutura e a formatação só são impostas no momento em que um algoritmo de IA, Machine Learning ou PNL (Processamento de Linguagem Natural) puxa o dado para extrair valor.
