🚨 Pegadinhas Mortais de Prova

  • A Inversão de Papéis: A banca vai dizer que "É responsabilidade do DBA entrevistar os usuários de negócio para levantar os requisitos de informação e desenhar o Modelo Conceitual (MER)".
    Falso! Quem levanta requisitos de dados com os usuários e desenha o Modelo Conceitual é o Administrador de Dados (AD). O DBA entra em cena mais tarde, na fase Física.
  • Concessão de Acessos (Privilégios): "É função do AD definir quais as políticas de acesso e gerenciar as senhas no SGBD".
    Atenção! O AD pode até ajudar a definir *quem* deveria ter acesso (política de negócio), mas a execução técnica (dar o comando GRANT e REVOKE no sistema e criar usuários no SGBD) é tarefa essencial do DBA.

1. A Essência da Administração de Dados (AD)

A Administração de Dados (Data Administration) trata o dado como um ativo corporativo de alto valor. O profissional AD não precisa necessariamente saber programar em SQL avançado ou instalar servidores Linux. Seu foco é puramente informacional e negocial.

Principais Responsabilidades do AD:

  • Levantamento de Requisitos: Conversar com os usuários finais e gerentes para entender quais dados a empresa precisa guardar.
  • Modelagem de Dados (Conceitual e Lógica): Criar o Modelo Entidade-Relacionamento (MER) e aplicar as regras de normalização.
  • Resolução de Conflitos: Garantir que o departamento de Vendas e o departamento de Finanças usem o mesmo conceito para "Cliente".
  • Manutenção do Dicionário de Dados: Documentar o que cada campo significa (Metadados). Ex: "O campo vlr_venda armazena o valor em reais, com duas casas decimais, sem incluir o frete".

2. A Essência do Administrador de Banco de Dados (DBA)

O Administrador de Banco de Dados (Database Administrator) é o "mecânico chefe". Ele assume o trabalho assim que o AD termina a modelagem lógica. O DBA pega o modelo lógico e o transforma em tabelas reais num software SGBD (como Oracle, SQL Server ou PostgreSQL).

Principais Responsabilidades do DBA:

  • Instalação e Configuração: Instalar o software SGBD no servidor e aplicar patches de segurança.
  • Modelagem Física: Criar as tabelas reais (via DDL), definir tablespaces, tipos de dados físicos e alocação de discos.
  • Performance e Tuning: Monitorar a lentidão, reescrever consultas SQL ruins, criar índices e particionar tabelas pesadas.
  • Disponibilidade (Backup e Restore): Garantir que os backups estejam sendo feitos corretamente e, em caso de desastre, conseguir restaurar o banco rapidamente.
  • Segurança Física e de Acesso: Criar os usuários no sistema, atribuir as permissões corretas (GRANT/REVOKE) e blindar o banco contra invasões.

3. Metadados e o Dicionário de Dados

Para a prova, guarde esta definição mestre: Metadados são "dados sobre dados". Eles fornecem contexto e significado aos dados brutos.

Por exemplo, o número "35" solto no banco não significa nada. O metadado diz que aquele "35" pertence à coluna "Idade", da tabela "Clientes", cujo formato é "Inteiro", e que não pode ser nulo.

O Dicionário de Dados (Repositório)

É o catálogo central mantido pelo SGBD que armazena todos esses metadados. Ele contém:

  • Nomes e descrições exatas de todas as tabelas e colunas.
  • Restrições de integridade (Primary Keys, Foreign Keys, Not Null).
  • Regras de segurança e níveis de privilégios dos usuários.
  • Estatísticas de uso (para o otimizador de consultas do SGBD).

Curiosidade de Prova: Em muitos SGBDs (como o Oracle), o próprio dicionário de dados é armazenado no formato de tabelas e visões do sistema, que podem ser consultadas pelo DBA usando a linguagem SQL normal.
