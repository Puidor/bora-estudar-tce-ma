🚨 Pegadinhas de Prova: Hadoop vs Spark vs Kafka

  • Hadoop faz Streaming? A banca diz que "O MapReduce do Hadoop é altamente recomendado para o processamento de dados em tempo real (streaming)".
    Falso! O MapReduce é estritamente Batch (em lote) e lê/grava intensamente no disco, sendo lento. Para tempo real, usamos Kafka, Spark Streaming ou Flink.
  • O Spark substitui o HDFS? "O Apache Spark é um sistema de arquivos distribuído que substituiu completamente o HDFS".
    Falso! O Spark é um motor de PROCESSAMENTO em memória. Ele NÃO possui sistema de arquivos próprio de armazenamento persistente. O Spark usa o HDFS, o S3 da AWS ou bancos NoSQL para armazenar/ler os dados que ele vai processar.

1. A Essência do Big Data (Além do Volume)

Big Data não é apenas "ter muitos dados". É um ecossistema de tecnologias desenhado para tratar conjuntos de dados que são tão grandes e complexos que os aplicativos tradicionais de processamento de dados (como bancos relacionais únicos e Excel) simplesmente não conseguem lidar.

O Gartner e autores como Doug Laney definiram inicialmente os 3 V's originais, que depois foram expandidos para 5 (os mais cobrados):

  • Volume: A dimensão massiva. Exige armazenamento distribuído através de dezenas ou milhares de servidores paralelos (cluster).
  • Velocidade: O fluxo constante e o tempo de resposta exigido. Vai desde batch processing noturno (lento) até processamento de cartões de crédito em milissegundos (Streaming/Real-time).
  • Variedade: O calcanhar de Aquiles dos BDs antigos. Lidar não apenas com tabelas, mas com áudios, vídeos, JSONs de APIs e texto livre (caos estrutural).
  • Veracidade: Com muitos dados, surge o "ruído". Como garantir a qualidade, autenticidade e limpar dados espúrios de bots do Twitter ou sensores de IoT defeituosos?
  • Valor: Tudo isso custa caro. O Big Data tem que gerar insights acionáveis que se revertam em lucros, segurança ou melhoria para a empresa.

2. Ingestão Massiva de Dados

A ingestão é o processo de trazer os dados das fontes externas (ERPs, APIs Web, Redes Sociais, Dispositivos IoT) para dentro do Data Lake da empresa.

A. Ingestão em Lote (Batch)

Os dados são recolhidos de uma só vez numa janela de tempo programada (ex: toda meia-noite). É o método clássico para mover grandes bancos de dados (SGBDRs) sem sobrecarregar as redes durante o dia.
🛠️ Ferramenta clássica de prova: Apache Sqoop (SQL-to-Hadoop). Ferramenta para transferir dados de bancos relacionais para o HDFS de forma massiva.

B. Ingestão em Tempo Real (Streaming)

Aqui o dado flui de forma ininterrupta, e o sistema tem que reagir milissegundos após a geração do dado (ex: sensores de um carro autônomo, transações de banco).

A Estrela da Prova: Apache Kafka

É uma plataforma distribuída de streaming de eventos, funcionando com o paradigma Publish/Subscribe (Produtor/Consumidor).

  • Produtores (Producers): Sistemas que geram a informação e a jogam no Kafka.
  • Tópicos (Topics): "Pastas" ou categorias para onde as mensagens vão. São divididos em Partições para permitir processamento paralelo e tolerância a falhas.
  • Consumidores (Consumers): Sistemas (como o Spark Streaming) que "assinam" os tópicos e processam os dados assim que chegam.
  • Vantagem: O Kafka atua como um amortecedor (buffer) gigante. Se o banco de dados cair, o Kafka guarda as mensagens em disco até que o sistema retorne, não perdendo nada.

3. Armazenamento Distribuído: Hadoop HDFS

O Hadoop foi a primeira grande resposta open-source ao Big Data. O HDFS (Hadoop Distributed File System) é o coração do armazenamento. Ele resolve o problema: "Como eu guardo um arquivo de 1 Terabyte se o disco rígido mais barato só tem 500GB?"

Arquitetura Master-Slave

  • NameNode (O Mestre): É o "cérebro" do sistema. Ele NÃO guarda os dados reais. Ele guarda os Metadados (o índice). Ele sabe que o "Arquivo_X" foi fatiado e os pedaços foram para as Máquinas 2, 8 e 15. Ele coordena as operações.
  • DataNode (Os Escravos/Trabalhadores): São as máquinas (geralmente hardwares simples e baratos) que guardam os blocos reais de dados no disco rígido.

Características Clássicas de Prova (HDFS)

  • Tamanho do Bloco (Block Size): O HDFS trabalha com blocos gigantes, por padrão de 128 MB (em sistemas antigos era 64MB). Arquivos são divididos nestes blocos de 128MB e espalhados pelos DataNodes.
  • Tolerância a Falhas (Replicação): Máquinas baratas pifam o tempo todo. Para evitar perder dados, o HDFS faz cópias idênticas. O fator de replicação padrão é 3 (ou seja, cada bloco existe em 3 máquinas diferentes).
  • Write-Once, Read-Many (WORM): O HDFS foi feito para arquivos que são escritos uma vez e lidos milhares de vezes. Ele é péssimo se você precisar ficar atualizando os arquivos constantemente.

4. Processamento Massivo: MapReduce vs Spark

Armazenar é metade da batalha; processar milhares de discos em paralelo é o desafio real.

O Avô: MapReduce (Hadoop)

Modelo de programação criado para processamento paralelo de grandes lotes em clusters. Divide-se em duas fases:
1. Map (Mapear): O problema é dividido em pequenos pedaços e enviado para onde os dados já residem fisicamente (evitando trafegar 1TB pela rede). Ele gera pares de chave-valor temporários.
2. Reduce (Reduzir): Pega nos resultados temporários dos nós "Map", consolida tudo e entrega a resposta final.
O Problema: A cada etapa, o MapReduce escreve os resultados temporários no Disco Rígido. Isso causa uma extrema lentidão de Entrada/Saída (I/O).

O Sucessor Triunfante: Apache Spark

O Spark veio para corrigir a lentidão do MapReduce, sendo até 100 vezes mais rápido para certos trabalhos. A sua grande "arma secreta" cobrada em concursos é:

Processamento Em Memória (In-Memory Processing)

Ao invés de gravar os dados temporários no disco rígido, o Spark usa uma abstração chamada RDD (Resilient Distributed Dataset) ou, mais modernamente, DataFrames, e realiza todas as transformações complexas na Memória RAM dos servidores espalhados no cluster. Só vai para o disco no final, na hora de guardar o resultado!
