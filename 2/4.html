4. Abordagem Relacional

🗣️ O Dicionário Relacional (Codd)

Relação

O equivalente à Tabela na implementação SQL. É um conjunto matemático de tuplas.

Tupla

O equivalente a uma Linha / Registro. Representa uma ocorrência (um fato) daquela relação.

Atributo

O equivalente a uma Coluna / Campo. Tem um nome e os valores pertencem a um Domínio.

🧮 Álgebra Relacional (O Coração do SQL)

Como extrair dados usando as operações matemáticas formais de Codd.

Operação (Símbolo) O que faz? (Cai na Prova!)
Seleção ( σ )
Sigma 		Filtra LINHAS (Tuplas) que satisfazem uma condição.
Em SQL: Equivalente à cláusula WHERE. (Ex: Pessoas com Idade > 30).
Projeção ( π )
Pi 		Filtra COLUNAS (Atributos). Escolhe quais atributos mostrar e descarta o resto.

Produto Cartesiano ( × ) Combina CADA linha da Tabela A com CADA linha da Tabela B. Se A tem 5 linhas e B tem 10, o resultado terá 50 linhas.
Junção ( ⨝ )
Join 		É uma operação derivada (Produto Cartesiano + Seleção). Junta duas tabelas com base numa condição de igualdade (ex: ID_Dep = ID_Dep).

🚨 Top Pegadinhas de Prova (Teoria pura!)

  • A Ordem Importa? A banca diz que "No modelo relacional, a ordem das linhas e das colunas determina o significado do dado armazenado".
    Falso! Por definição matemática (teoria dos conjuntos), não existe ordem para as tuplas (linhas) nem para os atributos (colunas) em uma Relação.
  • Linhas Duplicadas: "Uma relação relacional pura permite a existência de duas tuplas exatamente idênticas".
    Falso! Como uma relação é um conjunto matemático (Set), ela NÃO possui tuplas duplicadas. (Nota: Sistemas SQL na prática permitem se não houver Primary Key, mas na TEORIA RELACIONAL cobrada em prova, não pode!).
  • Seleção vs Projeção: Eles SEMPRE trocam os conceitos. Lembre-se: Projeção = Pé direito = Vertical (Colunas). Seleção = Horizontal (Linhas).

1. A Gênese do Modelo Relacional

Criado por Edgar F. Codd em 1970, o Modelo Relacional revolucionou o mundo dos bancos de dados. Antes dele, a navegação de dados era feita por modelos em rede ou hierárquicos, onde os programadores tinham que saber os caminhos físicos no disco para achar um dado.

A grande sacada de Codd foi usar a Teoria dos Conjuntos Matemáticos e a Lógica de Predicados para abstrair os dados. Tudo é organizado em coleções chamadas de "Relações" (que hoje conhecemos popularmente como Tabelas).

Conceitos Chave (Dicionário de Prova)

  • Relação (Relation): Tabela bidimensional bidimensional composta por tuplas e atributos.
  • Tupla (Tuple): Uma linha da relação. Representa uma instância ou entidade única.
  • Atributo (Attribute): O cabeçalho de uma coluna da relação.
  • Domínio (Domain): O conjunto de valores atômicos e indivisíveis permitidos para um atributo. (Ex: O domínio do atributo "Idade" seria os números inteiros positivos).
  • Grau ou Aridade (Degree/Arity): O número de atributos (colunas) de uma relação.
  • Cardinalidade (Cardinality): O número de tuplas (linhas) de uma relação no momento presente.

2. Propriedades de Ouro de uma Relação

As bancas de alto nível (como a FGV) não cobram apenas os nomes, mas cobram se você sabe as restrições teóricas que fazem de uma tabela uma "Relação" matemática pura.

  • 1. Não há tuplas duplicadas: Em um conjunto matemático, todos os elementos são distintos. Portanto, deve existir uma combinação de atributos que identifique a tupla (A Chave Primária).
  • 2. Tuplas não são ordenadas: A ordem física em que as linhas (tuplas) estão inseridas na tabela não importa, nem logicamente nem semanticamente. O Banco de Dados exibe como preferir, a menos que se aplique um filtro (como o `ORDER BY` em SQL).
  • 3. Atributos não são ordenados: Similar às tuplas, a ordem das colunas da esquerda para a direita não tem significado.
  • 4. Valores Atômicos (Primeira Forma Normal intrínseca): Todo valor de cruzamento de Linha x Coluna deve ser escalar e indivisível. O modelo relacional puro NÃO suporta valores multivalorados (ex: uma lista de telefones na mesma célula) nem atributos compostos.

3. Integridade e Chaves Relacionais

Para garantir que o banco não se torne um depósito de lixo inconsistente, Codd definiu Restrições (Constraints).

A. Restrição de Entidade (A Chave Primária)

A Restrição de Integridade de Entidade dita que NENHUM componente da Chave Primária (PK) de uma relação base pode aceitar valores NULOS (NULL). O Nulo significa "desconhecido", e se a PK serve para identificar a linha, ela não pode ser desconhecida.

B. Restrição Referencial (A Chave Estrangeira)

A Restrição de Integridade Referencial é a cola que une o banco relacional. Uma Chave Estrangeira (FK) é um atributo em uma Relação 1 que se refere à Chave Primária (PK) da Relação 2.

A regra diz: O valor da Chave Estrangeira (FK) DEVE OBRIGATORIAMENTE coincidir com algum valor existente de PK na tabela de origem, OU deve ser totalmente NULL (se a participação for opcional). Você não pode cadastrar um funcionário no "Departamento 99" se o departamento 99 não existir na tabela Departamentos.

As Subdivisões das Chaves

  • Superchave: Qualquer atributo ou combinação de atributos que identifica a tupla. (Ex: Nome + CPF + Email). Não importa se tem "excesso de bagagem".
  • Chave Candidata: Uma superchave mínima. Se você tirar uma coluna dela, perde a unicidade. (Ex: O CPF sozinho, ou o E-mail sozinho).
  • Chave Primária (PK): A chave candidata que foi "eleita/promovida" pelo DBA para ser o identificador principal.
  • Chaves Alternativas: As chaves candidatas que "perderam a eleição" para a PK (ex: O Email sobrou, mas ainda garante unicidade). O banco cria uma restrição UNIQUE para protegê-las.

4. A Álgebra Relacional (Fundamentos Teóricos)

Antes do SQL existir como linguagem computacional, as consultas eram baseadas puramente nesta matemática. A Álgebra Relacional pega em uma ou mais relações (tabelas) de entrada e produz uma nova relação como resultado (fechamento relacional).

As 5 Operações Primitivas (Básicas):
  • 1. Seleção (σ - Sigma): Operação unária (atua sobre 1 tabela). Retorna LINHAS baseadas num predicado (ex: salário > 5000). A Aridade (nº de colunas) do resultado é igual à da entrada, mas a Cardinalidade (nº de linhas) será menor ou igual.
  • 2. Projeção (π - Pi): Operação unária. Retorna COLUNAS específicas. Importante: a Projeção elimina automaticamente linhas duplicadas do resultado final.
  • 3. Produto Cartesiano (×): Operação binária (2 tabelas). Combinação explosiva de tudo com tudo.
  • 4. União (∪): Operação binária de conjuntos. Para unir duas tabelas, elas precisam ser "compatíveis para união" (ter o mesmo número de colunas e os mesmos domínios). Adiciona as linhas da Tabela B embaixo da Tabela A. Elimina duplicatas.
  • 5. Diferença de Conjuntos (-): Retorna as tuplas que estão na Tabela A, mas NÃO estão na Tabela B. Também exige compatibilidade de união.

A partir destas primitivas, derivam-se operações avançadas que usamos muito em SQL:

  • Intersecção (∩): Retorna o que é comum às duas tabelas (O que está em A E em B).
  • Junção (⨝ - Join): É a operação mais famosa. Junta duas tabelas quando há atributos em comum. Uma Junção Natural (Natural Join) junta tabelas baseada em atributos com nomes idênticos nas duas pontas e remove a coluna duplicada no resultado final.
