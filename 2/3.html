3. Modelagem de Dados

📐 O Ciclo de Vida da Modelagem

1️⃣

Conceitual

Independente do SGBD. Focado no Negócio e no Usuário. Responde "O QUE" deve ser guardado.

Ferramentas: Entidades, MER, DER.

2️⃣

Lógica

Depende do tipo de SGBD (Ex: Relacional), mas não do fabricante. Responde "COMO" será guardado.

Ferramentas: Tabelas, PK, FK, Normalização.

3️⃣

Física

Totalmente amarrado ao SGBD escolhido (Oracle, MySQL). Focado na Máquina/Software.

Ferramentas: Scripts SQL, DDL, Índices, Tipos (VARCHAR).

🔄 Mapeamento Clássico (Conceitual ➔ Lógico)

Como os Losangos (Relacionamentos) viram Tabelas? (Despenca em Provas)

Relacionamento (Conceitual) Como Fica no Modelo Lógico (Relacional)?
1 : 1

(Um para Um) 		A Chave Primária (PK) de uma entidade vai como Chave Estrangeira (FK) para a outra. A escolha de quem recebe a FK é do projetista (geralmente vai para a entidade que tem participação total).
1 : N

(Um para Muitos) 		A entidade do lado "N" (Muitos) RECEBE a Chave Estrangeira (FK) da entidade do lado "1".
Ex: 1 Departamento tem N Funcionários. A tabela "Funcionário" ganha a coluna "ID_Departamento".
N : M

(Muitos para Muitos) 		Cria uma NOVA Tabela! Esta nova tabela terá uma chave primária composta, formada pelas Chaves Estrangeiras (FKs) das duas entidades originais.
Atributo Multivalorado Ex: Telefone. Vira uma NOVA Tabela no modelo lógico, contendo o valor do telefone + a FK do dono do telefone. (Isso resolve a Primeira Forma Normal - 1FN).

🚨 Top Pegadinhas de Modelagem

  • Tabelas na Modelagem Conceitual? A banca diz que "No modelo conceitual, definimos as tabelas e suas chaves primárias usando o DER".
    Falso! No modelo conceitual (MER/DER) nós NÃO falamos de Tabelas ou Chaves Estrangeiras (FK). Nós falamos de Entidades, Relacionamentos e Atributos. O conceito de "Tabela" e "Chave Estrangeira" nasce no Modelo Lógico Relacional.
  • A Confusão MER vs DER:
    O MER (Modelo Entidade-Relacionamento) é o modelo/teoria de dados (os conceitos de Peter Chen).
    O DER (Diagrama Entidade-Relacionamento) é o desenho (os retângulos e losangos) que representa o MER graficamente.

1. A Modelagem Conceitual (O Começo)

A modelagem de dados é o processo de projetar a estrutura do banco de dados para representar as informações do mundo real. O processo começa sempre pelo Nível Conceitual.

Nesta fase, o engenheiro ou analista senta com os usuários do sistema (o cliente) para entender o negócio. A ferramenta padrão universal é a Modelagem Entidade-Relacionamento (MER).

Elementos do MER/DER (Notação de Peter Chen)

  • Entidade (Retângulo): Representa um objeto do mundo real sobre o qual se deseja guardar dados (Ex: CARRO, CLIENTE).
    Atenção à Entidade Fraca (Retângulo com linha dupla): Aquela cuja existência depende de outra entidade forte. (Ex: O Dependente de um Plano de Saúde só existe no banco se o titular Funcionário existir).
  • Relacionamento (Losango): Associação entre entidades. É o verbo que as liga (Ex: O Cliente COMPRA o Carro).
  • Atributos (Elipse): As propriedades das entidades.
    • Atômicos / Simples: Indivisíveis (Ex: CPF, Idade).
    • Compostos: Podem ser divididos (Ex: Endereço é dividido em Rua, CEP, Bairro).
    • Multivalorados (Elipse dupla): O usuário pode ter vários ao mesmo tempo (Ex: Telefone, E-mail).
    • Derivados (Elipse tracejada): É um valor que pode ser calculado a partir de outro, não precisando ser guardado rigidamente (Ex: A Idade pode ser derivada da Data de Nascimento).

2. A Modelagem Lógica (O Padrão Relacional)

Após o DER ser aprovado pelo cliente, o projetista precisa traduzir esse desenho para a linguagem de um Banco de Dados. A Modelagem Lógica assume o modelo de implementação (por exemplo, Modelo Relacional de Edgar Codd).

Aqui é onde o linguajar muda. Deixamos de falar de "Entidades" e passamos a falar de Tabelas (Relações), Linhas/Registros (Tuplas) e Colunas (Atributos).

As Chaves do Modelo Lógico

  • Superchave: Qualquer conjunto de um ou mais atributos que, juntos, consigam identificar uma linha de forma única. (Ex: Nome + CPF + Idade).
  • Chave Candidata: É uma superchave "enxuta", mínima. Se você tirar qualquer atributo dela, ela perde a capacidade de identificar unicamente a linha. (Ex: O CPF sozinho é candidato. O Email sozinho também é).
  • Chave Primária (PK): É a chave candidata que o DBA (ou você) ESCOLHEU para ser a identificadora oficial da tabela. Ela nunca pode ser nula (Regra de Integridade de Entidade) e não pode se repetir.
  • Chave Estrangeira (FK): É um atributo numa tabela que é Chave Primária noutra tabela. É ela que faz o "Join" (a amarração/relacionamento) entre tabelas separadas. O valor da FK deve ser Nulo, ou deve existir obrigatoriamente na tabela de origem (Regra de Integridade Referencial).

3. Derivação e Cardinalidades

Para provas discursivas e objetivas difíceis, você deve saber ler as regras de negócio de acordo com a restrição de cardinalidade. A leitura é sempre (Mínimo, Máximo).

Exemplos Clássicos:
  • (0, 1) - Zero para Um: Participação Opcional. O funcionário pode NÃO ter ou ter no máximo 1 carro da empresa.
  • (1, 1) - Um para Um: Participação Obrigatória (Total). O departamento TEM QUE TER no mínimo 1, e no máximo 1 gerente.
  • (0, N) - Zero para Muitos: O cliente pode nunca ter comprado nada, ou ter várias compras no histórico.
  • (1, N) - Um para Muitos: Um curso tem que ter pelo menos 1 aluno matriculado para abrir turma, e pode ter infinitos (N) alunos.

O N:M Vira Tabela!

Se o diagrama DER mostra que 1 Aluno cursa N Disciplinas, e 1 Disciplina tem M Alunos (relação M:N), o SGBD relacional NÃO SUPORTA ISSO DIRETAMENTE.

No Mapeamento Lógico, nós quebramos esse losango e criamos uma tabela intermediária (ex: Historico_Escolar). Esta nova tabela vai "sugar" a Chave Primária (PK) da tabela Aluno e a PK da tabela Disciplinas. Juntas, elas formam a PK Composta da nova tabela (e ambas atuarão como FKs apontando para as tabelas originais).

4. A Modelagem Física

É a última etapa. Aqui nós implementamos o projeto lógico no hardware e no software reais. Deixa de ser teoria e vira código.

  • Scripts DDL: Geração dos comandos CREATE TABLE.
  • Tipos de Dados: Tradução do genérico para o específico. O que no lógico era "Texto", no físico vira VARCHAR2(50) (no Oracle) ou NVARCHAR (no SQL Server).
  • Performance: Onde aplicamos os Índices (B-Tree, Bitmap) para acelerar as buscas, e decidimos sobre o particionamento de tabelas nos discos rígidos (Tablespaces).
