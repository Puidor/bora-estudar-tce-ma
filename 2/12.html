E T L

Extract ➔ Transform ➔ Load

  • 📍 Onde Transforma? No meio do caminho. Num servidor dedicado de ETL (ex: PowerCenter, Pentaho), ANTES de gravar no destino.
  • 🎯 Destino Típico: Data Warehouse Tradicional (Relacional). O DW só recebe dados limpos e perfeitos.
  • 🛡️ Segurança/LGPD: Excelente. Pode mascarar CPFs no servidor do meio antes que o dado sensível chegue ao destino.
  • 🐢 Velocidade: Lento para grandes volumes (Big Data), pois o servidor do meio vira um gargalo.
🚀

E L T

Extract ➔ Load ➔ Transform

  • 📍 Onde Transforma? No próprio destino! Puxa o dado bruto e joga no destino. O próprio destino faz o cálculo (via SQL pesado).
  • 🎯 Destino Típico: Data Lake ou Cloud Data Warehouse Moderno (Snowflake, BigQuery).
  • 💾 Flexibilidade: Excelente. O dado bruto ("Raw") fica salvo. Se as regras de negócio mudarem amanhã, você ainda tem o dado original.
  • ⚡ Velocidade: Altíssima ingestão. Aproveita o poder de processamento em cluster da Nuvem.

🚨 Pegadinhas de Ingestão e Integração

  • A Ilusão do ELT Leve: A banca diz que "No modelo ELT, como a transformação ocorre no destino, não é necessário hardware potente para processamento".
    Falso! Exige-se hardware MUITO potente no destino (ex: clusters na nuvem), pois o Data Warehouse ou Data Lake assumirá 100% da carga pesada de transformar dados via consultas SQL massivas.
  • CDC baseado em Log vs Query: "O Change Data Capture feito através de queries (SELECT na tabela de origem procurando por Data_Alteracao) é a forma que menos impacta a performance do banco de origem".
    Falso! Fazer SELECTs constantes impacta fortemente a origem. A melhor forma (e mais cobrada) é o CDC baseado em LOG (Log-based CDC), onde a ferramenta apenas "lê" silenciosamente o arquivo de log de transações do banco, gerando quase zero overhead na origem.

1. A Evolução: Do ETL para o ELT

A integração de dados envolve mover dados de Sistemas Operacionais (OLTP) para Sistemas Analíticos (OLAP). Historicamente, a arquitetura clássica é o ETL. Contudo, com o barateamento do armazenamento na nuvem e o surgimento dos Data Lakes, o mercado migrou fortemente para o ELT.

O Modelo ETL Clássico

No ETL, existe um Servidor de Integração dedicado no meio do caminho. O dado é Extraído da origem, colocado na memória desse servidor do meio, onde sofre Transformações pesadas (limpeza, junções, cálculos, anonimização), e só então a versão final e limpa é Carregada no Data Warehouse.
Ponto Forte (Prova): Protege o Data Warehouse, pois o DW só recebe dados modelados. Evita que dados sensíveis cruzem a rede, pois são limpos no meio do caminho.

O Modelo ELT Moderno (Big Data)

No ELT, o dado é Extraído e imediatamente Carregado no destino (geralmente um Data Lake no formato "Bruto" / Raw Zone). Somente depois de estar a salvo no Data Lake/Warehouse é que scripts SQL ou Spark são acionados para Transformar o dado lá dentro.
Ponto Forte (Prova): A Ingestão é ultra-rápida. Você não perde tempo transformando dados durante o tráfego. Além disso, você guarda o histórico do dado cru; se daqui a dois anos a regra de negócio mudar, você pode reprocessar o dado bruto original sem precisar extrair de novo do banco operacional.

2. Integração via Base de Dados (CDC)

Se uma empresa faz 10 mil vendas por dia, é inviável fazer um "SELECT * FROM Vendas" toda noite para alimentar o Data Warehouse (demoraria horas e derrubaria a performance). A solução exigida em provas é a ingestão incremental usando CDC (Change Data Capture).

O CDC captura apenas o que mudou (INSERTS, UPDATES, DELETES) desde a última extração.

  • CDC Baseado em Trigger: Cria-se um "Gatilho" no banco de origem. Sempre que alguém vende algo, o gatilho grava o dado novo numa tabelinha auxiliar.
    Ponto Fraco: O overhead (peso) na origem é enorme, deixando o banco operacional lento.
  • CDC Baseado em Query (Data/Timestamp): A origem tem uma coluna "Data_Atualizacao". O ETL roda a query WHERE Data_Atualizacao > Ontem.
    Ponto Fraco: Não consegue capturar "DELETES" (pois se a linha foi apagada, a data sumiu com ela).
  • CDC Baseado em LOG (O Padrão Ouro): Todo SGBDR guarda um log de segurança interno de tudo que acontece (Write-Ahead Log, Redo Log). Ferramentas como o Debezium ou GoldenGate "leem" silenciosamente esse arquivo de texto de log por fora, sem usar a CPU do banco de dados, e replicam a mudança para a nuvem.
    Vantagem em Prova: Quase zero overhead e captura TUDO (inclusive deletes) em tempo real.

3. Transferência de Arquivos (Batch Ingestion)

Mesmo com APIs e CDC modernos, grande parte do volume de dados do mundo ainda trafega através de arquivos (Files). Essa modalidade é chamada de Ingestão Batch (em Lote).

Formatos de Arquivo (Foco em Big Data)

  • CSV: Padrão aberto. Fácil leitura humana, mas ocupa muito espaço, não guarda tipos de dados (metadados ruins) e é terrível para consultas analíticas pesadas.
  • JSON: Padrão para APIs e dados Semi-estruturados. Autodescritivo, mas também ocupa muito espaço e é lento para fazer agregações.
  • Parquet / ORC: Os Queridinhos do Big Data. São arquivos Orientados a Colunas e com Alta Compressão. Se você quer integrar 1 Terabyte de dados do seu sistema de faturamento para um Data Lake na nuvem, exportar e transferir um arquivo Parquet gastará menos banda de rede e permitirá consultas milhares de vezes mais rápidas no destino.

MFT (Managed File Transfer)

Em concursos superiores, cita-se o MFT. Quando bancos precisam enviar arquivos TXT bilionários de compensação de boletos, o FTP não serve (não tem auditoria e cai no meio). Ferramentas MFT garantem que a Transferência de Arquivo tenha criptografia de ponta-a-ponta, retry automático se o link cair, compressão on-the-fly e trilha de auditoria para compliance.
