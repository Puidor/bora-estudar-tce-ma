Arquitetura e Políticas de Armazenamento

🗄️ Arquitetura de Banco: Linha vs Coluna

Como os dados são fisicamente gravados no disco rígido.

Característica Row-oriented (Por Linha) Column-oriented (Por Coluna)
Gravação Física Grava o "Registro 1" inteiro, depois o "Registro 2" inteiro. Grava todos os "Nomes" juntos, depois todas as "Idades" juntas.
Melhor para... Transações Rápidas (OLTP) Análise de Dados (OLAP / DW)
Operação Ideal INSERT e UPDATE de um único cliente. SUM() ou AVG() de uma única métrica para milhões de linhas.
Compressão de Dados Baixa (Dados misturados). Altíssima (Dados do mesmo tipo agrupados).

🌡️ Data Tiering (Armazenamento em Camadas)

HOT

Dados Quentes (Online)

Dados acessados constantemente. Precisam de latência zero e alto desempenho. Custo de armazenamento mais alto.
Hardware: SSDs NVMe, Memória RAM (Redis), SANs de alta performance.

WARM

Dados Mornos (Nearline)

Dados acessados com frequência moderada. Desempenho aceitável com custo menor.
Hardware: Discos rígidos (HDD) SAS/SATA rápidos, Storage NAS, Cloud Standard.

COLD

Dados Frios (Offline / Archive)

Dados raramente ou nunca acessados (guardados por lei/compliance). Acesso muito lento (horas), mas com o menor custo financeiro possível.
Hardware: Fitas Magnéticas (LTO), Cloud Glacier (AWS Glacier, Azure Archive).

🚨 A Pegadinha Suprema: Backup vs Arquivamento

  • A banca vai tentar te convencer que fazer o Arquivamento (Archiving) de dados velhos é a mesma coisa que fazer Backup.
  • Backup (Cópia): É criar uma CÓPIA secundária de um dado que continua em uso (ativo). O objetivo é Recuperação (Restore) caso o dado original quebre ou suma. É uma foto do "agora".
  • Arquivamento (Mover): O dado perdeu a utilidade diária (ex: notas fiscais de 2010). O sistema REMOVE o dado do servidor rápido principal e o MOVE para um armazenamento lento e barato (Cold). O objetivo é liberar espaço e manter a conformidade (Compliance). Não é uma "cópia redundante", o dado só existe no arquivo morto.

1. Arquitetura Física do SGBD (Oracle/SQL Server)

A administração de banco de dados exige entender como o SGBD mapeia objetos lógicos (tabelas e índices) em arquivos físicos no disco.

A Estrutura Lógica vs Física

  • Instância vs Banco de Dados: A Instância é a "alma" (processos rodando na CPU + estruturas na memória RAM). O Banco de Dados é o "corpo" (os arquivos físicos gravados no disco rígido).
  • Tablespace: É o maior nível lógico. Um agrupamento lógico de dados afins. Exemplo: Criar uma Tablespace separada só para Índices, ou só para a tabela de Vendas.
  • Datafiles: O nível físico! Toda Tablespace é apoiada por um ou mais Datafiles (arquivos reais .dbf ou .mdf no Windows/Linux). Se o datafile sumir, o banco quebra.
  • Blocos (Blocks) ou Páginas (Pages): É a menor unidade de leitura/escrita do SGBD (geralmente 8KB). Quando você faz um SELECT, o banco não lê "uma linha", ele traz o bloco inteiro do disco para a memória RAM.

2. Armazenamento Row-store vs Column-store

Por décadas, os bancos relacionais (SGBDR) usaram armazenamento orientado a linhas (Row-store). Nele, todos os campos de um registro são gravados sequencialmente num bloco do disco. Isso é excelente para o caixa do supermercado (OLTP): para gravar a "Venda 1", o disco grava a linha inteira de uma vez.

Porém, para Data Warehouses (OLAP), isso é péssimo. Se você quiser saber a "Média do Salário" de 1 milhão de clientes, o banco Row-store terá que carregar para a memória 1 milhão de blocos contendo os Nomes, Endereços, CPFs... só para ler o campo "Salário" no meio deles, causando um imenso desperdício de I/O (Input/Output).

A Revolução Colunar (Column-store)

Bancos colunares (ex: Amazon Redshift, Apache Cassandra, Google BigQuery) e formatos de arquivo Big Data (Parquet, ORC) gravam os dados agrupados por coluna no disco.

  • Vantagem de I/O: Se você pedir a "Média do Salário", o banco lê apenas o bloco do disco que contém puramente "Salários", ignorando Nomes e Endereços. A consulta é incrivelmente mais rápida.
  • Vantagem de Compressão: Como uma coluna inteira de "Status" pode ter apenas "Ativo" e "Inativo", o algoritmo de compressão funciona de forma formidável. A compressão num banco colunar é exponencialmente maior que num banco em linha.
  • Desvantagem: Fazer um INSERT de uma linha nova inteira é muito lento, porque o banco precisa quebrar o registro e gravar em 50 arquivos/blocos de colunas diferentes espalhados pelo disco.

3. Políticas de Armazenamento e Retenção

A administração de dados envolve o ciclo de vida da informação (ILM - Information Lifecycle Management). A Política de Retenção define regras corporativas baseadas em leis ou necessidades de negócio.

Exemplo Prático (A Regra do Negócio):

  • Ano 1 (Dados Quentes): A lei exige acesso imediato a Notas Fiscais recentes para fiscalizações de rotina. Ficam armazenados no Banco de Dados em SSDs rápidos.
  • Anos 2 a 5 (Dados Mornos): Os dados são consultados esporadicamente para relatórios anuais. O DBA cria uma rotina automatizada que move esses dados (particionamento de tabela) para uma Tablespace que fica gravada num disco HDD mais barato e lento.
  • Ano 6 a 10 (Dados Frios): O prazo legal pede para guardar por 10 anos. Ninguém consulta isso. A política manda Arquivar (Archiving). Os dados saem do banco online e vão para Fitas LTO ou Cloud Glacier.
  • Ano 11 (Expurgo/Descarte): Fim do ciclo de vida. A política de retenção expirou. O dado é destruído de forma segura (Purge/Wipe) para economizar custos e reduzir riscos (LGPD).

4. Deduplicação e Storage

Parte vital da política de armazenamento e controle de custos de Data Centers modernos (SAN/NAS) é a Deduplicação de Dados.

Diferente da Compressão clássica (ZIP) que foca em diminuir o tamanho de um único arquivo encontrando padrões repetidos dentro dele, a Deduplicação atua em todo o disco ou storage eliminando blocos idênticos entre vários arquivos diferentes.

Tipos de Deduplicação em Provas:
  • Na Origem (Source-based): O dado é deduplicado no servidor antes de ir pela rede até ao backup. Poupa muita banda de rede, mas consome CPU do servidor original.
  • No Destino (Target-based): O dado viaja cru pela rede. Chegando no Storage de destino, ele é deduplicado. Não pesa na CPU da origem, mas consome banda pesada.
