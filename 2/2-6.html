📐 O Teorema CAP (Decoreba de Ouro)

Num sistema distribuído (como clusters de Big Data e NoSQL), é IMPOSSÍVEL garantir simultaneamente as três propriedades. Você só pode escolher DUAS.

C

Consistency

(Consistência)
Todos os nós do cluster veem exatamente os mesmos dados ao mesmo tempo.

A

Availability

(Disponibilidade)
Toda requisição recebe uma resposta de sucesso ou falha, mesmo se algum nó "morrer".

P

Partition tolerance

(Tolerância a Partição)
O sistema continua operando mesmo se houver falha de rede/comunicação entre os nós.

Como a tolerância a falhas de rede (P) é obrigatória na internet, bancos NoSQL geralmente são CP (Consistência + Tolerância) ou AP (Disponibilidade + Tolerância).

Família Como armazena? (Estrutura) Casos de Uso Ideais Exemplos (Em Prova)

🚨 Pegadinhas de Prova: ACID vs BASE

  • A Ilusão do Fim do SQL: "A sigla NoSQL significa que não se deve usar a linguagem SQL para consultar os dados."
    Falso! A sigla significa "Not Only SQL" (Não Apenas SQL). Vários bancos NoSQL suportam dialetos de SQL para fazer consultas de forma mais amigável.
  • Bancos Relacionais (SQL) usam ACID. Bancos NoSQL usam BASE: As bancas dizem que os bancos NoSQL garantem integridade transacional ACID absoluta (Atomicidade, Consistência, Isolamento, Durabilidade).
    Falso! Para conseguir distribuir dados massivos em múltiplos nós e ter alta performance (Big Data), os bancos NoSQL geralmente abrem mão do rigor do ACID e usam as propriedades BASE (Basicamente Disponível, Estado Flexível, Consistência Eventual). Se você fizer uma compra no site, pode demorar alguns milissegundos para outro servidor no mundo refletir a compra, mas "eventualmente" tudo ficará consistente.

1. O Que é NoSQL e Por Que Surgiu?

Bancos de dados relacionais clássicos (SGBDR, como Oracle ou MySQL) são fantásticos para consistência transacional e relacionamentos organizados (Tabelas normalizadas). Contudo, eles possuem um "teto de vidro": a Escalabilidade Vertical. Para fazer um SGBDR aguentar mais carga, você geralmente precisa comprar um servidor maior, com mais CPU e mais RAM (Scale-up), o que é caríssimo e tem limite físico.

Com a explosão da Web 2.0 (Google, Amazon, Facebook) e do Big Data, surgiu a necessidade de guardar Petabytes de dados não estruturados gerados a cada segundo. A solução foi a Escalabilidade Horizontal (Scale-out): espalhar os dados por centenas de computadores baratos normais (cluster).

O modelo relacional lida muito mal com divisão de tabelas em dezenas de computadores (joins distribuídos são muito lentos). Assim nasceram os bancos NoSQL (Not Only SQL): feitos para serem distribuídos desde o nascimento, lidando com grandes volumes, sem esquema fixo (schema-less), oferecendo flexibilidade extrema.

2. O Teorema CAP e Propriedades BASE

A teoria de sistemas distribuídos tem um pilar inquebrável formulado pelo cientista da computação Eric Brewer: O Teorema CAP. Ele afirma que em um sistema que compartilha dados pela rede, você pode escolher, no máximo, duas destas três garantias simultâneas:

  • C - Consistência (Consistency): Toda leitura recebe a escrita mais recente ou um erro. Ninguém lê dados "velhos".
  • A - Disponibilidade (Availability): Toda requisição (leitura/escrita) feita a um nó do cluster que não esteja quebrado recebe uma resposta de sucesso garantida, mas não garante que contém o dado mais recente.
  • P - Tolerância a Partição (Partition Tolerance): O sistema continua a funcionar mesmo que alguns pacotes de rede sejam descartados ou atrasados entre os nós do cluster (a rede foi "partida").

A Lógica da Consistência Eventual

Como falhas de rede na internet (P) são inevitáveis, a sua escolha prática é entre CP ou AP. Bancos de dados relacionais são tradicionalmente CA (assumem que a rede local nunca falha e, se falhar, o sistema para).

Para conseguir ter AP (Disponibilidade e Tolerância altíssimas para os sites não caírem), os bancos NoSQL adotam as Propriedades BASE. A mais importante para as provas é o E de BASE: Eventual Consistency (Consistência Eventual). Significa que, se não houver novas atualizações a um dado, *eventualmente* todos os acessos retornarão o último valor atualizado. Enquanto os nós do cluster se sincronizam, os usuários podem temporariamente ver valores antigos.

3. Deep Dive nas 4 Famílias de Bancos NoSQL

1. Chave-Valor (Key-Value Store)

São os bancos de dados NoSQL mais simples e de execução mais rápida. Tratam os dados como um gigantesco dicionário (Hash table). Cada valor bruto é mapeado para uma chave primária.

Prós: Desempenho absurdo (muitos rodam inteiramente na memória RAM). Perfeitos para Cache, controle de sessão de usuários (login), carrinhos de e-commerce.
Sistemas Clássicos: Redis, Memcached, Amazon DynamoDB, Riak.

2. Orientado a Documentos (Document Store)

Um "upgrade" lógico do chave-valor. Em vez de o valor ser uma "caixa preta", o valor é um documento de formato estruturado (geralmente arquivos JSON ou BSON). Isso permite que o banco entenda a estrutura interna da informação e você consiga fazer consultas pelas propriedades internas do documento (ex: "Traga todos os documentos cujo campo 'Idade' > 30").

Prós: Esquema flexível (um documento cliente pode ter telefone, o outro pode não ter). Ideal para sistemas de conteúdo (CMS), catálogos, perfis de redes sociais.
Sistemas Clássicos: MongoDB (O grande rei das provas), CouchDB.

3. Família de Colunas (Column-Family / Wide-Column)

Enquanto o SGBDR grava e lê uma linha inteira de dados no disco (ex: Lê ID, Nome, Idade, Salário), os bancos colunares gravam as colunas juntas. Se você só quer processar a média dos salários, ele só puxa a coluna "Salários" do disco, ignorando o resto. É esmagadoramente mais rápido para agregações analíticas (OLAP).

Prós: Processamento muito veloz em colunas, alta capacidade de compressão de dados semelhantes, exímios em ingerir montanhas de dados (altíssima taxa de Escrita).
Sistemas Clássicos: Apache Cassandra (Desenvolvido pelo Facebook), HBase (Da família Hadoop).

4. Baseado em Grafo (Graph Database)

Criados não para os dados em si, mas para os RELACIONAMENTOS entre os dados. Os dados são armazenados como Nós (Vértices) e os relacionamentos como Arestas (Edges). No banco de dados relacional (SGBDR), fazer dezenas de conexões (`JOINs`) para descobrir os "amigos dos amigos dos amigos" de uma pessoa causa o travamento completo do sistema. O banco de grafo faz isso nativamente em milissegundos.

Prós: Navegação ágil em redes complexas de relacionamentos, sistemas de recomendação (Amazon/Netflix sugerindo produtos baseados nos seus amigos), deteção de fraudes financeiras e grafos de conhecimento.
Sistemas Clássicos: Neo4j, Amazon Neptune.
