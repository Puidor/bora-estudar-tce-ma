Linguagem DDL (Definição de Dados)

⚡ Direto ao Ponto

  • O que faz? A DDL (Data Definition Language) é responsável por criar, alterar e destruir a ESTRUTURA (Esquema) do banco de dados (Tabelas, Visões, Índices).
  • Não manipula as linhas: A DDL não insere valores nas células nem atualiza o salário de um funcionário. Quem faz isso é a DML (Data Manipulation Language).
  • Auto-Commit (Pegadinha clássica): Na maioria dos SGBDs comerciais (como Oracle), executar um comando DDL salva a transação instantaneamente (commit implícito). Não dá para dar ROLLBACK num DROP TABLE!

🛠️ O Quarteto Fantástico da DDL

CREATE

Cria um novo objeto no banco de dados. Seja uma Tabela (CREATE TABLE), um Índice, uma View ou um Schema.

ALTER

Modifica a estrutura de um objeto existente. Usado para adicionar uma nova coluna, mudar o tipo de dado (de INT para VARCHAR) ou renomear coluna.

DROP

Apaga a ESTRUTURA inteira. O DROP TABLE joga no lixo a tabela, todas as suas colunas e todos os dados dentro dela. Deixa de existir.

TRUNCATE

Limpa os dados, mantém a estrutura. Apaga TODAS as linhas de uma vez só, de forma ultra rápida, zerando a tabela.

1. A Essência do Comando CREATE

O CREATE TABLE é o comando mais básico da DDL. Ele não apenas define as colunas e os tipos de dados (VARCHAR, INT, DATE), mas também é o momento onde aplicamos as Restrições (Constraints) de Integridade.

Principais Constraints (Restrições)

  • PRIMARY KEY: Garante a identificação única. É a junção automática de UNIQUE com NOT NULL. Pode ser aplicada a uma coluna (chave simples) ou várias (chave composta).
  • FOREIGN KEY: Garante a integridade referencial. Obriga que o valor inserido nesta coluna exista na tabela Pai.
  • UNIQUE: Garante que não haja valores repetidos naquela coluna (ex: Coluna de CPF). Ao contrário da PK, permite a inserção de valor nulo em muitos SGBDs.
  • NOT NULL: Obriga o preenchimento do campo na hora do INSERT.
  • CHECK: Valida o dado antes de entrar. Exemplo: CHECK (idade >= 18). Se você tentar inserir um menor de idade, o banco rejeita.
  • DEFAULT: Insere um valor padrão automaticamente caso o usuário esqueça de preencher o campo no INSERT.

2. A Flexibilidade do ALTER TABLE

A modelagem de dados muda com o tempo. A empresa cresceu e o campo de CEP ficou pequeno, ou é preciso adicionar uma coluna de "Status". Para não termos que dar um DROP na tabela e perder tudo, usamos o ALTER TABLE.

-- Adicionar uma coluna:
ALTER TABLE clientes ADD telefone VARCHAR(15);

-- Modificar um tipo de dado:
ALTER TABLE clientes MODIFY telefone VARCHAR(20);

-- Apagar uma coluna inteira:
ALTER TABLE clientes DROP COLUMN telefone;

3. A Fina Arte do TRUNCATE vs DELETE

Por que inventaram o TRUNCATE se já tínhamos o DELETE para apagar dados? Esta é a pergunta de ouro.

  • A Mecânica do DELETE (DML): Quando você faz um DELETE FROM tabela;, o banco de dados lê linha por linha e escreve no arquivo de log de transações: "Apaguei a linha 1... Apaguei a linha 2...". Isso permite que você faça um ROLLBACK caso se arrependa. O lado ruim? Se a tabela tiver 50 milhões de linhas, o DELETE vai demorar horas e inchar o log.
  • A Mecânica do TRUNCATE (DDL): O Truncate não apaga linha por linha. Ele vai direto no ponteiro de alocação de memória do disco e diz: "Esta tabela está vazia agora". O tempo de execução é de frações de segundo, independentemente se a tabela tem 10 linhas ou 50 milhões. O lado ruim? Não faz log linha a linha, e na imensa maioria dos SGBDs, não pode ser revertido por um simples ROLLBACK.

4. Auto-Commit e Implicações Transacionais

Esta é uma pegadinha muito sutil cobrada pela FGV.

Quando usamos comandos DML (Insert, Update, Delete), iniciamos uma Transação. Se o computador desligar no meio de um update, o SGBD faz o rollback quando religar. Você precisa digitar ativamente o comando COMMIT para gravar os dados de forma permanente no disco duro.

No entanto, os comandos DDL (Create, Alter, Drop, Truncate) efetuam um COMMIT IMPLÍCITO (no Oracle e na maioria dos bancos).

Cenário de Prova: Um usuário faz um UPDATE alterando salários. Em seguida, sem dar commit, ele escreve um CREATE TABLE teste(id int);. O que acontece com o Update?
Resposta: O Update é comitado/salvo automaticamente! O comando DDL salvou todo o trabalho pendente da sessão antes de executar a criação da tabela.
