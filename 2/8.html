Linguagem SQL (DQL)

⚡ Direto ao Ponto (A Sintaxe)

  • Linguagem Declarativa: No SQL, você diz O QUE quer (ex: os salários maiores que mil), e não COMO o banco deve fazer para encontrar essa informação.
  • DQL (Data Query Language): É o subconjunto do SQL focado apenas em consultas. O seu único e todo-poderoso comando é o SELECT.
SELECT coluna1, SUM(coluna2)
FROM tabela_a
JOIN tabela_b ON a.id = b.id
WHERE condicao = verdadeiro
GROUP BY coluna1
HAVING SUM(coluna2) > 100
ORDER BY coluna1 DESC;

🔗 Tipos de JOIN (Junções)

INNER

INNER JOIN

Retorna APENAS os registros que possuem correspondência em ambas as tabelas (Interseção).

LEFT

LEFT JOIN

Retorna TODOS da tabela Esquerda (A), e os correspondentes da Direita (B). Se não achar no B, preenche com NULL.

RIGHT

RIGHT JOIN

Retorna TODOS da tabela Direita (B), e os correspondentes da Esquerda (A). Se não achar no A, preenche com NULL.

FULL

FULL OUTER JOIN

Retorna TODOS os registros de ambas as tabelas. Onde não há correspondência, preenche com NULL dos dois lados.

🚨 Pegadinhas Mortais de Prova

  • WHERE vs HAVING: "A cláusula WHERE pode ser usada para filtrar resultados de funções de agregação, como SUM(valor) > 10".
    Falso! O WHERE filtra linhas antes do agrupamento. Ele NÃO enxerga funções de agregação. Para filtrar o resultado de um SUM, AVG ou COUNT, você DEVE usar a cláusula HAVING.
  • A Armadilha do COUNT: Qual a diferença entre COUNT(*) e COUNT(coluna_X)?
    Atenção! O COUNT(*) conta TODAS AS LINHAS da tabela, inclusive as que têm valores NULL. Já o COUNT(coluna_X) IGNORA os valores NULL e conta apenas as linhas onde a coluna X está preenchida.
  • UNION vs UNION ALL: O comando UNION junta os resultados de duas consultas e REMOVE os registros duplicados (o que exige processamento/sort extra do banco). O UNION ALL junta tudo e MANTÉM as duplicatas, sendo bem mais rápido.

1. A Essência da DQL (SELECT)

A DQL (Data Query Language) tem o comando SELECT como sua estrela solitária. A estrutura básica para extrair dados exige pelo menos as cláusulas SELECT e FROM. O resto é opcional para refino da busca.

  • SELECT: Projeta as colunas. O curinga * traz todas as colunas. O uso do DISTINCT logo após o SELECT remove as linhas duplicadas do resultado.
  • FROM: Informa a(s) tabela(s) de origem dos dados. É aqui que os JOINs acontecem.
  • WHERE: O filtro bruto. Funciona linha a linha. Operadores lógicos (AND, OR, NOT) e relacionais (>, <, =, <>, LIKE, IN) são aplicados aqui.
  • ORDER BY: Ordena o resultado final. Pode ser Crescente (ASC - padrão) ou Decrescente (DESC).

2. Agrupamentos e Agregações

As funções de agregação pegam várias linhas e as resumem em um único valor. As clássicas são: SUM() (Soma), AVG() (Média), MIN() (Mínimo), MAX() (Máximo) e COUNT() (Contagem).

A Cláusula GROUP BY

Quando usamos uma função de agregação junto com uma coluna normal no SELECT, somos obrigados a agrupar pela coluna normal.
Exemplo: Queremos a soma de salários por departamento.

SELECT departamento, SUM(salario)
FROM funcionarios
GROUP BY departamento;

A regra de ouro do SQL é: Toda coluna que estiver no SELECT e não estiver dentro de uma função de agregação, DEVE constar na cláusula GROUP BY.

O Filtro Pós-Agrupamento (HAVING)

O WHERE não entende grupos, ele só enxerga linhas individuais. Se quisermos exibir apenas os departamentos cuja soma dos salários seja maior que 10.000, não podemos usar `WHERE SUM(salario) > 10000`. Aqui entra o HAVING, que atua exclusivamente filtrando os grupos já formados.

3. Entendendo os JOINs

O JOIN (Junção) é a operação que combina colunas de duas tabelas diferentes baseando-se num relacionamento (geralmente uma Chave Estrangeira ligada a uma Chave Primária).

  • CROSS JOIN (Produto Cartesiano): Acontece quando você junta tabelas sem especificar a condição de ligação (ON). Ele combina TODAS as linhas da Tabela A com TODAS as linhas da Tabela B. Se A tem 10 linhas e B tem 10 linhas, o resultado terá 100 linhas. Na prática, quase sempre é um erro de codificação.
  • INNER JOIN (Padrão): Se você escrever apenas JOIN no SQL, o banco assume que é um INNER. Ele filtra de forma restritiva: só mostra o cliente se ele tiver um pedido, e só mostra o pedido se ele tiver um cliente vinculado. É a intersecção matemática perfeita.
  • LEFT / RIGHT OUTER JOIN: É a junção inclusiva. O LEFT garante que todos os registros da "Tabela Esquerda" (a primeira a ser escrita no FROM) apareçam no resultado, mesmo que não tenham correspondência na tabela da direita. Onde não há casamento de chaves, os campos da direita vêm preenchidos com NULL.

4. Operadores de Conjunto vs JOINs

Um erro comum é confundir JOIN com os operadores de conjunto. O JOIN junta colunas de tabelas diferentes horizontalmente. Os operadores de conjunto juntam linhas de duas consultas diferentes verticalmente (empilhando os resultados).

Para usar operadores de conjunto, ambas as consultas devem ter o mesmo número de colunas e tipos de dados compatíveis na mesma ordem.

  • UNION: Une os resultados da Consulta A com os da Consulta B e elimina todos os registros duplicados.
  • UNION ALL: Une os resultados, mas mantém absolutamente todas as linhas, inclusive as duplicadas. É muito mais rápido para o banco executar que o UNION normal.
  • INTERSECT: Retorna APENAS as linhas que existem simultaneamente tanto na Consulta A quanto na Consulta B.
  • EXCEPT (ou MINUS na Oracle): Retorna as linhas que estão presentes na Consulta A, MAS QUE NÃO ESTÃO presentes na Consulta B (é uma subtração de conjuntos).
