Conteinerização e DevOps

⚡ Direto ao Ponto

  • DevOps: União de Development (Desenvolvimento) e Operations (Operações). É uma cultura/filosofia que visa quebrar os silos entre equipes, automatizando processos para entregar software de forma mais rápida e segura.
  • Conteinerização: Empacotamento de uma aplicação com TODAS as suas dependências (bibliotecas, binários) para que rode de forma idêntica em qualquer ambiente (portabilidade).
  • Pipeline CI/CD: O coração técnico do DevOps. É a esteira rolante automatizada que pega o código do desenvolvedor, testa, constrói e coloca em produção.

⚖️ O Confronto: Máquina Virtual vs Contêiner

A distinção arquitetural mais cobrada nas provas de Infraestrutura e Cloud.

Característica Máquina Virtual (VM) Contêiner (Container)
Abstração base Virtualiza o Hardware. Virtualiza o Sistema Operacional.
Sistema Operacional Cada VM possui o seu próprio SO Convidado (Guest OS) completo. COMPARTILHA o Kernel do SO Anfitrião (Host OS).
Tamanho e Inicialização Pesada (Gigabytes). Demora minutos para iniciar. Muito leve (Megabytes). Inicializa em segundos.
Isolamento Alto (Isolamento total por Hardware/Hypervisor). Baixo (Mais vulnerável, pois compartilham o mesmo Kernel).

🔄 A Diferença Crucial: CI / CD / CD

Continuous Integration (CI)

Integração Contínua

Os desenvolvedores enviam código (commits) para o repositório central várias vezes ao dia. O sistema compila o código e executa testes automáticos de imediato para detectar erros cedo.

Continuous Delivery (CD)

Entrega Contínua

Após o CI, a aplicação é empacotada e fica pronta para ser lançada no ambiente de Produção. Mas atenção: o envio final para produção exige um clique manual / aprovação humana.

Continuous Deployment (CD)

Implantação Contínua

É o nível máximo. Se o código passar em todos os testes automáticos, ele é enviado para os servidores de produção automaticamente, sem qualquer intervenção humana.

🚨 Pegadinhas de Prova

  • DevOps é uma Ferramenta? A banca afirma que "DevOps é um novo software da Microsoft para automatizar infraestrutura". Falso. DevOps é uma cultura, um movimento e um modelo de trabalho. Ferramentas (como Jenkins ou Docker) apenas apoiam essa cultura.
  • Docker vs Kubernetes: "O Docker é o orquestrador padrão do mercado". Falso. O Docker é o "Motor" (engine) que roda um contêiner individual. Quem orquestra (gerencia milhares de contêineres, reinicia os que caem, faz balanceamento) é o Kubernetes (K8s) ou o Docker Swarm.

1. A Cultura DevOps e o CAMS

Historicamente, existia o "Muro da Confusão" nas empresas de TI. O Desenvolvedor (Dev) queria lançar novidades rapidamente (mudança). O Administrador de Sistemas (Ops) queria manter o servidor estável (imobilidade).

A filosofia DevOps surgiu para unir estas duas perspectivas. Para caracterizar o DevOps em provas teóricas, usamos o acrônimo CAMS (ou CALMS):

  • C - Culture (Cultura): Foco nas pessoas e comunicação. O problema principal nunca é a ferramenta, é a cultura organizacional de "culpa".
  • A - Automation (Automação): Eliminar o trabalho manual repetitivo. (Ex: CI/CD, Infraestrutura como Código).
  • M - Measurement (Medição): Se não consegue medir, não consegue melhorar. Coleta rigorosa de métricas (logs, performance).
  • S - Sharing (Compartilhamento): Compartilhamento de responsabilidades, conhecimento e sucesso entre equipes.

    • *Muitas bancas usam CALMS, adicionando o L (Lean) - foco em reduzir o desperdício.

2. Docker e os Fundamentos de Contêineres

A frase de terror das equipes de TI sempre foi: "Mas na minha máquina funciona!". O código funcionava no notebook do desenvolvedor, mas quando ia para o servidor (com outra versão do Linux ou do Java), falhava.

A Conteinerização resolveu isso. Ela empacota o código junto com as bibliotecas, dependências e configurações necessárias num "Contêiner" imutável. Esse contêiner vai rodar exatamente da mesma forma no PC do desenvolvedor, no servidor de testes ou na Cloud.

A Magia do Linux (Namespaces e cgroups)

Como é que os contêineres são tão leves e não precisam instalar um SO completo (como fazem as VMs)?

  • Eles baseiam-se em tecnologias do Kernel do Linux. O cgroups (Control Groups) limita os recursos de hardware (ex: "Este contêiner só pode usar 500MB de RAM").
  • O Namespaces faz o isolamento lógico (ex: "Este contêiner acha que tem um disco próprio e não consegue ver os arquivos do contêiner vizinho").
  • Conclusão: Contêineres compartilham o mesmo Kernel da máquina anfitriã. Por isso são ultraleves.

Vocabulário Docker de Prova

  • Dockerfile: É o "receituário" em formato de texto. Um arquivo com instruções passo-a-passo para construir uma imagem.
  • Imagem (Image): É o pacote final criado pelo Dockerfile. É somente-leitura (imutável). É o "molde" ou a "forma de bolo".
  • Contêiner (Container): É a instância em execução de uma imagem. É o "bolo" pronto.
  • Registry: Onde as imagens são armazenadas e compartilhadas (ex: Docker Hub).

3. Kubernetes (K8s): A Orquestração

Se tiver apenas 3 contêineres, gerencie o Docker manualmente. Mas empresas como a Netflix ou o Google rodam milhares de contêineres. Como gerenciar se um falhar? Como garantir balanceamento de carga? O Kubernetes é o sistema *open-source* padrão mundial para automatizar a implantação, dimensionamento e gestão de aplicações contêinerizadas.

A Arquitetura do Kubernetes

  • Cluster e Nodes
    O K8s junta várias máquinas físicas ou virtuais num Cluster. Cada máquina desse cluster é chamada de Node (Nó). Existe o Master Node (o cérebro que controla tudo) e os Worker Nodes (os operários que rodam as aplicações).
  • O Pod (Conceito Crítico)
    Em provas, lembre-se: O Kubernetes NÃO interage diretamente com os contêineres. A unidade atômica e mais pequena que o Kubernetes gerencia é o Pod. Um Pod é um "casulo" que pode conter um ou mais contêineres que compartilham o mesmo endereço IP e espaço de armazenamento.
  • Kubelet
    É o agente primário que roda em cada Worker Node. A sua função é garantir que os contêineres estão rodando de forma saudável dentro do Pod e reportar ao Master Node.

4. O Pipeline DevOps na Prática

Uma esteira típica de CI/CD num ambiente DevOps corporativo flui através da seguinte cadeia de ferramentas (Toolchain):

  1. Planejar & Codificar: O desenvolvedor escreve o código e faz o "Commit" num sistema de controle de versões distribuído como o Git (GitHub, GitLab).
  2. Build & Test (Integração Contínua): O repositório notifica um servidor de CI (como o Jenkins). O Jenkins puxa o código, compila-o e executa os testes unitários de forma automática.
  3. Empacotamento: O Jenkins chama o Docker para criar uma nova Imagem contendo a versão aprovada do código e guarda essa imagem num Registry.
  4. Deploy (Entrega/Implantação): O Kubernetes (ou ferramentas como o Terraform/Ansible) puxa a nova imagem do Registry e atualiza a aplicação em ambiente de Teste ou Produção, muitas vezes usando estratégias como o Blue-Green Deployment (Zero Downtime).
  5. Monitoramento: A aplicação em produção é monitorada continuamente (com ferramentas como o Prometheus e Grafana) para fechar o ciclo de feedback aos desenvolvedores.
Atenção ao termo IaC: Infrastructure as Code (Infraestrutura como Código) é a prática de escrever scripts (usando Terraform ou Ansible) para criar e gerenciar servidores, em vez de um humano clicar manualmente em painéis de controle na Cloud. É a materialização da letra "A" (Automação) do CAMS.
