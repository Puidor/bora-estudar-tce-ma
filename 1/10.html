⚖️ O Confronto: Servidor Web vs Servidor de Aplicação

Característica Servidor Web Servidor de Aplicação
Foco Principal Conteúdo Estático (HTML, Imagens, CSS). Conteúdo Dinâmico e Lógica de Negócio.
Protocolos Suportados Principalmente HTTP / HTTPS. HTTP, RMI, RPC, JMS, protocolos de banco de dados.

⚙️ A Tríade da Alta Disponibilidade (HA)

⚖️

Load Balancing

Distribui as requisições dos usuários entre vários servidores para evitar sobrecarga em um único nó. (Escalabilidade Horizontal).

🔄

Fail-over

Mecanismo de tolerância a falhas. Se o Servidor A "morrer", o Servidor B detecta (via Heartbeat) e assume o tráfego instantaneamente.

💾

Session Replication

Copia os dados da sessão (ex: carrinho de compras) para os outros nós. Se o servidor cair, o usuário não perde o que estava fazendo.

📌 Sticky Sessions (Afinidade)

  • Quando o Load Balancer roteia o Usuário X para o Servidor 1 na primeira vez, e "gruda" (prende) esse usuário sempre no Servidor 1 nas requisições futuras.
  • Vantagem: Evita a necessidade complexa de replicar o estado da sessão para todos os nós.
  • Desvantagem: Se o Servidor 1 cair, o Usuário X perde a sessão inteira (é deslogado).

🚨 Pegadinhas de Desempenho

  • Connection Pooling (Pool de Conexões): A banca vai dizer que serve para criptografar o banco de dados. Falso. Serve para reaproveitar conexões abertas com o Banco de Dados, economizando CPU e tempo. É crucial para o desempenho.
  • Ativo-Ativo vs Ativo-Passivo:
    👉 Ativo-Ativo: Todos os servidores atendem requisições ao mesmo tempo (Usa Load Balancer).
    👉 Ativo-Passivo: O Servidor 2 fica dormindo. Só acorda e trabalha se o Servidor 1 pifar (É puramente Fail-over).

1. A Arquitetura em 3 Camadas (3-Tier)

Para entender o Servidor de Aplicação, precisamos entender onde ele mora. A arquitetura corporativa clássica divide o sistema em três camadas físicas ou lógicas:

  • 1. Camada de Apresentação (Web Server): Fica na linha de frente (frequentemente numa DMZ). Lida com o protocolo HTTP e entrega páginas HTML, CSS e Imagens direto para o navegador. (Ex: Nginx, Apache HTTPd).
  • 2. Camada de Lógica de Negócio (App Server): O "Cérebro". Fica escondido na rede interna. Processa os cálculos complexos, validações de negócio, gerencia transações financeiras e compila páginas dinâmicas (PHP, Java EE, .NET). (Ex: JBoss, Tomcat, WebLogic).
  • 3. Camada de Dados (Database Server): Onde a informação é armazenada persistentemente. O Servidor de Aplicação é o único que tem permissão para conversar com o Banco de Dados. (Ex: Oracle, PostgreSQL).

2. Balanceamento de Carga (Load Balancing)

Para garantir Escalabilidade Horizontal (adicionar mais máquinas lado a lado), colocamos um Balanceador de Carga na frente dos servidores de aplicação. Ele atua como um maestro.

Algoritmos de Distribuição (Atenção para Provas)

  • Round-Robin (Chave de Prova): É o método mais simples. Distribui as requisições de forma circular e sequencial. Requisição 1 vai pro Servidor A; Requisição 2 vai pro Servidor B; Requisição 3 vai pro Servidor C; a Requisição 4 volta pro A. Não leva em conta se o servidor A já está sobrecarregado.
  • Least Connections (Menos Conexões): Mais inteligente. Envia a nova requisição para o servidor que tiver o menor número de sessões/conexões ativas no momento.
  • IP Hash: Usa o Endereço IP do cliente para gerar um hash e determinar o servidor. Garante que o Cliente X sempre caia no Servidor A (é uma forma de implementar Sticky Sessions no nível do balanceador).

3. Gestão e Replicação de Estado (State)

O protocolo HTTP é Stateless (não guarda estado). Se você faz login na página 1 e vai para a página 2, o servidor não lembraria quem você é. Os Servidores de Aplicação resolvem isso criando uma Sessão (Session) mantida na memória RAM.

O problema: Num Cluster com 3 servidores, se você fez login no Servidor A, e a próxima requisição cair no Servidor B, o B não tem a sua sessão em memória. O sistema pedirá para você fazer login de novo.

Técnicas de Resolução

  • 1. Afinidade de Sessão (Sticky Sessions)
    Mandar o usuário sempre para o mesmo servidor.
    Problema: Fere o balanceamento perfeito e, se o nó falhar, a sessão morre.
  • 2. In-Memory Session Replication
    Sempre que o Servidor A atualiza uma sessão, ele envia uma cópia (via rede, normalmente Multicast) para as memórias RAM dos Servidores B e C.
    Problema: Se o cluster crescer muito (ex: 20 servidores), a rede fica engarrafada só copiando sessões (Traffic Overhead).
  • 3. Cache Externo / Banco Centralizado
    A melhor abordagem moderna. Nenhum servidor guarda a sessão na sua própria RAM. Eles gravam as sessões num servidor de Cache externo ultra-rápido (como Redis ou Memcached). Qualquer nó pode atender qualquer requisição e ler o estado no Redis.

4. Otimização de Desempenho (Tuning) e Detecção de Problemas

A administração de servidores de aplicação (como o Tomcat ou JBoss) exige configurações finas ("Tuning") para evitar gargalos (Bottlenecks) e vazamentos de memória (Memory Leaks).

A Mágica do Connection Pooling

Abrir uma conexão TCP com um Banco de Dados é um processo demorado e custoso para a CPU. Se o App Server abrisse e fechasse uma conexão para cada clique de cada usuário, o sistema colapsaria. O Pool de Conexões mantém um "lote" de conexões já abertas e ativas com o banco. O usuário pega uma emprestada, usa em milissegundos e devolve ao Pool. Economiza recursos massivamente.

Thread Pooling

Idêntico ao banco de dados, mas focado no processamento de requisições web. O servidor pré-cria um número X de "operários" (Threads). Se chegar uma carga súbita de 1000 usuários e só houver 200 threads no pool, 800 usuários ficarão na fila de espera. O tuning correto do maxThreads é vital.

Detecção de Problemas (APM e JVM)

  • APM (Application Performance Monitoring): Ferramentas (ex: Dynatrace, New Relic) que injetam agentes no código e mostram exatamente em qual linha de código a lentidão está a ocorrer.
  • Garbage Collection (GC): No ecossistema Java (onde rodam os maiores App Servers), o GC limpa a memória RAM automaticamente. Se mal configurado, o GC pode "congelar" o servidor inteiro (fenômeno Stop-the-World) por vários segundos. Monitorizar os logs do GC é rotina de administração obrigatória.
